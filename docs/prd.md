# PRD

## 文档定位

本文件只回答“做什么”，不回答“怎么做”。

当前 PRD 以 `vision` 为上层输入，收敛产品需要具备的核心能力、关键流程、状态约束与成功指标，不展开系统架构、技术选型、接口协议或实现细节。

当前产品形态不是面向泛用户的 SaaS，而是给开发者自己长期使用、运行在 agent 工作区里的高级 skill 集合。

当前范围也需要明确区分两层：

- 主系统当前服务于加密货币交易协作
- 系统可以读取更广泛的跨市场信息，作为判断背景与上下文输入
- 其他市场的信息可以参与分析，但现阶段不默认进入和加密交易同等完整的正式判断、持续跟踪、复盘与方法演化链路

## 当前目标

产品需要支持一个最小但完整的加密交易协作闭环：

1. 用户用很短的话发起高意图决策问题
2. 系统判断这是新链还是旧链延续，并重建必要上下文
3. 系统给出正式判断，或给出继续观望的正式结论
4. 进入持续跟踪的判断被纳入活跃链管理
5. 已结束链路进入复盘
6. 必要时把复盘结论升级为后续默认行为

## 核心对象

### 1. 决策链

`决策链` 是产品中的核心对象。它不是任意一段市场聊天，而是一条围绕同一个判断对象持续维护的链路。

一条正式决策链至少需要具备以下信息：

- `subject`：当前在看什么对象
- `thesis`：当前判断是什么
- `basis`：当前判断主要建立在什么依据上
- `invalidation`：什么变化会触发下一次更新或直接结束
- `timeframe`：主判断时间尺度
- `position_context`：当前持仓或计划状态

要求：

- 系统可以在承接过程中自动补齐其中大部分内容
- 在进入正式活跃管理前，上述信息必须达到足以支撑持续跟踪的完整度
- “继续观望”也是合法判断，仍应具备上述核心信息

### 2. 交易链路资产

`交易链路资产` 指历史决策链、执行结果、复盘结论，以及已确认的偏好边界。

要求：

- 交易链路资产默认保留和承接，优先保证链路不丢
- 历史链路必须允许被后续判断重新读取和引用
- 交易链路资产应保留“当时如何判断”的信息，而不只保留最终结果

### 3. 策略资产

`策略资产` 指会直接影响后续默认行为的稳定规则，例如风险边界、常见错误、优先级和复盘框架。

要求：

- 策略资产默认保守升级
- 策略资产升级必须可见、可确认、可回退
- 明显只属于单条链的经验，默认继续留在交易链路资产，不直接进入策略资产

## 决策链生命周期

一条决策链当前需要有四个状态：

1. `draft`
2. `active`
3. `closed`
4. `reviewed`

状态定义：

- `draft`：系统根据高意图输入创建的候选链，尚未进入正式持续管理
- `active`：链路已经被正式承接，需要持续跟踪、更新或等待触发
- `closed`：持仓、计划或观察条件已经结束，可以进入结果归因
- `reviewed`：已完成至少一次正式复盘

状态约束：

- 系统可以创建 `draft`，也可以建议状态流转
- 从 `draft` 进入 `active`、从 `active` 进入 `closed`，都需要明确确认
- 已关闭链在出现新的真实管理需求时，可以重新进入活跃状态
- 复盘应附着在已结束链路上，而不是独立漂浮

## 核心能力

### 1. 短入口承接

用户应该可以直接用很短的话发起问题，而不需要每次先重述完整背景。

要求：

- 只要输入已经明显指向真实资金决策，系统就必须先承接
- 系统不能把“入口短”理解为“分析可以变浅”
- 用户可以只给一句问题、一个对象、一个动作倾向，系统仍应启动正式处理

### 2. 新链 / 旧链归属判断

系统需要先判断一次输入更像旧链延续还是新链开始。

归属判断至少要考虑：

- 标的是否匹配
- 方向是否一致
- 时间尺度是否一致
- 与最近一次相关更新的时间距离
- 当前持仓或计划描述
- 用户当前动作意图

要求：

- 高意图问题默认先进入候选链，而不是退回普通聊天
- 如果归属证据足够，系统应直接挂接旧链
- 如果归属证据不足，系统应以最低摩擦方式完成一次确认
- 系统不能在低置信度下强行把新输入挂到旧链

### 3. 上下文补齐与正式判断

在给出正式判断前，系统需要先补齐必要上下文。

正式判断至少需要覆盖：

- 当前立场
- 主要依据
- 失效前提或反向证据
- 仍在等待的条件
- 是否值得行动，还是继续观望
- 所属时间尺度
- 与当前持仓或计划的关系

要求：

- 系统必须区分“有观点”和“值得行动”
- “不做”或“继续等”必须被视为正式、有效的结论
- 系统应优先形成共享的判断底板，再把结论投影到具体对象

### 4. 持续跟踪与判断快照

活跃链的后续变化不能覆盖原判断，而应形成可回看的判断演变。

每次有效更新至少应记录：

- 更新时间
- 更新类型
- 更新后的当前判断
- 更新原因
- 是否影响失效条件、执行状态或链路结束判断

当前需要支持的更新类型至少包括：

- 判断调整
- 执行记录
- 市场变化
- 情绪标注

要求：

- 同一条链允许反复更新
- 系统要能区分“想法变化”“真实执行”“市场变化”“情绪干扰”
- 用户回看任意一条链时，应能看见从创建到当前的判断演变

### 5. 回拉机制

系统不能把回拉当作可有可无的提醒，而要把它当成正式链路的一部分。

当前至少需要支持四类回拉触发：

- 失效条件接近
- 活跃链长时间未更新
- 计划动作达到待确认时点
- 链路疑似可以结束

要求：

- 一次回拉只追问一个关键问题
- 回拉应优先采用最低摩擦确认，而不是长消息打断
- 活跃链不应该完全依赖用户自己想起再回来更新
- 回拉失败或未回应，不应直接导致链路被静默改写

### 6. 结束与复盘

当持仓清完、计划取消、失效条件触发后结束，或观察窗口自然走完时，系统应把链路转入结束与复盘阶段。

复盘至少要回答：

- 结果主要由什么决定
- 当时判断质量如何
- 执行是否偏离计划
- 是否存在明显情绪干扰
- 有没有值得提议保留的经验

要求：

- 用户可以显式发起复盘
- 系统也应识别哪些活跃链已经到达可复盘状态
- 复盘不能只看盈亏，还要区分判断、执行和结果归因

### 7. 策略资产升级

不是每次复盘都必须升级策略资产。

一条经验要被提议升级为策略资产规则，至少需要同时满足：

- 具有重复性，而不是一次性偶发经验
- 归因足够一致，而不是表面相似但根因不同
- 可以被表述为具体默认行为，而不是泛化提醒
- 升级提议发生在复盘阶段，而不是普通对话中悄悄漂移

要求：

- 系统只能提议升级，不能静默升级
- 用户需要能看到规则来源于哪些链路或哪些复盘
- 已生效规则后续如果被证明不合适，应允许降级或回退

## 关键用户流程

### 流程一：新问题进入

1. 用户发起高意图市场或交易问题
2. 系统判断这是新链还是旧链延续
3. 系统补齐必要上下文
4. 系统输出正式判断，或输出继续观望的正式结论
5. 如有必要，系统创建或更新决策链

### 流程二：活跃链持续跟踪

1. 用户采纳判断，或链路已进入持续管理状态
2. 系统将其纳入活跃链
3. 后续围绕同一条链记录更新
4. 系统在关键时点发起最低摩擦回拉

### 流程三：链路结束与复盘

1. 系统或用户决策链路已到达结束条件
2. 用户确认结束
3. 系统发起或承接复盘
4. 若结论足够稳定，系统提议升级到策略资产

## 成功指标

### 北极星

30 天滚动窗口内，高意图决策问题中，被承接成可回看决策链并至少发生一次后续更新、执行记录或复盘回看的比例。

### 核心指标

- `链路承接率`：高意图决策问题中，被承接成决策链的比例
- `链路存活率`：决策链创建后，至少发生过一次后续有效更新的比例
- `复盘完成率`：已结束链中，完成正式复盘的比例
- `链路归属准确率`：系统对新链 / 旧链的默认判断中，不需要用户纠正的比例
- `重复错误收敛率`：已被策略资产标记的错误类型，在后续链中再次出现的频率变化

### 证伪条件

如果在一个可比的 30 天周期里，以下情况持续存在，则说明当前产品方向没有成立：

- 系统仍然经常无法把高意图问题承接成决策链
- 决策链在创建后仍然很快断裂，没有后续更新
- 用户仍然经常需要从零重建自己上次为什么那样判断
- 已结束链很少真正进入复盘
- 已被识别的问题模式没有体现在后续默认行为中

## 非功能要求

- 对真实资金决策保持正式判断标准
- 不因用户输入简短而降低严谨度
- 支持长期上下文连续性
- 支持并行决策链，而不是只保留单一路径
- 保持时间尺度清晰，避免不同周期观点混淆
- 不把记录数量当成目标，只在需要持续管理时进入存储
- 所有关键状态变化、策略资产升级与回退，都应对用户可见

## 当前平台边界

- **首要支持平台：Codex。** Claude Code / Gemini CLI 等其他 agent 工具暂不在当前实现范围内。
- **分发方式：** 用户从 GitHub 下载 skill 集合包，导入自己的 Codex 环境中使用。无独立安装包，无应用市场分发。
