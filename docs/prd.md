# PRD

## 1. 流程控制

### 本章范围

这一章固定整个系统的路由入口、各阶段分工边界、长期容器定义，以及对所有阶段都有约束力的最小固定规则。

- `OBSERVE / PLAN / EXECUTE / REVIEW / BACKTEST / ITERATE` 的详细设计，放到各自章节展开
- 各阶段和容器的内部字段、数据结构与详细设计，不在本章提前展开

### 核心术语

先固定以下几类概念：

| 类别 | 内容 | 作用 |
| --- | --- | --- |
| `路由` | `ROUTER` | 根据用户消息决定本轮先进入哪个阶段 |
| `交易` | `OBSERVE / PLAN / EXECUTE` | 表达在线主流程本轮工作重点在哪一侧 |
| `投研` | `REVIEW / BACKTEST / ITERATE` | 表达策略演化链路本轮工作重点在哪一侧 |
| `状态` | `WAIT-CONDITION / WAIT-UNTIL-FILL` | 表达当前计划或订单处于什么等待位置 |
| `订单` | `挂单 / 撤单 / 改单` | 表达订单层发生了什么真实操作 |
| `仓位` | `开仓 / 加仓 / 减仓 / 止损 / 止盈 / 平仓 / 对冲` | 表达仓位层发生了什么真实操作 |
| `容器` | `PLAN-POOL → PLAN-CHAIN` / `STRATEGY-POOL → STRATEGY-CHAIN` | POOL 承载对应 CHAIN |

补充定义如下：

- `PLAN-CHAIN` 是一笔或一组绑定暴露从观测到全部平仓或失效闭合的完整生命周期记录
- `PLAN-POOL` 承载全部 `PLAN-CHAIN`，活跃链只是其中一种默认读取或分组视图
- `STRATEGY-CHAIN` 是一个策略假设从候选、影子、验证到生效或归档的完整演化记录
- `STRATEGY-POOL` 承载全部 `STRATEGY-CHAIN`
- 当前阶段只固定这些概念边界，不提前展开 plan 自身的 shape、数据库存储形态和动作语义细节

### 流程总图

#### 在线主流程

```text
[USER MESSAGE] --> [ROUTER]
                       |
  +--------------------+
  |
  +--> 观察类入口 --> [OBSERVE]
  |    适用：单标的观察 / 监控条件 / 全市场扫描
  |    |
  |    +--> 先识别 observe 运行形态
  |    |    `single-symbol`
  |    |    `binance-market-scan`
  |    |    `monitor-existing-chain`
  |    |
  |    +--> 先按形态整理 observe checklist
  |    |    列出本轮要补的语境槽位 / 账户事实 / 外部上下文
  |    |
  |    +--> checklist 先成型，再并发派发获取动作
  |    |    主 agent 直接补 item
  |    |    必要时派发 sub-agent 补部分 item
  |    |    OHLCV / 宏观 / 新闻 / 用户上下文
  |    |    持仓 / 挂单 / 成交 / 余额
  |    |    必要时 web search
  |    |
  |    +--> 获取结果持续回填 checklist / 观测上下文
  |    |
  |    +--> 整理为可供 PLAN 消费的观测结果
  |    |
  |    +--> 若为 `binance-market-scan`
  |    |    先粗筛产出候选列表
  |    |    主 agent 再收敛 shortlist
  |    |    派发 sub-agent 逐个走 `single-symbol` 细筛链路
  |    |
  |    +--> 当前轮进入 `PLAN` 所需 item 已补齐 --> [PLAN] --> 回复用户
  |
  +--> 执行类入口 --> [EXECUTE]
  |    适用：基于当前生效 plan 执行 / 开仓 / 挂单 / 撤单 / 改单
  |    以及：加仓 / 减仓 / 止损 / 止盈 / 平仓 / 对冲
  |    |
  |    +--> 执行动作、读取订单 / 成交 / 持仓变化
  |    +--> 调用币安 API 核实持仓 / 挂单 / 成交变化
  |    |
  |    +--> 订单已挂出但未完成 --> [WAIT-UNTIL-FILL]
  |    |    用户消息触发 --> [EXECUTE / OBSERVE] 跟进
  |    |
  |    +--> 执行完成 --> 回复用户
  |
  +--> 复盘类入口 --> [REVIEW]
       适用：复盘 / 解释 / 总结
       |
       +--> 输出复盘 / 解释 / 阶段性结论
       +--> 回复用户
```

#### 研究与沉淀侧

```text
[REVIEW / 外部研究输入]
  |
  +--> 产出：candidate hypothesis / policy change --> [主 agent 研究调度]
       |
       +--> 派发 subagent
       |    适用：抽假设 / 收敛规则 / 局部自测
       |
       +--> 派发 [BACKTEST]
       |    适用：长样本回测 / sibling 对照 / 失败聚类 / 历史回放（方法）
       |
       +--> 汇总结果 --> [ITERATE]
            |
            +--> 升格 / 影子 / 分叉 / 归档 --> [STRATEGY-POOL]
            |    |
            |    +--> 当前语境下适用的单一生效分支版本 --> [OBSERVE / PLAN]
            |
            +--> 继续研究 --> [主 agent 研究调度]
```

- `外部研究输入`：主流程之外独立产出的研究材料，包括用户自提假设、公开策略文献、市场研究报告等；不依赖实盘持仓触发，产物经主 agent 研究调度接入策略演化链路
- `主 agent 研究调度`：主 agent 在研究侧承担的编排角色；负责按研究目标派发 subagent 或触发 `BACKTEST`，并汇总结果推进至 `ITERATE`

### 阶段约束与流转

非阶段对象（`PLAN-POOL / PLAN-CHAIN / STRATEGY-POOL / WAIT-CONDITION / WAIT-UNTIL-FILL`）的定义见前页概念表。

| 阶段 | 职责 | 典型流转 |
| --- | --- | --- |
| `ROUTER` | 根据用户消息决定本轮先去哪里，标出 `OBSERVE` 入口类型 | 意图不明或状态不清 → 优先路由至 `OBSERVE` |
| `OBSERVE` | 先按运行形态整理 checklist，再以 checklist 为并发工作面派发 skill / 数据获取 / 必要的 sub-agent，持续回填并收敛成 `PLAN` 可消费的观测结果 | checklist 成型后并发补 item；新缺口继续回填 checklist 并补齐；达到本轮最小充分语境后 → `PLAN`；新消息 / 条件变化 / 持仓语境下可再次触发 |
| `PLAN` | 回答”做不做、怎么做、为什么这么做”，生成下一版 plan 快照和状态标签 | `wait-condition` → 用户消息触发 `OBSERVE` 重查条件；重查发现入场条件已失效 → 本阶段产出 `abandon`；`ready-execute` → `EXECUTE` |
| `EXECUTE` | 回答”是否真的下出去了、成交了多少、还剩多少真实风险” | `WAIT-UNTIL-FILL` → 用户消息触发 `EXECUTE / OBSERVE` 跟进；用户要求重新判断 → 路由回 `PLAN`；执行确认 → 回复用户 |
| `REVIEW` | 对已结束或阶段性闭合的 plan 变化序列做解释、归因和提炼 | 产出 candidate hypothesis → 主 agent 研究调度 → `BACKTEST` → `ITERATE`；无 hypothesis → 回复用户 |
| `BACKTEST` | 对候选策略分支版本或策略轮廓进行历史样本验证，产出统计结论和下一轮研究方向 | 来自 `STRATEGY-POOL` 中的候选版本 → 验证完成 → `ITERATE` |
| `ITERATE` | 基于回测结论决定候选版本的下一状态，并回写 `STRATEGY-POOL` | 来自 `BACKTEST` → 回写 `STRATEGY-POOL` → 可回接 `OBSERVE / PLAN` |

### 固定约束

| 约束主题 | 固定约束 |
| --- | --- |
| 事实源优先级 | 只要交易所 API 可用，订单、持仓、余额、活跃挂单和成交默认先以 API 为事实源 |
| 链连续性 | 用户下次回来必须能在 `PLAN-POOL` 中继续当前活跃 `PLAN-CHAIN`，不能因为对话中断就丢失上下文 |
| 默认读取视角 | 默认读取当前活跃 `PLAN-CHAIN` 的最新 plan 快照 |
| 建议与执行分层 | 系统建议、用户确认、真实已生效状态是三层不同东西，未对齐前不能当成已经执行完毕 |
| plan 快照 | 一个 plan 默认就是用户本轮提问后，agent 给出的这一版交易策略；至少应覆盖市场分析、做单点位、仓位、止盈止损、杠杆、数量等关键信息；同一条 `PLAN-CHAIN` 上的每轮新判断默认 append 为新的 plan 快照，保留系统原建议、用户反馈、改判原因和历史版本序列 |
| 链边界 | 只要还在管理同一笔仓位、或同一交易计划下相互关联的多笔仓位的完整生命周期，从观测、挂单、成交、改止盈止损、加减仓到全部平仓前，默认都算同一条 `PLAN-CHAIN`；旧仓位已全部结束后再次围绕新机会开仓，或当前新增仓位本身就是独立新机会时，才切新链 |
| 跨链关系 | 若新开的仓位本质上是为旧链做对冲、补救或风险转移，应视为新链，但保留与旧链的关联；若只是旧链内部的保护、减灾或管理动作，则仍留在原链 |

## 2. 观察（OBSERVE）

### 作用与边界

- 负责把当前轮用户消息收敛成 `PLAN` 可直接消费的最小充分语境。
- 负责补事实、补证据、补上下文；不负责拍板执行，不负责提前穷举所有信息。
- 默认服务三类入口：`单标的直达`、`全市场扫描`、`沿当前链监控 / 重判`。

### 输入

- 当前轮用户消息、追问和反馈
- 当前市场信息
- 按 checklist 命中的账户事实：持仓 / 挂单 / 成交 / 余额
- 当前活跃 `PLAN-CHAIN` 的最新 plan 快照
- 当前适用的策略分支版本
- 最近讨论摘要、未闭合追问链、用户偏离建议记录、上一轮改判原因
- 上一阶段返回的状态
- 外部研究输入给出的市场语境、观察重点、风险提示
- 宏观信息、新闻和必要的 web search 结果

### 输出

- 观察记录
- 已补齐上下文的观测结果
- 候选机会列表，或需要并入同一版 plan 的机会集合

### 管理对象

- `PLAN-CHAIN`
- 当前生效 plan
- 当前适用的策略分支版本
- plan 状态标签
- 判断记录

### 运行规则

- 先成型 checklist，再按 item 并发补齐、持续回填；`OBSERVE` 不是串行重跑整套大分析。
- checklist 命中账户项时，才调用币安 API 拉取持仓 / 挂单 / 成交 / 余额。
- 优先补“判定证据”，不是优先追加新数据源；已有数据的派生证据不足时，再补新的外部上下文。
- 同一条 `PLAN-CHAIN` 上出现新追问、旧证据过期、用户反馈变化时，应带着上一版 plan 和历史反馈重新进入本阶段。
- 持仓语境下可重复进入 `OBSERVE -> PLAN` 管理循环。

### 运行形态

| 形态 | 目标 | 默认处理 |
| --- | --- | --- |
| `single-symbol` | 判断当前标的是否值得继续做 plan | 直接进入细筛 |
| `binance-market-scan` | 先找候选，再收敛 shortlist | 先粗筛，不在 scan 内做完整细筛 |
| `monitor-existing-chain` | 复查旧 plan 是否仍有效 | 带旧 plan、旧证据和新事实重判 |

- `binance-market-scan` 不应把全市场原始噪音直接交给 LLM；应先由 `binance-market-scan` 产出候选，再由主 agent 收敛 shortlist，并派发 sub-agent 逐个走 `single-symbol` 路线。

### 策略读取与匹配

这里只固定主流程如何在观察阶段读取策略，不提前展开成规则 DSL、表结构或类图。

#### 策略流与共享内核

- 当前不额外再抽一层独立调度对象；直接把每个策略视作一条持续 append 的 plan 流，拥有自己的标的范围、运行节奏、市场状态观察方式，以及 plan 更新逻辑。
- `OBSERVE` 不要求所有策略同频更新；每轮 cron 只推进本轮命中的策略流，未命中的策略本轮不更新。
- 每条策略流都在同一套共享内核里持久化；不额外给每个策略维护独立的 plan 持久化、账户风控、执行与对账栈。
- 同一时刻数据库里可以并存多条策略流；主流程读取的是这些流各自的最新 plan、最新 observe 和最新执行事实，不假设系统只有一条主流。

#### 默认顺序

- `原始数据 -> 基础分析层 -> 默认判定层 -> 个性策略层 -> plan 候选收敛`
- `weak-support`、`weak-match` 都是正式信号，含义是“可继续看，但应降级、比较、等待或缩小参与方式”。
- `不追 / 继续等 / 只观察` 这类结论必须绑定具体周期和监控有效窗口，不能把高周期慢信号误当成盘中全局结论。

#### 三层角色

| 层级 | 默认执行 | 职责 | 直接产物 |
| --- | --- | --- | --- |
| `基础分析层` | 是 | 把原始市场数据整理成公共语境 | 公共语境槽位 |
| `默认判定层` | 是 | 把公共语境转成主流程默认参考的信号矩阵 | 标准化判定信号 |
| `个性策略层` | 否；需召回 | 承接真正带 edge 的具体策略，做私有深判和比较 | 策略证据包 |

#### 基础分析层的最小公共槽位

| 槽位 | 作用 |
| --- | --- |
| `market_regime` | 当前更像趋势、震荡、事件驱动、高波动还是低流动性 |
| `directional_bias` | 当前方向偏置是多、空、中性还是冲突 |
| `structure_state` | 当前关键结构、关键位和结构破坏情况 |
| `momentum_volatility_state` | 当前动量和波动是扩张、压缩、放量还是衰减 |
| `execution_constraints` | 当前执行约束，如距离过远、止损过大、流动性差、事件窗口不利 |
| `risk_context` | 当前风险语境，如高周期逆风、上下方空间、结构拥挤度 |
| `feature_conflicts` | 当前公共特征之间的冲突 |
| `missing_context` | 当前仍缺哪些公共语境，导致不宜推进 |

- 这里固定的是槽位，不是固定指标清单；指标、结构算法、事件输入只是生成槽位的手段。

#### 默认判定层的最小接口

| 接口块 | 作用 |
| --- | --- |
| `name / version` | 判定项及其版本 |
| `depends_on` | 依赖哪些公共语境 |
| `judgement_scope` | 评估的是机会质量、参与质量、执行质量还是风险级别 |
| `signal` | `support / weak-support / neutral / warning / deny` |
| `reason` | 当前信号的主要依据、冲突项、触发原因 |
| `downgrade_path` | 不直接支持时，进入观察、等待、降级参与、换载体还是阻断 |

- 默认判定层输出更适合是矩阵，不是拍板式结论。

#### 个性策略层的最小接口

| 接口块 | 作用 |
| --- | --- |
| `family / branch / version / status` | 标识当前策略属于哪一族、哪一分支、哪一版本、处于什么状态 |
| `base_requirements` | 召回前置过滤依赖的公共语境 |
| `default_judgement_requirements` | 进入深判前需要满足哪些默认判定 |
| `private_feature_requirements` | 该策略自己的私有特征 |
| `timeframe_scope` | 观察、触发、确认、持有窗口 |
| `applicable_scenarios` | 适用环境、启动形态、执行前提 |
| `directional_applicability` | 做多、做空，还是双向但不对称 |
| `monitoring_window` | `继续观察 / 暂不追 / 等确认` 这类结论的有效窗口 |
| `hard_rejects / soft_signals / evaluation_mode` | 否决项、加分项、以及解释方式 |

- 个性策略层不能只返回一句 `match / reject / wait`；至少要同时带出时间维度、适用场景、方向和监控窗口。

#### checklist 与 LLM

- 三层都可以把缺口回写到 checklist；`missing-info` 应驱动本轮继续补齐，而不是伪装成确定结论。
- checklist 清空的含义是“本轮进入 `PLAN` 所需的最小语境已齐”，不是“世界上已无更多信息”。
- LLM 更适合做候选内二次软匹配，不适合直接面对全市场零散指标裸判策略。
- 送给 LLM 的输入应是 `策略证据包`，至少含公共语境、默认判定矩阵、私有特征已知项、未满足项、冲突项和当前主要风险。

#### 当前不预先固定

- 不固定指标全集、总分公式、universe 大小、规则 DSL、类继承树或数据库表。
- 不固定默认判定层究竟有多少条公共判定。
- 不要求所有个性策略都被公共接口完整表达；当前目标是先稳定三层职责和收敛顺序。

### 需要详细设计

- 观察记录保留什么，以及如何挂到当前 plan 快照
- `OBSERVE` 的观测结果以什么结构交给 `PLAN`
- 何时判定“本轮上下文足够完整，可以结束观察”
- checklist 的最小字段：`缺什么 / 为什么缺 / 谁提出 / 补到什么程度算完成`
- 账户事实何时命中 checklist 并触发 API 读取
- 多个候选机会如何挂到同一个 `PLAN-CHAIN`
- 外部研究、宏观、新闻、web search 如何并入同一轮观察上下文

## 3. 决策（PLAN）

### 作用与边界

- 负责回答“做不做、怎么做、为什么这么做”，并产出下一版 plan。
- 负责必要确认、分歧处理和风险重算。
- 不负责账户事实拉取；需要新事实时，应回到 `ROUTER -> OBSERVE / EXECUTE`。

### 输入

- 观察记录
- 已补齐上下文的观测结果
- 当前活跃 `PLAN-CHAIN` 的最新 plan 快照
- 当前适用的策略分支版本
- 最近讨论摘要、未闭合追问链、系统原建议、用户反馈、上一轮改判原因
- 持仓中的再判断需求
- 已被 `OBSERVE` 收敛进当前轮上下文的外部研究结论

### 输出

- 系统建议
- `方向状态`
- `执行结论`
- `不参与 / 只观察 / 延后 / 放弃` 这类正式输出
- 带 `wait-condition / ready-execute / noop / abandon` 等状态的下一版 plan
- 面向执行阶段的动作建议
- 单笔 plan 或组合判断 plan

### 管理对象

- `PLAN-CHAIN`
- 当前生效 plan
- 当前适用的策略分支版本
- 下一版 plan
- plan 状态标签
- 判断记录

### 输出契约

- 监控、盯盘、条件复查、追价这类场景，`PLAN` 默认拆成两层：`方向状态` 与 `执行结论`。
- 合法示例是：`方向状态 = 偏多已确认`，`执行结论 = 不追`；不能让“方向成立但不适合追”被误写成“没有信号”。
- 当用户意图已经明显是在追问当前判断，尤其是持仓中的 `继续拿 / 减一点 / 还看多吗 / 现在怎么办`，回复必须先给 plan 摘要，再给解释。
- plan 摘要至少让用户一眼扫到：`当前动作`、`方向状态`、`仓位处理建议`、`保护 / 止损要求`、`下一检查点`。
- 若读完整段后用户仍不知道下一步该 `继续持有 / 减仓 / 补保护 / 等待 / 退出` 哪一个动作，这次输出就不算合格的 plan。

### 当前先固定的 plan 最小结构

| 模块 | 最少回答什么 |
| --- | --- |
| `方向` | 当前方向状态、支持证据、反方证据、失效条件 |
| `量` | 总仓位 / 总预算、杠杆或载体、每一段量、每一段占比 |
| `价` | 入场价或区间、触发条件、失效位、退出价或条件价 |
| `保护` | 止损、止盈、移动保护、保本上移等规则 |
| `分段` | 每一段的价格或条件、数量或比例、建议强度 |
| `不做分支` | 什么情况下不参与、延后、缩仓、换载体或放弃 |

- `建议强度` 不是笼统信心分，而是“这一段不执行会不会明显破坏原计划”。
- 默认三段止盈必须允许安全折叠成两段；系统要明确哪一段是弱建议、可合并、可省略，以及省略后损失什么。
- 若某一段属于高强度核心保护或核心兑现位，应明确标注“不宜省略”。
- 分段止损沿用同一原则；若有分阶段减风险、保本上移、移动保护，必须写清触发条件、处理比例和建议强度。
- plan 应同时支持简洁视图和展开视图；简洁视图服务扫读，展开视图回答每一段“为什么这样设计”。

### 与上下游关系

- 来自 `OBSERVE`。
- 在 `PLAN` 内完成确认、分歧处理和风险重算后，生成下一版 plan，并写入状态属性等待下一轮路由。
- 若收到新的用户反馈、原证据过期、或判断发现上下文不足，应带着当前 plan 回到下一轮 `OBSERVE`。
- 持仓语境下可多次进入本阶段。

### 需要详细设计

- 一版 plan 如何结构化保存
- 当前重构方向见 [design-architecture.md](design-architecture.md) 的 `Plan 设计`：采用事件流持久化——plan 主体收窄为 `intent` 事件 body（最小 shape 9 个硬字段），证据 / 订单 / 成交 / 心跳 / 复盘分别作为独立 event kind append 到同一条 chain，视图均为读时投影
- 多轮 plan 如何 append，并同时保留原建议、用户反馈和改判原因
- 默认判定层与个性策略层冲突时，如何稳定输出 `support / weak-support / neutral / warning / deny / match / weak-match / reject / missing-info`
- 如何把“机会是否成立”和“当前是否适合参与”稳定拆成两层判断
- 如何把 plan 摘要固定成稳定输出，而不是被长解释吞掉
- 多机会何时收敛成组合 plan，何时拆成互斥 plan
- 外部研究结论与当前观察记录冲突时如何处理

## 4. 执行（EXECUTE）

### 作用与边界

- 负责把已确认的 plan 变成真实订单、真实成交和真实风险状态。
- 负责订单层对账；不负责条件等待，`wait-condition` 仍属于 `PLAN`。
- 只要交易所 API 可用，订单、成交、撤改结果默认以 API 为事实源。

### 输入

- 经 `PLAN` 确认后的当前生效 plan
- 用户确认的执行意图
- 持仓语境下的再执行动作
- 交易所 API 返回的订单、成交、撤改单状态

### 输出

- 订单记录
- 订单组或组合部分状态
- 执行动作记录
- `wait-until-fill`、对账结果、成交结果或恢复所需状态

### 管理对象

- `PLAN-CHAIN`
- 当前生效 plan
- 当前执行等待状态
- 订单

### 运行规则

- 执行建议、用户口头意图、真实已生效订单必须分层保存，不能混成同一事实。
- 组合判断 plan 进入执行时，可展开成订单组、执行顺序和部分成交状态，但仍应保持与原 plan 的映射关系。
- 订单未完全对齐前，系统必须能表达 `提交结果未定 / 撤单在途 / 改单在途 / 部分成交 / 旧单残留 / 订单失效`。
- 订单层已经失效但交易想法仍成立时，应回到 `PLAN` 生成新执行版本，而不是直接把整条链判死。

### 与上下游关系

- 来自 `PLAN`。
- 收到用户执行确认后，可调用币安 API 核实执行结果。
- 执行后可能进入 `wait-until-fill`、回到 `OBSERVE / PLAN`、或在链闭合后进入 `REVIEW`。
- 持仓语境下可重复发生。

### 需要详细设计

- 执行建议、用户反馈和订单生命周期如何区分留痕
- API 回报、用户反馈和本地状态冲突时如何裁决
- 组合 plan 如何展开成订单组，并保持部分成交状态
- 撤单、改单、失效如何回写主线
- 执行结果如何保持链不断开

## 5. 复盘（REVIEW）

### 作用与边界

- 负责还原“当时为什么这么看、后来为什么变了、这次问题到底出在哪一层”。
- 产物至少分三层：`复盘解释记录`、`待验证假设`、`候选策略或修订建议`。
- 不把一次复盘里的临时想法直接升格成长期规则。

### 输入

- 待复盘池中的已结束或阶段性闭合的 `PLAN-CHAIN`
- 交易记录
- 判断记录

### 输出

- 复盘记录
- 待验证假设
- 进入 `策略池` 的候选策略或分支修订建议

### 管理对象

- `PLAN-CHAIN`
- 待复盘池
- 交易记录
- 判断记录
- 候选策略

### 固定要求

- 默认先还原时间顺序：`当时能看到什么 -> 当时为什么这么判断 -> 后来发生了什么 -> 事后验证结果如何`。
- 不能只复述结果；至少拆开 `高周期静态结构`、`盘中动态异变`、`执行位置与方向优势`、`最终结果`。
- 当用户用真实盘中结果 challenge 旧判断时，要显式回答：是原判断错了，还是判断成立但盘中状态后来已经变了。
- 复盘要保留高价值追问链，因为这些追问本身就是候选假设来源。
- 当系统曾给出 `不追 / 继续等 / 无实质变化`，但事后行情启动了，错因至少要拆成：`策略错配`、`时间窗口错配`、`监控结论过期`、`识别漏判`、`执行约束阻断却被表达成方向否定`。
- 事件专题 review 的第一产物应是 `策略轮廓`，不是完整策略规则。

### 事件专题 review 的最小产物

#### 最小步骤

- 选 `事件锚点`
- 还原 `事件前语境`
- 拆解 `事件过程`
- 做 `错因归类`
- 提炼 `策略轮廓`

#### 策略轮廓的最小字段

- `thesis`
- `event_family`
- `market_regime`
- `direction`
- `timeframe_stack`
- `first_signal`
- `confirmation`
- `invalidation`
- `execution_style`
- `do_not_trade`

### 与上下游关系

- 来自主流程阶段性闭合后进入的待复盘池
- 从待复盘池中被拉起
- 产物进入 `策略池`，再进入 `BACKTEST`
- 与外部研究输入共享候选策略入池入口

### 需要详细设计

- 哪些结果进入待复盘池，如何筛选和拉起
- 如何在同一条 `PLAN-CHAIN` 里同时保留 `当时 plan 快照` 与 `事后验证版本`
- 如何把系统建议、用户执行和真实结果并排对照
- 如何把 edge 丢失点拆到观察、判断、执行、持有、保护、恢复衔接等不同环节
- 如何从聊天式复盘里抽取可复用模式，但不直接写成长期规则
- 复盘产物如何进入 `策略池` 并挂到具体分支和版本

## 6. 回测（BACKTEST）

### 作用与边界

- 负责验证候选策略、局部模板、事件轮廓或待验证假设，不只验证整套长期策略。
- 当前 `历史回放` 默认指按时间窗口滑动验证集、统计触发与结果分布；不默认指带模拟用户交互的历史主流程陪练。
- 回测前先绑定验证对象、样本范围和规则口径；没有口径，不给“成功率”。

### 输入

- 来自 `策略池` 的候选策略分支版本
- 来自历史回放或用户历史交易的验证集
- 来自事件专题 review 的策略轮廓或候选假设

### 输出

- 回测结果
- 进入 `ITERATE` 的候选内容
- 继续回测所需状态

### 管理对象

- 策略池中的候选分支版本
- 验证集
- 回测结果

### 固定要求

- 每次回测都必须绑定：验证对象、数据范围、标的、周期、入场定义、止损定义、止盈或持有窗口、去重规则。
- 结果不能只报胜率；至少同时给出 `样本数`、`触发频率`、`1R / 2R 命中`、`先止损`、`未完成样本`。
- 样本过少时必须显式标注 `样本不足`，不能硬说有效。
- 允许同一候选假设并行存在 `严格版 / 实战版 / sibling 版`，用来解释“定义严格但样本太少”与“定义更贴近实盘但质量一般”的差别。
- 默认按时间维度拆开验证；不能把 `5m / 15m / 1h / 4h / 日线` 混成一个总结果。
- 要显式记录策略是否多空对称；做多有效、做空无效时必须分开呈现。
- 当策略源自单次复盘时，应先做局部窗口验证，判断它是 `首发信号`、`弱延续` 还是噪音，再决定是否进入长样本回测。
- 结果除了统计结论，还要输出下一轮研究动作，例如 `继续放宽定义`、`增加延续过滤`、`拆成 sibling 版`、`只保留影子观察`、`样本不足先暂停`。

### 与上下游关系

- 来自 `策略池`
- 使用历史回放或用户历史交易
- 输出进入 `ITERATE`

### 需要详细设计

- 两个策略入口如何先进入 `策略池` 再共用同一回测口
- 自然语言假设如何收敛成最小可检验规则
- `事件专题 review -> 策略轮廓 -> 候选假设 -> 局部自测 / 长样本回测` 如何形成稳定闭环
- 如何稳定支持 `严格版 / 实战版 / sibling 版` 并排验证
- 如何在回测结果里区分 `首发信号` 与 `延续信号`
- 失败样本如何按原因聚类
- 回测结果如何沉淀进数据库

## 7. 迭代（ITERATE）

### 作用与边界

- 负责决定候选内容是继续观察、影子使用、限定语境使用、正式升级，还是归档。
- 是研究闭环的闸口，不是“看到漂亮案例就直接升格长期规则”的入口。
- 允许受控发散，但不允许滑向无边界指标挖矿。

### 输入

- 回测结果
- 相关策略分支和版本的当前状态

### 输出

- 下一轮可使用的策略结果
- 回写到 `策略池` 的分支、版本和状态决议
- 下一轮主线回接落点

### 管理对象

- 策略池
- 策略分支与版本
- 回测结果

### 固定要求

- 一次复盘或一次漂亮案例不能直接升格长期规则；先回答它是 `复盘结论 / 待验证假设 / 已通过验证的候选版本` 中哪一种。
- 当回测显示样本不足、定义过严或定义偏离实盘时，应允许保留候选版本并分叉 sibling 版本，而不是强行二选一。
- 迭代产物可落在不同层：`观察模板`、`决策模板`、`执行模板`、`持仓语境约束`；不必都升格成同一层长期策略。
- 迭代必须给出下一轮动作：`继续观察收样本`、`仅影子使用`、`限定语境小范围使用`、`升格为当前分支版本`、`归档`。
- 允许 agent 在明确语境下结合结构特征、波动特征、事件特征和少量指标做受控发散，但结果默认只算 `候选假设`，必须继续经过局部自测、`BACKTEST` 或 sibling 对照。

### 回写落点

- `观察层提示`
- `决策层输出口径`
- `执行层约束`
- `持仓管理约束`
- `策略分支 / 版本状态`

### 与上下游关系

- 来自 `BACKTEST`
- 回写 `策略池` 后重新进入主线使用或后续研究流程
- 为后续 `OBSERVE / PLAN / EXECUTE` 提供当前适用的生效分支版本
- 可显式回接 `OBSERVE / PLAN`，并由后续 checklist 决定是否读取账户事实

### 需要详细设计

- 哪些回测结果能进入迭代
- 迭代结果如何回写 `策略池` 并重新被主线使用
- 当前生效分支版本如何切换、回退或保留旧版对照
- 受控发散允许哪些特征组合、解释要求和停止条件
- 当严格版样本太少、实战版样本够但质量一般时，如何决定是继续收样本、只做影子版本，还是有限进入主线参考
- 如何记录“为什么这次不直接升级为长期规则”，避免后面重复讨论同一候选模式

## 8. Infra 层

### 8.1 作用

在 0-7 章的业务流转收敛之后，提供统一承载；不反向定义业务链路本身。

### 8.2 数据库

#### 输入

- 流程控制确定下来的 `PLAN-POOL`、当前活跃 `PLAN-CHAIN` 的最新 plan 快照、执行等待状态，以及策略分支和版本上的证明材料状态
- 2-7 章最终确定的长期对象和事件

#### 输出

- 持久化对象与状态
- 历史记录
- 上下文恢复所需数据
- 供主流程直接读取的结构化上下文，以及账户事实快照缓存
- UI 界面读取的数据

#### 管理对象

- `PLAN-CHAIN`
- `PLAN-POOL`
- 当前生效 plan
- plan 摘要
- 执行等待状态
- 订单
- 持仓
- 交易记录
- 判断记录
- 恢复上下文快照
- 最近讨论摘要与未闭合追问链
- plan 历史与分歧记录
- 策略分支与版本证明记录
- 策略池
- 策略分支与版本
- 验证集
- 回测结果

#### 对话素材层（chat-history）

这里补一层独立于 `PLAN-POOL / PLAN-CHAIN` 的对话素材层，用来承载高价值聊天过程里的原始业务素材。

- `PLAN-POOL` 更像决策池，用来挂住多条 `PLAN-CHAIN`。
- `PLAN-CHAIN` 更像单条在线推进对象，保存的是一笔或一组绑定暴露从观测、挂单、成交、管理到全部平仓或失效闭合的生命周期里，各版 plan 如何推进、怎么改判、怎么执行。
- `chat-history` 更像对话素材层，保存的是用户目标如何暴露、系统为什么改建议、哪些上下文和阻塞真正影响了决策，以及这些过程对后续 `PRD / user-story / 复盘` 有什么价值。
- 这一层不是为了替代 `PLAN-CHAIN`，也不是默认把所有聊天都升格成长期 memory。

这里先固定几个边界：

- `chat-history` 不是流程阶段，也不参与 router 分发。
- `chat-history` 不是策略证明材料；它最多是进入 `复盘 / PRD / user-story / 后续结构化洞察` 的上游素材。
- `chat-history` 不要求逐字保存完整原对话；默认只 capture 高价值片段。
- capture 后的内容必须是“压缩但保留判断变化链条”，不能只剩一句结论。

当前更适合把 `chat-history` 理解成一组按时间追加的 `history item`，但这里先不固定成完整表结构。一个 `history item` 至少应该能保留这些内容：

- 一段 `USER-> / AI<-` 的主对话脉络
- 用户目标如何变化
- AI 为什么这么判断或改判
- 当时用了哪些数据、技能、工具或账户事实
- 哪些 API 失败、截图回退、信息缺失或其他阻塞改变了执行方式
- 哪些内容值得后续升格为 `PRD` 主题、`user-story` 素材或复盘样本

这层的最小产品价值不是“记住所有对话”，而是把真实使用过程中长出来的需求、判断链和失败点留下来，避免后面写文档、做复盘或设计产品时，只剩零散聊天记录和事后回忆。

因此当前阶段更合适的 MVP 是：

- 先允许 agent 把一次高价值对话增量追加到 `chat-history.md`
- 主体采用 `USER-> / AI<-` 的方向格式
- 在主对话前补一层结构化摘要，例如 `Session Goal Evolution`、`Decision Log`、`Why Recommendations Changed`、`Friction And Failure Points`、`Product Insight`
- 默认 append，不覆盖旧内容
- 默认只记录高价值增量，不做全量自动记忆

这层与现有长期对象的关系先固定到这里：

- `PLAN-POOL` 负责回答“当前有哪些 `PLAN-CHAIN`、哪一条是活跃链”
- `PLAN-CHAIN` 负责回答“这条 plan 的生命周期现在到哪里了”
- `chat-history` 负责回答“我们为什么走到这里、过程中目标和判断怎么变了”
- 两者可以相互引用，但当前不要求所有聊天线程都强绑定到单一 `PLAN-CHAIN` 的某一版 plan
- 只有当对话素材已经被进一步提炼成候选假设、复盘样本或结构化结论时，才继续进入更正式的长期对象

这里再固定一个读取原则：

- `PLAN-CHAIN` 默认采用渐进读取，而不是每次恢复都全链展开
- 最近几版 plan 默认读详情；较早版本默认只读摘要
- 摘要只承担“快速说明这版 plan 在讲什么、为什么继续保留”这类职责，不替代原始 plan 详情
- 只有命中复盘、分歧回放、追责，或当前判断必须依赖旧细节时，才升级为回读旧版 plan 详情

#### 需要详细设计

- 哪些对象长期保存
- 各对象如何挂接
- 用户回来时最少恢复哪些内容
- 聊天线程、压缩摘要、未闭合追问、plan 历史和判断记录如何结构化挂到同一条 `PLAN-CHAIN`
- `chat-history` 与 `PLAN-CHAIN`、恢复上下文快照和长期 memory 的边界如何收敛
- `history item` 追加、去重、引用和升格为 `PRD / user-story / 复盘` 素材的最小规则是什么
