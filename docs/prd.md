# PRD

## 1. 流程控制

### 本章范围

这一章只固定主流程骨架、核心对象边界、常见回返和最小约束。

- `OBSERVE / PLAN / EXECUTE / REVIEW / BACKTEST / ITERATE` 的详细设计，放到各自章节展开
- `plan` 的内部字段、策略版本粒度、等待状态记录粒度、执行层记录粒度等结构化细节，不在本章提前展开

### 流程总图

先固定四类概念：

| 类别 | 最小集合 | 作用 |
| --- | --- | --- |
| `阶段` | `OBSERVE / PLAN / EXECUTE / REVIEW / BACKTEST / ITERATE` | 表达本轮工作重点在哪一侧 |
| `状态` | `WAIT-CONDITION / WAIT-UNTIL-FILL` | 表达当前计划或订单处于什么等待位置 |
| `订单动作` | `挂单 / 撤单 / 改单` | 表达订单层发生了什么真实操作 |
| `仓位动作` | `开仓 / 加仓 / 减仓 / 止损 / 止盈 / 平仓 / 对冲` | 表达仓位层发生了什么真实操作 |
| `长期容器` | `PLAN-POOL / STRATEGY-POOL` | 表达长期沉淀的对象 |

其中，`PLAN-POOL` 承载在线推进中的交易计划，`STRATEGY-POOL` 承载长期策略分支、版本和理论支持。

- `开仓` 只表达“开始进入一笔仓位”这一事实；具体执行方式可再区分为 `挂单成交 / 市价进入 / 条件触发进入`
- 不使用 `买单` 作为统一动作名，因为它只覆盖买入侧，不覆盖做空方向的进入
- `对冲` 作为独立仓位动作保留；它表达的是风险转移或保护意图，底层仍可能由一笔或多笔 `开仓 / 挂单` 组成

`WAIT-CONDITION` 和 `WAIT-UNTIL-FILL` 只是状态

### 上半部分：在线主流程

```text
这一部分只描述在线交易推进，所有在线交互都从 USER MESSAGE -> INTENT ROUTER 进入，
先决定本轮优先进入 OBSERVE、EXECUTE 或 REVIEW；若进入 OBSERVE，再按入口类型决定是直接深判、市场扫描后收敛，还是沿当前链做监控 / 重判。
OBSERVE 内部带 checklist；只要当前轮语境未齐，就继续留在本阶段补上下文。
PLAN 只消费已经补齐的观察上下文，负责生成下一版 plan 和其状态标签。
EXECUTE 只负责把已确认动作变成真实执行事实，并在必要时核实订单 / 成交 / 持仓变化。

[USER MESSAGE] --> [INTENT ROUTER]
  |
  +--> 观察类入口 --> [OBSERVE]
  |    适用：单标的观察 / 当前判断追问 / 监控条件 / 全市场扫描 / 外部观点映射
  |    |
  |    +--> 先识别 observe 运行形态
  |    |    `single-symbol`
  |    |    `market-scan`
  |    |    `monitor-existing-chain`
  |    |    `thesis-to-symbols`
  |    |
  |    +--> 运行 observe checklist
  |    |    补OHLCV / 宏观 / 新闻 / 用户上下文
  |    |    获取持仓 / 挂单 / 成交 / 余额
  |    |    必要时 web search
  |    |
  |    +--> 若为 `market-scan` / `thesis-to-symbols`
  |    |    候选生成 -> 粗筛 -> 排序 -> shortlist
  |    |
  |    +--> checklist 全完成 --> [PLAN]
  |         |
  |         +--> 在当前活跃 PLAN-CHAIN 上追加新的 plan 快照
  |         |    写入策略信息、判断结果和状态标签
  |         |    `noop / continue-scan / draft-closed`
  |         |    `wait-condition / ready-execute / abandon`
  |         +--> 回复用户
  |
  +--> 执行类入口 --> [EXECUTE]
  |    适用：基于当前生效 plan 执行 / 开仓 / 挂单 / 撤单 / 改单
  |    以及：加仓 / 减仓 / 止损 / 止盈 / 平仓 / 对冲
  |    |
  |    +--> 执行动作、读取订单 / 成交 / 持仓变化
  |    +--> 收到执行确认后调用币安 API 核实持仓 / 挂单 / 成交变化
  |    +--> 回复用户
  |
  +--> 复盘类入口 --> [REVIEW]
       适用：复盘 / 解释 / 总结
       |
       +--> 输出复盘 / 解释 / 阶段性结论
       +--> 回复用户
```

### 下半部分：研究与沉淀侧

```text
这一部分描述从 REVIEW 或外部研究输入进入的研究与沉淀侧。
上下两部分的交界点是 REVIEW；只有在 REVIEW 产出 candidate hypothesis / rule 时，
才继续进入这里。

[REVIEW / 外部研究输入]
  |
  +--> 产出：candidate hypothesis / rule --> [主 agent 研究调度]
       |
       +--> 派发 subagent
       |    适用：抽假设 / 收敛规则 / 局部自测
       |
       +--> 派发 [BACKTEST]
       |    适用：长样本回测 / sibling 对照 / 失败聚类 / 历史回放（方法）
       |
       +--> 汇总结果 --> [ITERATE]
            |
            +--> 升格 / 影子 / 分叉 / 归档 --> [STRATEGY-POOL]
            |    |
            |    +--> 当前语境下适用的单一生效分支版本 --> [OBSERVE / PLAN]
            |
            +--> 继续研究 --> [主 agent 研究调度]
```

### 核心边界

| 项 | 只负责什么 | 不负责什么 |
| --- | --- | --- |
| `INTENT ROUTER` | 根据用户消息和当前生效 plan 状态决定本轮先去哪里，并给 `OBSERVE` 标出入口类型 | 不负责补市场上下文，也不直接承担候选池过滤与排序 |
| `OBSERVE` | 补齐当前轮判断所需语境，产出可供 `PLAN` 消费的观察结果 | 不直接把未补齐语境硬送进 `PLAN` |
| `PLAN` | 回答“做不做、怎么做、为什么这么做”，并生成下一版 plan 快照 | 不负责补上下文，也不负责确认真实订单状态 |
| `EXECUTE` | 回答“是否真的下出去了、成交了多少、还剩多少真实风险” | 不重新做策略判断 |
| `REVIEW` | 对已结束或阶段性闭合的一段 plan 变化序列做解释、归因和提炼 | 不默认推进长期策略 |
| `PLAN-POOL` | 作为决策池，挂住多条 `PLAN-CHAIN`，并在恢复时定位当前活跃链 | 不是流程阶段 |
| `PLAN-CHAIN` | 对应一笔或一组绑定暴露从观测、挂单、成交、管理到全部平仓或失效闭合的完整交易生命周期，由 plan 快照序列构成 | 不是流程阶段 |
| `STRATEGY-POOL` | 保存长期策略分支、版本和证明材料，供主流程读取 | 不负责驱动研究循环 |
| `WAIT-CONDITION / WAIT-UNTIL-FILL` | 表示等待条件或等待成交的位置 | 不是独立阶段 |

### 常见回返

| 场景组 | 触发 → 落点 | 例 |
| --- | --- | --- |
| 刚收到用户消息、状态不清、执行未决 | 任意点 → `OBSERVE / EXECUTE / REVIEW` | 用户发来"现在怎么样"，不清楚是要看盘还是确认订单 |
| 开仓前反复判断 | 新消息 / 反馈 / 市场更新 → `OBSERVE` checklist 循环 → `PLAN` 生成新版本 | 用户说"再等等"，补了新阻力位，重出一版 plan |
| 当前判断已形成但条件未到 | `PLAN` 写入 `wait-condition`；用户消息触发 → `OBSERVE` 重查条件 | 做多方向定了，但等价格回踩 82000 再进 |
| 挂单后未最终成交 | `EXECUTE` 写入 `WAIT-UNTIL-FILL`；用户消息触发 → `EXECUTE / OBSERVE` | 限价单已挂出，价格还未到，等待成交 |
| API 事实与记录冲突 | → `OBSERVE / EXECUTE` 重新 API 校验 | 对话里记着有仓，API 返回持仓为 0 |
| 持仓后继续推进 | 持仓语境下仍走 → `OBSERVE / PLAN / EXECUTE` | 已持多仓，用户说"再看看加仓点" |
| 一条链结束或阶段性闭合 | 用户要求解释 / 总结 → `REVIEW` | 止盈平仓后，用户说"总结一下这笔" |
| `REVIEW` 发现值得沉淀的方法、假设或模板 | `REVIEW` → 主 agent 研究调度 → `BACKTEST` → `ITERATE` | 复盘发现某均线规律在这笔里效果好，值得系统回测 |
| 外部研究已足够像候选规则 | 外部研究输入 → 主 agent 研究调度 / `BACKTEST` | 用户粘贴一段研究报告说"这个规律试试" |
| 当前轮研究得到可保留结果 | `ITERATE` → `STRATEGY-POOL`；主流程读取当前适用版本 | 回测确认策略有效，升格写入策略池 |

### 固定约束

| 约束主题 | 固定约束 |
| --- | --- |
| 账户信息获取 | 获取持仓 / 挂单 / 成交 / 余额是 checklist 内部按需触发的动作，不是独立流程节点 |
| 事实源 | 只要交易所 API 可用，订单、持仓、余额、活跃挂单和成交默认先以 API 为事实源 |
| 连续性 | 用户下次回来必须能在 `PLAN-POOL` 中继续当前活跃 `PLAN-CHAIN`，不能因为对话中断就丢失上下文 |
| 当前读取视角 | 默认读取当前活跃 `PLAN-CHAIN` 的最新 plan 快照 |
| 判断与执行 | 系统建议、用户确认、真实已生效状态是三层不同东西，未对齐前不能当成已经执行完毕 |
| plan 快照定义 | 一个 plan 默认就是用户本轮提问后，agent 给出的这一版交易策略；至少应覆盖市场分析、做单点位、仓位、止盈止损、杠杆、量单量等关键信息 |
| plan 状态标签 | `valid / noop / continue-scan / draft-closed / wait-condition / ready-execute / abandon` 默认挂在当前 plan 快照上，作为 plan 属性或状态标签保存 |
| plan 快照追加 | 同一条 `PLAN-CHAIN` 上的每轮新判断默认 append 为新的 plan 快照，保留系统原建议、用户反馈、改判原因和历史版本序列 |
| 链边界 | 只要还在管理同一笔或同一组绑定暴露的交易生命周期，从观测、挂单、成交、改止盈止损、加减仓到全部平仓前，默认都算同一条 `PLAN-CHAIN`；旧仓位已全部结束后再次围绕新机会开仓，或当前新增仓位本身就是独立新机会时，才切新链 |
| 跨链关系 | 若新开的仓位本质上是为旧链做对冲、补救或风险转移，应视为新链，但保留与旧链的关联；若只是旧链内部的保护、减灾或管理动作，则仍留在原链 |

## 2. 观察

### 输入

- 当前轮用户消息、追问和反馈
- 当前市场信息
- 当前轮按需获取的持仓 / 挂单 / 成交 / 余额信息
- 当前活跃 `PLAN-CHAIN` 的最新 plan 快照
- `策略池` 中当前适用的生效分支版本
- 最近讨论摘要、未闭合追问链、上一版 plan、用户偏离建议记录和上一轮改判原因
- 上一个阶段返回的状态
- 来自外部研究输入的市场语境、观察重点或风险提示
- 宏观信息、新闻消息面和必要的 web search 结果

### 输出

- 观察记录
- 已补齐上下文、可供 `PLAN` 直接消费的观测结果
- 候选机会列表或需要纳入同一版 plan 判断的机会集合

### 管理对象

- `PLAN-CHAIN`
- 当前生效 plan
- 策略池中的当前分支版本
- plan 状态标签
- 判断记录

### 流程交互

- 在主流程中负责围绕当前用户消息拉取市场数据、运行策略、补齐所需上下文，并给 `PLAN` 提供可直接消费的观测结果
- 当入口是 `全市场扫描` 或 `外部观点 / 事件 / thesis 映射` 时，不应直接把全市场原始噪音交给 LLM 自由判断；应先做候选生成、粗筛、排序和 shortlist，再进入后续深判
- 运行时默认按 `基础分析层 -> 默认判定层 -> 个性策略层` 逐层推进，而不是把所有候选策略一次性完整跑完
- 来自流程控制定义的起点或回返点
- 当观察 checklist 命中账户项时，可调用币安 API 拉取当前持仓 / 挂单 / 成交 / 余额
- 可接入外部研究输入给出的市场语境和观察重点
- 可主动补读宏观信息、新闻消息面和必要的 web search 上下文
- 进入 `PLAN`
- 在上下文未齐时可在本阶段内自循环
- 当前层运行发现缺少公共语境、策略私有特征或额外市场上下文时，应把缺口回写到 checklist，再继续本阶段循环；默认优先补已有数据的派生证据，只有证据仍不足时才追加新的外部数据源
- 同一条 `PLAN-CHAIN` 上的新用户追问、上一版 plan 证据过期或用户反馈变化时，会带着上一版 plan 和历史反馈再次发生
- 可在持仓语境下再次发生

### 策略读取与匹配

这里只先固定“策略如何在观察阶段被主流程读取和匹配”的最小思路，不在这里提前展开成类图、表结构或完整 DSL。

#### 基本理解

- 主流程里的策略读取不应被理解成“一个统一策略直接下结论”，而应至少分三层：`基础分析层 -> 默认判定层 -> 个性策略层`
- 但在进入这三层之前，还需要先区分本轮 `OBSERVE` 属于哪种运行形态；至少要区分 `单标的直达`、`全市场扫描`、`沿当前链监控 / 重判`、`外部 thesis 映射`
- 若本轮是 `全市场扫描` 或 `thesis 映射`，应先经过 `候选生成 -> 粗筛 -> 排序 -> shortlist`，再把 shortlist 送进三层判断；否则 agent 会倾向直接做自由推理，输出不够结构化
- 第一层负责把原始市场数据整理成公共语境；第二层负责给出默认开启的公共判定信号；第三层才是策略池里那些需要被召回、筛选和比较的个性策略
- 因此主流程里的策略读取不是“把几十个指标全跑完再让每个策略自己解释”，而是 `原始数据 -> 基础公共语境 -> 默认判定信号矩阵 -> 个性策略召回与筛选 -> 当前 plan 收敛`
- 这三层在运行上也应与 `OBSERVE` 的内部 checklist 绑定：每一层既消费已补齐的语境，也可以把自己发现的缺口继续回写给 checklist，而不是默认一次跑到底；这里补的首先是“判定证据”，不是默认追加更多原始数据源

#### 三层角色

| 层级 | 默认是否执行 | 主要职责 | 直接产物 |
| --- | --- | --- | --- |
| `基础分析层` | 是 | 把原始市场数据整理成跨策略可复用的公共语境 | 公共语境槽位 |
| `默认判定层` | 是 | 基于公共语境产出一组默认开启的判定信号，帮助主线快速判断机会质量、参与质量和风险级别 | 判定信号矩阵 |
| `个性策略层` | 否；需先召回再筛选 | 承接真正带 edge 的具体策略，用私有特征进一步深判、比较和解释 | 候选策略及其证据包 |

这里固定的是“层级职责”，不是提前固定每一层里必须有哪些具体策略。

#### 第一层：基础分析层的最小公共接口

基础分析层至少应能输出这些公共语境槽位；它们不是某个单一指标，而是供后两层复用的高层特征类：

| 公共接口 | 作用 |
| --- | --- |
| `market_regime` | 当前更像趋势、震荡、事件驱动、高波动、低流动性等哪类环境 |
| `directional_bias` | 当前方向偏置是多、空、中性还是冲突 |
| `structure_state` | 当前关键结构状态，例如是否临近支撑/压力/共振区、结构是否破坏、是否接近突破边缘 |
| `momentum_volatility_state` | 当前动量与波动状态，例如扩张、压缩、放量、衰减、贴压整理等 |
| `execution_constraints` | 当前执行约束，例如离关键位是否过远、止损是否过大、流动性/滑点/事件窗口是否不利 |
| `risk_context` | 当前风险语境，例如高周期逆风强弱、上下方空间、结构拥挤度 |
| `feature_conflicts` | 当前哪些公共特征相互冲突，例如低周期偏强但高周期逆风强 |
| `missing_context` | 当前还缺什么公共语境，导致不宜直接推进到下一版 plan |

这里要固定的是“公共槽位”，不是固定“必须使用哪几十个指标”。指标、结构算法、趋势线算法、事件输入都只是生成这些公共槽位的手段。

#### 第二层：默认判定层的最小接口

默认判定层不是用户私有 edge，也不是只做底层描述。它位于 `基础分析层` 和 `个性策略层` 之间，负责把公共语境进一步变成主线默认会参考的一组判定信号。

这一层可以包含很多默认开启的公共判定策略，例如多周期关系、结构位置质量、趋势 / 震荡适配度、执行质量、风险收益空间、事件窗口约束等；这里先固定接口，不提前固定清单。

| 接口块 | 作用 |
| --- | --- |
| `name` | 默认判定项名称，例如某种结构质量或多周期关系判定 |
| `version` | 该判定项当前使用的版本 |
| `depends_on` | 依赖哪些基础分析层输出的公共语境 |
| `judgement_scope` | 这个判定在评估什么，例如机会质量、参与质量、执行质量、风险升降级 |
| `signal` | 输出的标准化信号，例如 `support / weak-support / neutral / warning / deny` |
| `reason` | 给出当前信号的主要依据、冲突项和触发原因 |
| `downgrade_path` | 当不是直接支持时，应该进入只观察、降级参与、等待、换载体还是直接阻断 |

默认判定层的输出更适合表现为矩阵，而不是一条拍板式结论。它回答的是“默认公共判断现在怎么看”，不是直接替代最终 plan 判断。

#### 第三层：个性策略层的接口

个性策略层才是策略池里那些真正需要被召回、比较和筛选的具体策略。它们建立在前两层之上，但不要求与前两层 `1:1` 完整对齐。

| 接口块 | 作用 |
| --- | --- |
| `family` | 策略族，例如突破、趋势跟随、回归、事件驱动 |
| `branch` | 同一策略族下的适用分支，例如适用标的、市场环境、周期 |
| `version` | 同一分支下逐步迭代的规则版本 |
| `status` | 当前处于 `candidate / shadow / active / deprecated` 等什么状态 |
| `base_requirements` | 依赖哪些基础分析层输出的公共语境，用于召回前置过滤 |
| `default_judgement_requirements` | 依赖哪些默认判定层信号，用于判断当前是否值得进入本策略深判 |
| `private_feature_requirements` | 该策略自己的私有特征，不要求基础分析层统一产出 |
| `timeframe_scope` | 该策略依赖哪些时间维度，例如观察周期、触发周期、确认周期和主要持有窗口 |
| `applicable_scenarios` | 该策略更适合什么市场环境、什么启动形态、什么执行前提 |
| `directional_applicability` | 该策略适用于做多、做空，还是双向都可；若双向不对称也要显式说明 |
| `monitoring_window` | 当监控进入“继续观察 / 暂不追 / 等确认”时，这个结论在多长时间窗口内有效，过期后是否必须重判 |
| `hard_rejects` | 明确否决条件 |
| `soft_signals` | 加分项、弱匹配项或需要解释的信号 |
| `evaluation_mode` | 规则如何被解释，例如硬条件、软条件、需要 LLM 二次判断的部分 |

- 个性策略层被读取时，不能只输出一句抽象的 `match / reject / wait`。至少要同时带出它所依据的 `时间维度`、`适用场景`、`适用方向` 和当前结论的 `监控有效窗口`
- 因此像 `不追 / 继续等 / 先观察` 这类结论，本质上都只能在具体周期和具体窗口内成立，不能把 `1h` 的等待结论误当成对 `5m` 启动段的全局否定，也不能把日线级别的慢信号直接套到盘中监控

#### 默认收敛顺序

- 第一步是 `基础分析层产出公共语境`：先把原始数据整理成稳定、可复用的公共槽位
- 第二步是 `默认判定层产出信号矩阵`：默认开启的公共判定策略基于公共语境给出 `support / weak-support / neutral / warning / deny` 等标准化信号
- 第三步是 `个性策略召回与筛选`：只有在基础语境和默认判定层支持继续看的前提下，才从策略池里召回值得进一步检查的个性策略
- 第四步是 `策略私有深判与 plan 收敛`：被召回的个性策略再补看自己的私有特征，结果允许是 `match / weak-match / reject / missing-info`，再与前两层结果一起收敛到下一版 plan
- `weak-support` 和 `weak-match` 都应被视为正式信号，而不是随手备注。它们表达的不是“错”，而是“可以继续看，但应降级、比较、等待或缩小参与方式”

#### `OBSERVE` 与 checklist 的关系

- `OBSERVE` 的 while 循环不应理解成“反复重跑同一套大分析”，而是“先跑到当前能跑的层，再把缺口补齐后继续往下跑”
- `基础分析层` 可以先产出 `missing_context`；例如缺高周期结构、缺波动状态、缺事件窗口、缺账户事实时，先把这些槽位加入 checklist
- `默认判定层` 若发现某类公共判定要成立还缺额外市场语境，也应把缺口回写到 checklist，而不是输出看似确定的结论
- `个性策略层` 若发现自身 `private_feature_requirements` 未齐、当前只够做弱匹配，或只能返回 `missing-info`，也应把这类缺口挂回 checklist，等待下一轮 `OBSERVE` 补齐后再继续比较
- 这类缺口多数不是“去找冷门指标”或“再接一个新数据源”，而是把已有数据继续加工成更细一层证据，例如多一层周期确认、局部窗口回看、结构失效方式、触发质量、执行约束或账户状态校验
- checklist 清空的含义不是“世界上已经没有更多信息”，而是“本轮推进到 `PLAN` 所需的最小语境已经齐全”；再往后的补充信息属于下一轮观察，不在本轮强行追全

#### LLM 在匹配中的位置

- LLM 更适合做 `候选内二次软匹配`，不适合直接面对全市场零散指标和零散材料裸判策略
- 因此应先给 LLM 一份 `策略证据包`，至少包含：基础分析层的公共语境、默认判定层的信号矩阵、个性策略的私有特征已知情况、未满足项、冲突项和当前主要风险
- LLM 的输出重点是解释：为什么匹配、为什么只是弱匹配、还缺什么信息、是否值得进入当前 `PLAN` 比较；不是跳过前两层直接替代全部策略匹配逻辑

#### 当前不预先固定的内容

- 不预先固定必须有哪些指标，也不要求每次主线都把所有指标完整平铺给策略层
- 不预先固定类继承树、数据库表或规则 DSL
- 不预先固定全市场扫描的最终过滤清单、统一总分公式或 universe 大小
- 不预先固定默认判定层里究竟有多少条公共判定策略，也不提前把它们固化成长期不变的制度
- 不预先要求所有个性策略都能被公共接口完整表达；目标是先让基础分析层和默认判定层稳定工作，再让个性策略在其上做召回、比较和私有扩展

### 需要详细设计

- 观察记录记录什么
- 观察结果如何挂到当前 `PLAN-CHAIN` 的最新 plan 快照
- `OBSERVE` 的观测结果以什么结构交给 `PLAN`
- `OBSERVE` 如何判断“当前轮上下文已经足够完整，可以退出 while 循环进入 `PLAN`”
- 观察 checklist 至少需要哪些最小字段，才能表达 `缺什么 / 为什么缺 / 谁提出的 / 补到什么程度算完成`
- 观察阶段如何读取当前适用的策略分支版本
- 原始市场数据、结构算法、指标和研究输入如何先收敛成基础分析层的公共语境，而不是把全部底层输出直接平铺给决策层
- 哪些公共特征由基础分析层稳定产出，哪些进入默认判定层，哪些只是临时研究特征或个性策略私有特征
- 多个候选机会如何挂到同一个 `PLAN-CHAIN` 的 plan 语境
- 观察 checklist 里的账户项何时应命中并调用币安 API 获取当前持仓 / 挂单 / 成交 / 余额
- 默认判定层和个性策略层产出的 `missing-info` 如何标准化回写到 checklist，并驱动下一轮 `OBSERVE`
- 外部研究结论如何映射成当前观察语境
- 用户消息、宏观信息、新闻消息面和 web search 结果如何并入同一轮观察上下文

## 3. 决策

### 输入

- 观察记录
- `OBSERVE` 产出的、上下文已补齐的观测结果
- 当前活跃 `PLAN-CHAIN` 的最新 plan 快照
- `策略池` 中当前适用的生效分支版本
- 最近讨论摘要、未闭合追问链、上一版 plan、系统原建议、用户反馈和上一轮改判原因
- 持仓中的再判断需求
- 来自外部研究输入的 thesis、反证和风险提示

### 输出

- 系统建议
- 方向状态，例如 `偏多已确认 / 偏空已确认 / 大区间震荡 / 区间边缘`
- 不操作建议
- 放弃当前机会
- 需要在 `PLAN` 内继续确认或处理分歧的判断建议
- 带有 `wait-condition`、`ready-execute`、`noop` 等属性的下一版 plan
- 面向执行阶段的动作建议
- 下一版 plan 或组合判断 plan

- 当场景是监控、盯盘、条件复查或“现在还能不能追”时，`PLAN` 输出默认应拆成两层：先给 `方向状态`，再给 `执行结论`。不能只给 `追多 / 追空 / 不追` 一层结论，否则很容易把“方向已成立但不适合追价”误说成“没有信号”。
- 因此像 `突破后延伸` 这类场景，允许正式输出为：`方向状态 = 偏多已确认`，`执行结论 = 不追`；核心是把 `方向识别` 和 `执行动作` 分开，而不是互相覆盖。
- 当用户意图已经明显是在追问当前判断，尤其是持仓中的 `继续拿 / 减一点 / 还看多吗 / 现在怎么办`，回复不能退化成一段长分析；默认必须先给一个可快速扫读的 plan 摘要，再给解释。
- 这个 plan 摘要至少要让用户在几秒内扫到：`当前动作`、`方向状态`、`仓位处理建议`、`保护 / 止损要求`、`下一检查点`。
- 解释层不能替代 plan 层。即使后面要补市场背景、跨周期依据、消息面和反方证据，前面也要先把“现在该怎么做”交代清楚。
- 若输出读完整段后，用户仍不知道自己下一步该 `继续持有 / 减仓 / 补保护 / 等待 / 退出` 哪一个动作，说明这次回复在产品意义上不算合格的 plan 输出。

### 管理对象

- `PLAN-CHAIN`
- 当前生效 plan
- 策略池中的当前分支版本
- 下一版 plan
- plan 状态标签
- 判断记录

### 流程交互

- 来自 `OBSERVE`
- 消费的外部研究结论默认应已在 `OBSERVE` 中被收敛进当前轮上下文
- 在 `PLAN` 内完成必要确认、分歧处理和风险重算后，生成下一版 plan 并写入其状态属性，等待后续用户消息决定下一轮路由
- `PLAN` 本身不负责账户事实拉取；若下一轮需要账户事实，应回到 `USER MESSAGE -> INTENT ROUTER -> OBSERVE / EXECUTE`，由对应 checklist 决定是否调用币安 API
- 不负责补上下文；若收到新的用户追问、用户反馈或发现原证据已过期，应带着当前 plan 重新进入下一轮 `USER MESSAGE -> INTENT ROUTER -> OBSERVE`
- 可在持仓语境下再次发生

### 当前先固定的 plan 最小结构

- 一版 plan 至少要同时回答三件事：`量怎么配`、`价怎么挂`、`止损止盈怎么收`；不能只有方向和几个价位。
- `量` 至少要能表达：总仓位 / 总预算、杠杆或载体、每一段分配量，以及每一段占总计划的比例。
- `价` 至少要能表达：入场价或入场区间、触发条件、失效位，以及每个退出阶段对应的目标价或条件价。
- `止损止盈` 至少要支持 `单段` 和 `分段` 两种形态；一旦进入 `分段`，每一段都必须带 `价格 / 条件`、`数量或比例`、`建议强度`。
- 这里的 `建议强度` 不是笼统信心分，而是“这一个阶段在当前建议里有多核心”。它服务的问题是：这一段如果不挂，会不会明显破坏原计划。
- 因此一套默认三段止盈的 plan，必须允许用户安全地折叠成两段；前提是系统能明确告诉用户：哪一段只是弱建议、可合并，可跳过后主要影响是什么。
- 若第一段止盈只是低强度兑现位，plan 应允许直接标成 `可省略 / 可并入下一段`，而不是把三段写成必须原样执行。
- 反过来，若某一段属于高强度核心保护或核心兑现位，就要明确说它不宜省略；否则用户很容易把“结构性关键段”误当成“随便一档”。
- 分段止损也沿用同一原则：若有分阶段减风险、保本上移、移动保护，必须能写清每一步处理多少仓位、在什么条件下触发、建议强度有多高。
- plan 的展开视图默认应支持把每一段退出动作解释成：`为什么是这一段`、`占比多少`、`强度多高`、`可以和哪一段合并`、`省略后会失去什么`。
- plan 的简洁视图默认应优先服务扫读；尤其在持仓追问场景，不能把关键动作埋在大段说明里。

### 需要详细设计

- 一版 plan 的最小内容如何结构化保存，至少覆盖市场分析、做单点位、仓位、止盈止损、杠杆、量单量，以及分段动作的比例与建议强度
- `PLAN` 如何消费 `OBSERVE` 的观测结果
- 同一条 `PLAN-CHAIN` 上的多轮 plan 如何 append 保存，并保留原建议、用户反馈和改判原因
- 当前适用的策略分支版本如何约束下一版 plan 的生成
- 如何基于基础分析层和默认判定层先形成默认判定矩阵，再决定召回哪些个性策略进入深判
- 当默认判定层与个性策略层给出冲突信号时，如何输出 `support / weak-support / neutral / warning / deny / match / weak-match / reject / missing-info`
- 如何把“机会成立 / 机会不成立”和“现在适合参与 / 现在不适合参与”拆成两层判断，并让 `不参与 / 只观察 / 缩仓 / 换载体 / 延后` 成为正式输出
- 如何在监控、追价和突破类场景下，把 `方向状态` 与 `执行结论` 同时保留，避免 `不追` 吃掉 `偏多已确认 / 偏空已确认`
- 如何在持仓追问场景下，把 `可快速扫读的 plan 摘要` 固定成稳定输出，而不是让长解释吞掉动作结论
- 下一版 plan 如何同时保留支持证据、反方证据、失效条件、`不做` 分支，以及入场、止损、止盈、分批、比例、建议强度和移动规则各自依据
- `wait-condition / ready-execute / noop / abandon` 这些 plan 属性如何结构化保存，并驱动下一轮用户消息路由
- 分歧处理点如何记录系统原建议、用户确认、风险提示、最终判断 plan，以及何时回到 `PLAN`
- 不同决策结果如何影响后续流转
- 多机会何时收敛成一个组合判断 plan，何时拆成多个互斥 plan
- 外部研究结论与观察记录冲突时如何处理

## 4. 执行

### 输入

- 经 `PLAN` 内必要确认、分歧处理和风险重算后确认的当前生效 plan
- 用户确认的执行意图
- 持仓语境下的再执行动作
- 交易所 API 返回的订单、成交和撤改状态

### 输出

- 订单记录
- 订单组或组合部分状态记录
- 执行动作记录
- 等待成交态、恢复所需状态、API 对账结果或成交结果

### 管理对象

- `PLAN-CHAIN`
- 当前生效 plan
- 当前执行等待状态
- 订单

### 流程交互

- 来自 `决策`
- 在收到用户执行确认消息后，可调用币安 API 核实执行结果
- 进入等待态、获取账户信息、`OBSERVE / PLAN` 或 `复盘`
- 可在持仓语境中重复发生

### 需要详细设计

- 执行建议、用户反馈和订单生命周期如何区分
- API 回报、用户反馈和本地持久化状态冲突时谁优先、如何留痕
- 执行只负责订单层等待，不负责条件等待
- 组合判断 plan 如何展开成订单组、执行顺序和部分成交状态
- 撤单、改单、失效如何回写主线
- 执行结果如何保持链不断开

## 5. 复盘

### 固定要求

- `复盘` 默认先还原时间顺序：`当时能看到什么`、`当时为什么会这么判断`、`后来发生了什么异变`、`事后验证结果如何`；不能直接用后验结果覆盖当时判断。
- `复盘` 不能只复述“最后涨了 / 跌了”。至少要把 `高周期静态结构`、`盘中动态异变`、`执行位置与方向优势`、`最终结果` 分层拆开。
- 当用户用真实盘中结果 challenge 系统先前判断时，复盘要能显式回答：是原判断本身错了，还是判断成立但盘中状态已经变了，只是当时没有及时切换解读。
- `复盘` 要支持把一次长对话里的追问链保留下来，例如从“做多还是做空更有优势”一路推进到“趋势线 + 水平线共振”“刚才为什么没看出异变”“这类异变后面怎么提前识别”；这些追问本身就是候选假设来源。
- `复盘` 产物至少分三层：`复盘解释记录`、`待验证假设`、`可进入策略池的候选策略`；不能把一次复盘里的临时想法直接当长期规则，也不默认把未验证结论跨窗口回接主流程。
- `复盘` 输出除了给人读，还要足够让主 agent 直接拿去派发下一轮 subagent；也就是说，至少要能显式提炼 `候选假设`、`假设依据`、`反例或疑点`、`优先验证问题`，避免下一轮研究又从整段聊天重新读起。
- 当监控或自动化曾给出 `不追 / 继续等 / 无实质变化`，但事后行情启动了，`复盘` 至少要能把原因拆到这些层：`策略本身不适配`、`时间窗口错配`、`监控结论已过期但未刷新`、`识别逻辑漏判`、`执行约束阻断但表达成了方向否定`。
- `复盘` 不只服务单条 `PLAN-CHAIN`，也要支持围绕 `过往事件` 做专题 review；事件可以是单次行情启动、单次宏观冲击、单次异常波动窗口，也可以是一组相似事件簇。这里的目标不是先修某条策略，而是先把事件拆成可比较、可提炼、可进入后续 mining 的研究样本。
- 围绕过往事件做 review 时，默认先回答这些问题：`事件前的市场语境是什么`、`第一批可见信号是什么`、`后续延伸和首发信号如何区分`、`当时系统漏掉了什么`、`这次更像方向判断问题、执行问题，还是输出口径问题`。
- `复盘` 与 `mining 策略` 之间不应直接跨栏。事件 review 的第一产物更适合是 `策略轮廓`，不是完整策略规则；只有当轮廓已经足够稳定，才继续进入候选假设、局部自测和回测。

### 输入

- 来自待复盘池的已结束或阶段性闭合的 `PLAN-CHAIN`
- 交易记录
- 判断记录

### 输出

- 复盘记录
- 待验证假设
- 进入 `策略池` 的候选策略或分支修订建议

### 管理对象

- `PLAN-CHAIN`
- 待复盘池
- 交易记录
- 判断记录
- 候选策略

### 流程交互

- 来自主流程阶段性闭合后进入的待复盘池
- 从待复盘池中被拉起
- 进入 `策略池`，再进入 `回测`
- 与外部研究输入共享候选策略入池入口

### 通过过往事件做专题 review 的最小产物

这里先固定从事件研究进入策略 mining 时，`复盘` 至少要产出什么，不提前把它写成完整策略 DSL。

#### 事件专题 review 的最小步骤

- 先选 `事件锚点`：例如某次突破启动、某次假突破、某次事件驱动急拉或急跌。
- 再还原 `事件前语境`：当时的大级别环境、关键结构、时间维度、事件窗口和执行约束。
- 再拆 `事件过程`：首发信号、第一次确认、第一次延伸、第一次失效或回撤分别是什么。
- 再做 `错因归类`：是没看到、看到了但没表达、表达了但动作结论被压掉，还是本来就不该做。
- 最后才进入 `轮廓提炼`：把这次事件先提炼成一个可继续验证的策略轮廓，而不是立刻改成主线默认规则。

#### 策略轮廓的最小字段

- `thesis`：这类事件想抓的核心现象是什么
- `event_family`：属于哪一类事件，例如突破启动、突破后延伸、假突破回落、事件驱动异动
- `market_regime`：更适合出现在哪类大环境
- `direction`：做多、做空，还是双向都可
- `timeframe_stack`：观察、触发、确认、持有主要分别看哪些周期
- `first_signal`：最早应该被识别的那一批信号是什么
- `confirmation`：哪些证据出现后，从“值得观察”升级到“值得做”
- `invalidation`：什么情况下这次轮廓不再成立
- `execution_style`：更像首发跟随、回踩确认、延伸跟随还是只适合观察
- `do_not_trade`：哪些场景即使方向对了也不该做

这里固定的是 `轮廓字段`，不是要求事件 review 一上来就把所有细节量化完。

### 需要详细设计

- 哪些结果进入待复盘池
- 待复盘池如何筛选和拉起
- 复盘记录保留什么
- 过往事件专题 review 如何选锚点、选相似案例、选反例，避免只挑最漂亮的单次样本
- 如何在同一条 `PLAN-CHAIN` 里同时保留 `当时 plan 快照` 和 `事后验证版本`，避免复盘时把后验信息倒灌回当时语境
- 如何沿同一条 `PLAN-CHAIN` 回看 plan 从生成到多次改动的过程，并把动作变化映射回具体证据、反证、事件、约束和用户确认
- 如何持续对照系统原建议、用户最终执行和真实结果，并区分系统判断、用户判断与执行偏离的影响
- 如何把 edge 从观察、判断 plan、下单、成交、仓位、持有、保护、恢复衔接和成本载体这些环节逐段拆开，区分“看对了但没赚到”到底丢在哪一段
- 如何把 `静态结构判断失效`、`盘中异变漏判`、`方向判断对但位置不好`、`位置好但不是趋势单` 这些不同问题显式分开，而不是都混成“判断错了”
- 如何把 `策略错配`、`时间窗口错配`、`监控结论过期`、`识别漏判`、`执行约束不满足` 这些不同错因拆开留痕，而不是都混成“当时不该说不追”
- 如何从聊天式复盘里抽取可复用模式，例如 `扩张阳线先行收复 -> 贴压横盘 -> 浅回踩不破 -> 低级别放量突破`
- 如何把一次复盘里的“第一次异变信号”和“后续加速延伸”区分开，避免把延续段误记成首发信号
- 如何让事件 review 先产出 `策略轮廓`，再决定哪些轮廓值得进入候选假设和回测，而不是直接把事件总结写成策略改动
- 复盘产物如何进入 `策略池`，并挂到具体分支和版本
- 哪些复盘产物只保留为当前窗口解释，哪些应继续推动策略迭代

## 6. 回测

### 固定要求

- `回测` 的对象不只是一整套长期策略，也要支持从复盘里抽出来的局部模板或候选假设，例如“盘中异变模板”“突破前吸收模板”“旧压力翻转支撑模板”。
- 当前 `历史回放` 默认指按时间窗口滑动验证集、统计触发次数和结果分布的回放方式，不默认指带模拟用户交互的历史主流程陪练。
- 每次 `回测` 都必须绑定清楚：验证对象、数据范围、标的、周期、入场定义、止损定义、止盈或持有窗口、去重规则；没有这些前提，就不要给出“成功率”。
- `回测` 结果不能只报胜率，至少要同时给出 `样本数`、`触发频率`、`1R / 2R 命中`、`先止损`、`未完成样本`；样本过少时必须显式说“样本不足”，不能硬说有效。
- `回测` 要允许同一候选假设并行存在 `严格版` 和 `实战版` 等 sibling 版本，用来回答“为什么严格版样本太少，但实战版仍值得观察”。
- `回测` 要支持回答这类问题：这次盘中异变究竟是 `第一次有效信号`，还是更早已有首发信号、当前只是延续加速；否则很容易把回测对象定义错。
- 当一个策略是从单次复盘里被发现时，`回测 / 自测` 应允许先围绕该发现点做局部时间窗口验证，例如检查发现点前后若干根 K 线、若干小时或若干个同类局部窗口，确认它究竟是 `首发信号`、`弱延续` 还是纯噪音，再决定是否进入更长样本回测。
- 上述局部时间窗口验证不替代正式长样本回测，但它是很重要的前置方法论：先回答“这套接口在发现当下能不能识别到该策略，以及能识别多久”，再回答“长期样本里总共有效多少次”。
- `回测` 不只要产出统计结果，还要产出下一轮研究可直接消费的结论，例如 `继续放宽定义`、`增加延续过滤`、`拆成 sibling 版`、`只保留影子观察`、`样本不足先暂停`；否则主 agent 无法稳定驱动下一轮。
- `回测` 默认要按时间维度拆开验证；同一候选策略至少要能回答 `5m / 15m / 1h / 4h / 日线` 中它究竟在哪些观察、触发和持有窗口内有效，不能把不同周期样本混成一个总结果。
- `回测` 也要显式记录策略是否 `多空对称`。若做多有效、做空无效，或只适用于单边趋势、不适用于震荡反手，都必须单独呈现，不能用汇总胜率掩盖。
- 当候选内容来自 `过往事件专题 review` 时，`回测` 不应默认把它当成完整策略；更合理的起点是先验证该 `策略轮廓` 的首发信号、确认条件、延伸条件和失效条件是否真的反复出现。

### 输入

- 来自 `策略池` 的候选策略分支版本
- 来自历史回放或用户历史交易的验证集
- 来自事件专题 review 的策略轮廓或候选假设

### 输出

- 回测结果
- 进入迭代的候选内容
- 继续回测所需状态

### 管理对象

- 策略池中的候选分支版本
- 验证集
- 回测结果

### 流程交互

- 来自 `策略池`
- 使用历史回放或用户历史交易
- 进入 `迭代`

### 需要详细设计

- 两个策略入口如何先进入 `策略池` 再共用同一回测口
- 每次回测如何绑定验证集来源
- 每次回测如何绑定具体分支、版本和回测前的策略状态
- 如何把聊天里的自然语言假设收敛成最小可检验规则，而不是要求用户先写成完整策略
- 如何把 `事件专题 review -> 策略轮廓 -> 候选假设 -> 局部自测 / 长样本回测` 串成稳定闭环，而不是一上来就硬写策略
- 如何支持 `严格版 / 实战版 / sibling 版` 并排验证，并保留它们之间的差异说明
- 如何对一次回测显式标注 `样本过少`、`定义过严`、`定义更贴近实盘` 这类研究结论
- 如何在回测结果里区分 `首发信号` 与 `延续信号`
- 如何围绕策略被发现的时间点，做局部窗口自测，并记录该策略在附近窗口内是 `match`、`weak-match` 还是 `reject`
- 如何把失败样本按原因聚类，例如 `突破后立刻失守`、`没有放量只有穿越`、`只是局部噪音不是结构异变`
- 回测结果如何沉淀进数据库

## 7. 迭代

### 固定要求

- `迭代` 不能因为一次复盘或一次漂亮案例就直接升格长期规则；默认先回答“这只是复盘结论、待验证假设，还是已通过验证的候选版本”。
- 当 `回测` 显示样本不足、定义过严或定义偏离实盘时，`迭代` 应支持继续保留候选版本并分叉 sibling 版本，而不是强行二选一地 `采纳 / 废弃`。
- `迭代` 产物要能落回不同层面：`观察模板`、`决策模板`、`执行模板`、`持仓语境约束`；不必把所有改动都升格成同一层的长期策略规则。
- `迭代` 要能明确给出下一轮动作：`继续观察收样本`、`仅影子使用`、`限定语境小范围使用`、`升格为当前分支版本`，以及回主线时是否需要额外读取账户事实。
- `迭代` 同时也是研究闭环的闸口；它必须明确回答当前候选版本是 `继续派发下一轮 subagent`、`暂停等待新样本`、`回到主流程影子使用`、`正式升级` 还是 `归档`，不能只给一段模糊总结。
- `迭代` 里的抽假设可以允许 agent 在明确语境下，结合结构特征、波动特征、事件特征和少量指标做受控发散；目标是发现值得验证的候选组合，而不是禁止 agent 主动探索“特征 + 指标”的可能性。
- 这类发散结果默认只算 `候选假设`，必须继续经过局部自测、`BACKTEST` 或 sibling 对照后，才有资格进入 `策略池` 的更高状态；不能因为一次发散结果好看就直接回写成主线默认策略。
- 这里允许的是“受控发散”，不是无边界指标挖矿；不应让 agent 穷举大量冷门指标组合，也不应把解释很弱、只在短样本相关的组合直接当作可用 edge。

### 输入

- 回测结果
- `策略池` 中相关分支和版本的当前状态

### 输出

- 下一轮可使用的策略结果
- 回写到 `策略池` 的分支、版本和状态决议
- 下一轮主线回接落点

### 管理对象

- 策略池
- 策略分支与版本
- 回测结果

### 流程交互

- 来自 `回测`
- 回写 `策略池` 后重新进入后续主线使用或后续策略流程
- 为后续 `观察 / 决策 / 执行` 提供当前适用的生效分支版本
- 可显式回接 `观察 / 决策`，并由后续 checklist 决定是否获取账户信息

### 需要详细设计

- 哪些回测结果能进入迭代
- 迭代后的内容如何回写 `策略池` 并重新被主线使用
- 当前生效分支版本如何切换、回退或保留旧版对照
- 如何把一次复盘里抽出来的局部模式回写成 `观察层提示`、`盘中异变 checklist` 或 `执行层约束`，而不是只能回写成整套策略
- 抽假设时允许 agent 带哪些结构特征、事件特征和指标做受控发散；组合上限、解释要求和停止条件如何约束，才能避免滑向指标挖矿
- 当严格版样本太少、实战版样本够但质量一般时，如何决定是继续收样本、只做影子版本，还是允许有限进入主线参考
- 如何记录“为什么这次不直接升级为长期规则”，避免后面重复讨论同一个候选模式
- `继续观察 / 影子恢复 / 小仓恢复 / 恢复常规` 如何由迭代结果决定

#### 对话素材层（chat-history）

这里补一层独立于 `PLAN-POOL / PLAN-CHAIN` 的对话素材层，用来承载高价值聊天过程里的原始业务素材。

- `PLAN-POOL` 更像决策池，用来挂住多条 `PLAN-CHAIN`。
- `PLAN-CHAIN` 更像单条在线推进对象，保存的是一笔或一组绑定暴露从观测、挂单、成交、管理到全部平仓或失效闭合的生命周期里，各版 plan 如何推进、怎么改判、怎么执行。
- `chat-history` 更像对话素材层，保存的是用户目标如何暴露、系统为什么改建议、哪些上下文和阻塞真正影响了决策，以及这些过程对后续 `PRD / user-story / 复盘` 有什么价值。
- 这一层不是为了替代 `PLAN-CHAIN`，也不是默认把所有聊天都升格成长期 memory。

这里先固定几个边界：

- `chat-history` 不是流程阶段，也不参与 router 分发。
- `chat-history` 不是策略证明材料；它最多是进入 `复盘 / PRD / user-story / 后续结构化洞察` 的上游素材。
- `chat-history` 不要求逐字保存完整原对话；默认只 capture 高价值片段。
- capture 后的内容必须是“压缩但保留判断变化链条”，不能只剩一句结论。

当前更适合把 `chat-history` 理解成一组按时间追加的 `history item`，但这里先不固定成完整表结构。一个 `history item` 至少应该能保留这些内容：

- 一段 `USER-> / AI<-` 的主对话脉络
- 用户目标如何变化
- AI 为什么这么判断或改判
- 当时用了哪些数据、技能、工具或账户事实
- 哪些 API 失败、截图回退、信息缺失或其他阻塞改变了执行方式
- 哪些内容值得后续升格为 `PRD` 主题、`user-story` 素材或复盘样本

这层的最小产品价值不是“记住所有对话”，而是把真实使用过程中长出来的需求、判断链和失败点留下来，避免后面写文档、做复盘或设计产品时，只剩零散聊天记录和事后回忆。

因此当前阶段更合适的 MVP 是：

- 先允许 agent 把一次高价值对话增量追加到 `chat-history.md`
- 主体采用 `USER-> / AI<-` 的方向格式
- 在主对话前补一层结构化摘要，例如 `Session Goal Evolution`、`Decision Log`、`Why Recommendations Changed`、`Friction And Failure Points`、`Product Insight`
- 默认 append，不覆盖旧内容
- 默认只记录高价值增量，不做全量自动记忆

这层与现有长期对象的关系先固定到这里：

- `PLAN-POOL` 负责回答“当前有哪些 `PLAN-CHAIN`、哪一条是活跃链”
- `PLAN-CHAIN` 负责回答“这条 plan 的生命周期现在到哪里了”
- `chat-history` 负责回答“我们为什么走到这里、过程中目标和判断怎么变了”
- 两者可以相互引用，但当前不要求所有聊天线程都强绑定到单一 `PLAN-CHAIN` 的某一版 plan
- 只有当对话素材已经被进一步提炼成候选假设、复盘样本或结构化结论时，才继续进入更正式的长期对象

这里再固定一个读取原则：

- `PLAN-CHAIN` 默认采用渐进读取，而不是每次恢复都全链展开
- 最近几版 plan 默认读详情；较早版本默认只读摘要
- 摘要只承担“快速说明这版 plan 在讲什么、为什么继续保留”这类职责，不替代原始 plan 详情
- 只有命中复盘、分歧回放、追责，或当前判断必须依赖旧细节时，才升级为回读旧版 plan 详情

#### 需要详细设计

- 哪些对象长期保存
- 各对象如何挂接
- 用户回来时最少恢复哪些内容
- 聊天线程、压缩摘要、未闭合追问、plan 历史和判断记录如何结构化挂到同一条 `PLAN-CHAIN`
- `chat-history` 与 `PLAN-CHAIN`、恢复上下文快照和长期 memory 的边界如何收敛
- `history item` 追加、去重、引用和升格为 `PRD / user-story / 复盘` 素材的最小规则是什么

### 8.3 UI 界面
