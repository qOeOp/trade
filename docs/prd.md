# PRD

## 1. 流程控制

### 1.1 固定主线

`观察 <-> 决策 -> 执行 -> 管理 -> 复盘 -> 验证 -> 迭代`

这条主线先定义“当前系统怎样持续推进一条交易链”，再决定 2-10 章各自在链上的职责。

### 1.2 主线与支线接点

- 主线负责围绕真实交易持续推进
- `r&d agent` 不依赖当前持仓，负责从外部想法生成可验证策略
- `backtest` 不生成策略，只为 `验证` 提供历史数据验证结果
- `复盘` 和 `r&d agent` 是两个策略入口
- `验证` 是统一验证口
- `迭代` 只能接在 `验证` 之后
- `infra` 不定义业务流转，只承载前面已经确定的业务对象和状态

### 1.3 章节在工作流中的位置

| 章节 | 在整体工作流中的职责 | 主要输入来自 | 主要输出给 |
| --- | --- | --- | --- |
| `2. 观察` | 为当前交易链补充最新市场语境和观察基础 | 市场信息、当前交易链、上个阶段回返状态 | `3. 决策` |
| `3. 决策` | 基于观察和上下文形成动作判断 | `2. 观察`、`5. 管理` 中的再判断需求 | `4. 执行` 或回退到 `2. 观察` |
| `4. 执行` | 把决策转成订单或动作结果 | `3. 决策`、`5. 管理` 中的再执行动作 | 等待态、`5. 管理` 或回返 `2/3` |
| `5. 管理` | 在持仓/已成交语境下持续推进链路 | `4. 执行` 的成交结果、当前持仓 | `2/3/4` 或 `6. 复盘` |
| `6. 复盘` | 对阶段性闭合的交易链产出策略结论 | 已结束或阶段性闭合的交易链 | `7. 验证` |
| `7. 验证` | 统一接收策略并绑定验证数据做验证 | `6. 复盘`、`9. r&d agent`、`10. backtest`、真实交易数据 | `8. 迭代` |
| `8. 迭代` | 把已验证结果变成下一轮可用内容 | `7. 验证` | 后续主线使用或后续策略流程 |
| `9. r&d agent` | 把零散想法整理成可验证策略 | 用户/agent/社区/公开资料中的想法 | `7. 验证` |
| `10. backtest` | 为策略提供历史数据模拟验证结果 | `6. 复盘` 或 `9. r&d agent` 的策略、历史数据 | `7. 验证` |

### 1.4 固定回返

| 场景 | 回返关系 |
| --- | --- |
| 开仓前反复判断 | `观察 <-> 决策` |
| 不建议操作、放弃计划 | `决策 -> 观察` |
| 撤单、改单、失效 | `执行/等待 -> 观察/决策` |
| 持仓中继续判断 | `管理 -> 观察/决策/执行` |
| 持仓中再次挂单 | `管理 -> 执行 -> 等待 -> 管理/观察` |

### 1.5 固定约束

- 用户下次回来必须能继续当前交易链
- 后续章节设计都要回答“谁触发、接到哪里、给谁输出”
- 订单变化会反向影响主线所处阶段
- `等待` 先按执行后的语境处理，不先独立固定成新阶段
- 复盘不会直接变长期规则，必须先进入验证
- 迭代只能发生在验证之后

### 1.6 仍需继续回答的前置问题

- 一条交易链何时开始、何时闭合
- `等待` 需要记录到什么粒度，才能支撑回返和恢复
- 系统在执行上是直接下单、辅助下单还是记录下单
- `管理` 何时算阶段性结束，何时进入 `复盘`

## 2. 观察

### 输入

- 当前市场信息
- 当前交易链上下文
- 上一个阶段返回的状态

### 输出

- 观察记录
- 进入决策的观察基础

### 管理对象

- 交易链
- 阶段状态
- 判断记录

### 流程交互

- 在主线中负责给 `决策` 提供当前语境
- 来自流程控制定义的起点或回返点
- 进入 `决策`
- 可被 `决策` 打回
- 可在 `管理` 中再次发生

### 需要详细设计

- 观察记录记录什么
- 观察如何挂到当前交易链
- 如何恢复到当前观察语境

## 3. 决策

### 输入

- 观察记录
- 当前交易链上下文
- 持仓中的再判断需求

### 输出

- 操作建议
- 不操作建议
- 放弃计划
- 进入执行的动作决定

### 管理对象

- 交易链
- 阶段状态
- 判断记录

### 流程交互

- 来自 `观察`
- 进入 `执行`
- 可回退到 `观察`
- 可在 `管理` 中再次发生

### 需要详细设计

- 决策记录记录什么
- 决策与观察的往返如何表达
- 不同决策结果如何影响后续流转

## 4. 执行

### 输入

- 决策结果
- 管理中的再执行动作

### 输出

- 订单
- 执行动作记录
- 等待态或成交结果

### 管理对象

- 交易链
- 阶段状态
- 订单

### 流程交互

- 来自 `决策`
- 进入等待态、`管理` 或返回 `观察/决策`
- 可在 `管理` 中重复发生

### 需要详细设计

- 执行动作如何进入订单生命周期
- 撤单、改单、失效如何回写主线
- 执行结果如何保持链不断开

## 5. 管理

### 输入

- 已成交订单
- 当前持仓
- 持仓中的新增判断

### 输出

- 管理动作
- 新订单
- 进入复盘的交易结果

### 管理对象

- 交易链
- 阶段状态
- 持仓
- 订单
- 判断记录

### 流程交互

- 来自 `执行`
- 内部可再次触发 `观察`、`决策`、`执行`
- 结束后进入 `复盘`

### 需要详细设计

- 持仓未结束但动作持续发生时如何表达
- 加仓、减仓、调杠杆、止损、止盈如何挂回同一交易链
- 管理何时算阶段性结束

## 6. 复盘

### 输入

- 已结束或阶段性闭合的交易链
- 交易记录
- 判断记录

### 输出

- 复盘记录
- 新策略或策略修正

### 管理对象

- 交易链
- 交易记录
- 判断记录
- 策略

### 流程交互

- 来自 `管理`
- 进入 `验证`
- 与 `r&d agent` 一起构成两个策略入口

### 需要详细设计

- 哪些结果进入复盘
- 复盘记录保留什么
- 复盘产物如何进入验证

## 7. 验证

### 输入

- 来自 `复盘` 的策略
- 来自 `r&d agent` 的策略
- 来自 `backtest` 或真实交易记录的验证数据

### 输出

- 验证结果
- 进入迭代的候选内容
- 继续验证所需状态

### 管理对象

- 策略
- 验证数据
- 验证结果

### 流程交互

- 来自 `复盘` 和 `r&d agent`
- 使用 `backtest` 或真实交易记录
- 进入 `迭代`

### 需要详细设计

- 两个策略入口如何共用同一验证口
- 每次验证如何绑定数据来源
- 验证结果如何沉淀进数据库

## 8. 迭代

### 输入

- 验证结果
- 已进入后续使用的策略内容

### 输出

- 下一轮可使用的策略结果

### 管理对象

- 策略
- 验证结果

### 流程交互

- 来自 `验证`
- 重新进入后续主线使用或后续策略流程

### 需要详细设计

- 哪些验证结果能进入迭代
- 迭代后的内容如何重新被主线使用

## 9. `r&d agent`

### 输入

- 用户提出的假设
- agent 发散出的假设
- 社区流传的策略
- 网络、论坛、论文里的公开策略

### 输出

- 可进入验证的策略

### 管理对象

- 策略

### 流程交互

- 不依赖当前持仓
- 与 `复盘` 一起构成两个策略入口
- 进入 `验证`

### 需要详细设计

- 零散想法如何进入系统
- 想法如何整理成可验证策略
- 如何与复盘产物共用同一验证入口

## 10. `backtest`

### 输入

- 来自 `复盘` 或 `r&d agent` 的策略
- 历史数据

### 输出

- 提供给验证阶段的验证数据

### 管理对象

- 验证数据

### 流程交互

- 不生成策略
- 不直接产生迭代结果
- 只向 `验证` 提供数据支持

### 需要详细设计

- 哪些策略进入 `backtest`
- `backtest` 向验证输出什么
- 历史数据验证如何与真实交易验证并存

## 11. Infra 层

### 11.1 作用

在 1-10 章的业务流转收敛之后，提供统一承载；不反向定义业务链路本身。

### 11.2 数据库

#### 输入

- 流程控制确定下来的当前状态
- 2-10 章最终确定的长期对象和事件

#### 输出

- 持久化状态
- 历史记录
- 上下文恢复所需数据
- UI 界面读取的数据

#### 管理对象

- 交易链
- 阶段状态
- 订单
- 持仓
- 交易记录
- 判断记录
- 策略
- 验证数据
- 验证结果

#### 需要详细设计

- 哪些对象长期保存
- 各对象如何挂接
- 用户回来时最少恢复哪些内容

### 11.3 UI 界面

#### 输入

- 数据库中的长期数据
- 流程控制中的当前状态

#### 输出

- 查看结果
- 基础 CRUD 结果

#### 管理对象

- 被允许展示和编辑的长期数据对象

#### 需要详细设计

- 第一批展示哪些对象
- 哪些对象允许 CRUD
- 如何展示当前链路、历史链路、策略、验证
