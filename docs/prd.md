这里表达的不是“章节顺序”，而是一个更完整的 user story。

其中最关键的是：

- 图里节点改成英文，是为了避免中文双宽字符把 ASCII 结构撑歪
- 右侧逐行中文注释负责解释每一步的业务含义
- `RECOVERY` 是正式阶段，不只是“重新加载上下文”，而是先对齐真实状态，再决定回到哪里继续
- 要先按 `开仓前 / 持仓中 / 平仓后` 三段来理解，而不是按线性章节来理解
- 开仓前不是 `OBSERVE -> DECIDE -> PLAN`，而是 `OBSERVE <-> PLAN`
- 这里的 `PLAN` 不是脱离策略临时生成；正常形态下，它应由当前观察语境结合当前生效策略推导出来
- 开仓前的交易计划会分成 `WAIT-CONDITION`、`DROP`、`MKT`、`PLACE` 四条路
- 流程图里的主操作对象不是单订单，而是 `交易链` 下的当前交易计划；订单只是执行层的落地结果
- 当前交易计划可以是单机会计划，也可以是组合交易计划，用来承接多机会排序、对冲和分支预案
- `WAIT-CONDITION` 表达的是“计划仍有效，但条件还没成熟，暂时还没有挂单”
- `WAIT-CONDITION` 承接的是计划级等待，不要求只对应单一订单或单一腿
- `MKT` 会直接进入 `IN-POS`，只有 `PLACE` 才进入 `WAIT-UNTIL-FILL`
- `WAIT-UNTIL-FILL` 必须显式包含 `cancel / amend / expire / partial fill / full fill`
- `WAIT-UNTIL-FILL` 承接的是执行中的订单或订单组状态；对组合交易计划，必须允许其中不同部分分别处在不同成交状态
- 当前产品先按“辅助用户决策并记录下单”设计；`执行建议` 不等于 `真实已生效`
- `WAIT-CONDITION` 和 `WAIT-UNTIL-FILL` 是两种不同等待：前者等计划触发，后者等订单成交
- `WAIT-UNTIL-FILL` 表达的是“已知有挂单在等待成交”，`RECOVERY` 表达的是“当前真实状态还需要重新对齐”
- `MANAGE` 不是单点，而是持仓语境下再次运行一轮 `OBS-POS <-> PLAN-POS`
- 持仓中的动作要区分成“继续持仓动作”和“平仓动作”
- 继续持仓动作成交后回到 `MANAGE`，平仓全部成交或阶段性闭合后先进入 `REVIEW-QUEUE`
- `REVIEW-QUEUE` 用来承接已闭链但暂未复盘的交易链，`REVIEW` 可以立刻触发，也可以后续再拉起
- `RESEARCH` 不是只产出策略的单一路径，而是横向服务主线的输入源
- `RESEARCH` 的输出至少分成三类：服务 `观察/决策` 的市场语境与 thesis、服务 `管理` 的持仓监控与反证、进入 `BACKTEST` 的候选策略
- `REVIEW` 和 投研 都能产出候选策略，统一进入 `BACKTEST`
- `REVIEW` 除了给 `BACKTEST` 提供候选策略，也可以把复盘结论直接回接下一轮 `OBSERVE / PLAN`
- `REVIEW -> OBSERVE / PLAN` 用来承接复盘后的快速重入、原方向结束后的新方向重开，但这不等于长期策略版本已经切换
- 主线第 8 阶段现在是 回测，不再叫 验证
- 回放 不是回测本身；它首先是在锁住未来信息后的历史语境里重跑一轮 `OBSERVE / PLAN / MANAGE / REVIEW`
- 回放 也可以给 回测 提供可作用策略的市场语境
- `BACKTEST` 既可以把策略作用于用户历史交易，也可以把策略作用于回放语境
- 投研 回接主线时补的是研究结论和判断语境，不直接替代 `RECOVERY` 对真实状态的对齐
- 回放 复用的是主线判断语义，但不会直接改写真实主线中的持仓、订单和阶段状态
- `ITERATE` 不只是产出新策略，还要明确下一轮主线是回到 `OBSERVE / PLAN`，还是先进入 `RECOVERY`

### 1.2 主线与支线接点

- 主线负责围绕真实交易持续推进
- 主线里的正式计划对象是 `交易计划`；必要时它表现为组合计划，并在执行层展开成一组订单或多条腿
- 主线里的 `观察 / 计划 / 管理` 正常都要读取当前生效策略；当前文档先不展开策略格式，只确认这层依赖存在
- `恢复` 负责在重新进入系统、对话中断、执行未决或状态不清时先对齐真实状态
- `等待条件成熟` 负责承接计划已存在但尚未进入下单的状态
- 已结束或阶段性闭合的交易链先进入 `待复盘池`，不要求立即复盘
- 投研 不依赖当前持仓，也可以被主线中的具体问题触发
- 投研 负责把外部材料、长期线索、反证和候选方法整理成可复用研究结论
- 投研 的输出至少包括：服务 `观察` 的市场语境、服务 `决策` 的 thesis/风险提示、服务 `管理` 的持仓监控/反证、以及可进入 `回测` 的候选策略
- `复盘` 和 投研 共享“候选策略 -> 回测”这条入口，但投研 还可以直接回接主线
- `复盘` 除了进入 `回测`，也可以把即时复盘结论直接回接下一轮 `观察 / 计划`
- 回放 是独立支线，负责把当前问题映射到历史时间锚点，并在锁住未来信息后重跑一轮历史语境下的 `观察 / 决策 / 管理 / 复盘`
- 回放 也可以向 `回测` 提供可供策略作用的市场语境
- 回测 是统一回测口，既可以把策略作用于用户历史交易，也可以把策略作用于回放语境
- `迭代` 只能接在 `回测` 之后
- `迭代` 负责确定当前生效策略，并明确下一轮主线的回接落点：继续观察、快速重入，或先经过恢复进入分级恢复路径
- `infra` 不定义业务流转，只承载前面已经确定的业务对象和状态

### 1.4 固定回返

| 场景 | 回返关系 |
| --- | --- |
| 重新进入系统、对话中断、状态不清 | `任意阶段 -> 恢复 -> 观察/条件等待/成交等待/管理` |
| 外部材料、长期线索或反证需要补到开仓前判断 | `投研 -> 观察/决策` |
| 需要在历史时间点重跑一轮开仓前判断 | `回放 -> 观察/决策` |
| 开仓前反复判断 | `观察 <-> 决策` |
| 开仓前继续等待计划触发 | `决策 -> 条件等待` |
| 不建议操作、放弃计划 | `决策 -> 观察` |
| 条件触发、歧义触发、机会错过、被替代 | `条件等待 -> 观察/决策` |
| 撤单、改单、失效 | `执行/等待 -> 观察/决策` |
| 用户反馈与已记录状态不一致 | `执行/等待/管理 -> 恢复` |
| 持仓中出现新的外部语境、反证或风险提示 | `投研 -> 管理 -> 观察/决策/执行` |
| 回放推进时间后继续处理历史仓位 | `回放 -> 管理 -> 观察/决策/执行` |
| 持仓中继续判断 | `管理 -> 观察/决策/执行` |
| 持仓中再次挂单 | `管理 -> 执行 -> 等待 -> 管理/观察` |
| 已结束或阶段性闭合 | `管理 -> 待复盘池 -> 复盘` |
| 复盘后基于新证据快速重入 | `复盘 -> 观察/决策` |
| 原方向结束后开启新方向 | `管理 -> 复盘 -> 观察/决策` |
| 迭代确认新的当前生效策略 | `迭代 -> 观察/决策` |
| 暂停后的分级恢复 | `复盘/迭代 -> 恢复 -> 观察/管理` |
| 回放中的历史链路阶段性结束 | `回放 -> 复盘` |
| 投研 产出候选方法并需要系统验证 | `投研 -> 回测` |

### 1.5 固定约束

- 用户下次回来必须能继续当前交易链
- 当前产品先按“辅助用户决策并记录下单”设计，不默认系统直接代用户下单
- 系统建议、用户确认、真实已生效状态是三层不同东西；未对齐前不能直接视为已执行完毕
- 对话中断、用户自行操作和持久化上下文缺口是常态，必须通过 `恢复` 重新对账
- 后续章节设计都要回答“谁触发、接到哪里、给谁输出”
- 主线判断、等待和管理的默认操作对象是 `交易链 + 当前交易计划`，不是单订单
- 主线里的交易计划默认来自“当前观察语境 + 当前生效策略”，不是长期允许 `OBSERVE` 脱离策略单独产出计划
- 订单变化会反向影响主线所处阶段
- `等待条件成熟` 和 `订单等待成交` 是两种不同等待，不能混成一个状态
- `结束旧方向` 和 `开启新方向` 必须是两个判断，不能在流程上被写成单一“反手动作”
- 投研 是横向能力，不替代主线阶段；它回接主线时只能补充研究结论、风险提示和候选动作，不能直接改写真实持仓、订单和阶段状态
- 回放 必须锁住未来信息；回放中的判断链只服务历史验证和学习，不直接改写真实主线中的状态
- 已结束或阶段性闭合的交易链必须先进入 `待复盘池`；`复盘` 可以即时触发，也可以后续触发
- 复盘不会直接变长期规则，必须先进入回测
- 迭代只能发生在回测之后
- `复盘` 可以直接回接下一轮主线，但直接回接的是复盘结论，不是绕过 `回测 -> 迭代` 直接切换长期策略版本

### 1.6 仍需继续回答的前置问题

- 一条交易链何时开始、何时闭合；多次加减仓、回补、再对冲或换周期后何时还是同一条链
- 当前生效策略的最小格式是什么；系统在没有显式策略版本时如何提供默认策略
- `条件等待` 与 `成交等待` 各需要记录到什么粒度，才能支撑回返和恢复
- 一个交易计划何时只是单机会计划，何时需要升格为组合计划
- 组合交易计划、对冲和订单组在执行层需要记录到什么粒度，才能支撑 `WAIT-UNTIL-FILL` 和 `RECOVERY`
- 用户口头反馈需要细到什么粒度，才能支撑恢复时的可信对账
- `管理` 何时算阶段性结束，何时进入 `复盘`
- 投研 结论如何区分“直接服务主线”与“升格进入回测”
- 除 `观察/决策/管理/回测` 外，投研 是否还需要直接服务 `复盘`，先不固定
- 回放 中的历史链路与真实主线链路如何并存又不混淆

## 2. 观察

### 输入

- 当前市场信息
- 当前交易链上下文
- 当前生效策略
- 当前交易计划上下文
- 上一个阶段返回的状态
- 来自 `投研` 的市场语境、观察重点或风险提示
- 来自 `回放` 的历史时间锚点和锁住未来信息后的市场语境

### 输出

- 观察记录
- 进入决策的观察基础
- 候选机会列表或需要纳入同一计划判断的机会集合

### 管理对象

- 交易链
- 策略
- 交易计划
- 阶段状态
- 判断记录

### 流程交互

- 在主线中负责给 `决策` 提供当前语境
- 来自流程控制定义的起点或回返点
- 可接入 `投研` 回接的市场语境和观察重点
- 可在 `回放` 中以历史语境方式再次发生
- 进入 `决策`
- 可被 `决策` 打回
- 可在 `管理` 中再次发生

### 需要详细设计

- 观察记录记录什么
- 观察如何挂到当前交易链
- 观察阶段如何读取当前生效策略
- 多个候选机会如何挂到同一条链和同一个计划语境
- 如何恢复到当前观察语境
- 投研 结论如何映射成当前观察语境
- 回放 里的历史观察语境如何与真实观察记录区分

## 3. 决策

### 输入

- 观察记录
- 当前交易链上下文
- 当前生效策略
- 当前交易计划上下文
- 持仓中的再判断需求
- 来自 `投研` 的 thesis、反证和风险提示
- 来自 `回放` 的历史语境下观察结果

### 输出

- 操作建议
- 不操作建议
- 放弃计划
- 进入条件等待的计划决定
- 进入执行的动作决定
- 当前交易计划或组合计划

### 管理对象

- 交易链
- 策略
- 交易计划
- 阶段状态
- 判断记录

### 流程交互

- 来自 `观察`
- 可直接消费 `投研` 提供的研究结论
- 可在 `回放` 中基于历史语境形成动作判断
- 进入 `执行` 或 `WAIT-CONDITION`
- 可回退到 `观察`
- 可在 `管理` 中再次发生

### 需要详细设计

- 决策记录记录什么
- 决策与观察的往返如何表达
- 当前生效策略如何约束计划生成
- 决策与 `WAIT-CONDITION` 的往返如何表达
- 不同决策结果如何影响后续流转
- 多机会何时收敛成一个组合计划，何时拆成多个互斥计划
- 投研 结论与观察记录冲突时如何处理
- 回放 中的历史决策如何与真实主线决策区分

## 4. 执行

### 输入

- 决策结果
- 当前交易计划
- 用户确认的执行意图
- 管理中的再执行动作

### 输出

- 订单记录
- 订单组或组合部分状态记录
- 执行动作记录
- 等待成交态、恢复所需状态或成交结果

### 管理对象

- 交易链
- 交易计划
- 阶段状态
- 订单

### 流程交互

- 来自 `决策` 或 `管理`
- 进入等待态、`恢复`、`管理` 或返回 `观察/决策`
- 可在 `管理` 中重复发生

### 需要详细设计

- 执行建议、用户反馈和订单生命周期如何区分
- 执行只负责订单层等待，不负责条件等待
- 组合交易计划如何展开成订单组、执行顺序和部分成交状态
- 撤单、改单、失效如何回写主线
- 执行结果如何保持链不断开

## 5. 恢复

### 输入

- 用户口头反馈的当前真实状态
- 系统已持久化但可能不完整的上下文
- 当前市场信息
- 当前交易计划或订单组上下文

### 输出

- 已对齐的当前状态
- 待确认项或可信度判断
- 继续进入 `观察`、条件等待、等待成交或 `管理` 的落点
- 分级恢复下的当前恢复层级

### 管理对象

- 交易链
- 交易计划
- 阶段状态
- 持仓
- 订单
- 判断记录

### 流程交互

- 来自重新进入系统、对话中断后的继续、条件等待未决、等待成交未决或管理态失配
- 可承接 `迭代` 给出的暂停后恢复路径
- 进入 `观察`、条件等待、等待成交或 `管理`
- 可在任意状态不清时再次发生

### 需要详细设计

- 哪些状态必须重新向用户确认
- 用户口头反馈如何覆盖或修正旧上下文
- 组合计划失配、部分成交和对冲未补齐时如何恢复到当前计划
- `继续观察 / 影子恢复 / 小仓恢复 / 恢复常规` 如何映射到恢复后的落点
- 恢复后如何判定下一步落点

## 6. 管理

### 输入

- 已成交订单
- 当前持仓
- 当前生效策略
- 当前交易计划
- `恢复` 后对齐出来的当前状态
- 持仓中的新增判断
- 来自 `投研` 的持仓监控点、反证或调整提示
- 来自 `回放` 推进后的历史持仓和历史订单语境

### 输出

- 管理动作
- 新订单
- 更新后的交易计划
- 进入待复盘池的交易结果

### 管理对象

- 交易链
- 策略
- 交易计划
- 阶段状态
- 持仓
- 订单
- 判断记录

### 流程交互

- 来自 `执行` 或 `恢复`
- 可接入 `投研` 回接的持仓中研究结论
- 可在 `回放` 中继续推进历史持仓管理
- 内部可再次触发 `观察`、`决策`、`执行`
- 结束后进入 `待复盘池`

### 需要详细设计

- 持仓未结束但动作持续发生时如何表达
- 加仓、减仓、调杠杆、止损、止盈如何挂回同一交易链
- 当前生效策略如何约束持仓中的计划更新
- 计划内分批建仓、对冲补齐和情绪化摊平如何区分
- 管理何时算阶段性结束
- 投研 产出的风险提示和反证如何挂回当前持仓
- 交易身份漂移何时只是计划更新，何时应视为新链
- 回放 中的历史管理动作如何与真实管理动作区分

## 7. 复盘

### 输入

- 来自待复盘池的已结束或阶段性闭合的交易链
- 来自 `回放` 的阶段性闭合历史链路
- 交易记录
- 判断记录

### 输出

- 复盘记录
- 新策略或策略修正
- 可回接下一轮主线的复盘结论

### 管理对象

- 交易链
- 待复盘池
- 交易记录
- 判断记录
- 策略

### 流程交互

- 来自 `管理` 进入的待复盘池
- 可由 `回放` 中阶段性结束的历史链路触发
- 从待复盘池中被拉起
- 进入 `回测`
- 可直接回接下一轮 `观察 / 计划`
- 与 投研 一起构成两个策略入口

### 需要详细设计

- 哪些结果进入待复盘池
- 待复盘池如何筛选和拉起
- 复盘记录保留什么
- 复盘产物如何进入回测
- 哪些复盘结论只回接下一轮主线，哪些要继续推动策略迭代
- 回放 里的复盘样本如何与真实闭链样本区分

## 8. 回测

### 输入

- 来自 `复盘` 的策略
- 来自 投研 的策略
- 来自 回放 或用户历史交易的验证集

### 输出

- 回测结果
- 进入迭代的候选内容
- 继续回测所需状态

### 管理对象

- 策略
- 验证集
- 回测结果

### 流程交互

- 来自 `复盘` 和 投研
- 使用 回放 或用户历史交易
- 进入 `迭代`

### 需要详细设计

- 两个策略入口如何共用同一回测口
- 每次回测如何绑定验证集来源
- 回测结果如何沉淀进数据库

## 9. 迭代

### 输入

- 回测结果
- 已进入后续使用的策略内容

### 输出

- 下一轮可使用的策略结果
- 可回接主线的当前生效策略版本
- 下一轮主线回接落点

### 管理对象

- 策略
- 回测结果

### 流程交互

- 来自 `回测`
- 重新进入后续主线使用或后续策略流程
- 为后续 `观察 / 计划 / 管理` 提供当前生效策略
- 可显式回接 `观察 / 计划` 或 `恢复`

### 需要详细设计

- 哪些回测结果能进入迭代
- 迭代后的内容如何重新被主线使用
- 当前生效策略如何切换、回退或保留旧版对照
- `继续观察 / 影子恢复 / 小仓恢复 / 恢复常规` 如何由迭代结果决定

## 10. 投研

### 输入

- 用户提出的研究问题
- 用户提出的假设
- agent 发散出的假设
- 社区流传的策略
- 网络、论坛、论文里的公开策略
- 宏观、跨资产、监管、地缘或市场结构材料
- 当前主线里需要补充解释的标的、thesis 或风险点

### 输出

- 可直接服务 `观察` 的市场语境和观察重点
- 可直接服务 `决策` 的 thesis、触发条件、风险提示和反证
- 可直接服务 `管理` 的持仓监控点、失效线索和调整提示
- 可进入回测的候选策略

### 管理对象

- 研究任务
- 研究结论
- 候选策略

### 流程交互

- 不依赖当前持仓，也可由主线中的具体问题触发
- 可直接回接 `观察`、`决策` 和 `管理`
- 其中一部分研究结论继续整理成候选策略后进入 `回测`
- 与 `复盘` 共享“候选策略 -> 回测”入口

### 需要详细设计

- 零散想法如何进入系统
- 研究任务如何绑定到具体标的、thesis 或交易链
- 想法如何区分成“直接服务主线的研究结论”和“可回测策略”
- 主线使用研究结论时如何保留来源、时点和失效条件
- 如何与复盘产物共用同一回测入口

## 11. 回放

### 输入

- 历史数据
- 起始时间点
- 当前问题映射到的历史案例或时间锚点

### 输出

- 提供给 `观察 / 决策` 使用的历史市场语境
- 提供给 `管理` 使用的历史持仓推进语境
- 提供给 `复盘` 使用的历史链路样本
- 提供给回测阶段使用的回放语境和验证集

### 管理对象

- 回放会话
- 历史时间锚点
- 回放语境
- 验证集

### 流程交互

- 不生成策略
- 不直接产生回测结果
- 可在锁住未来信息后驱动一轮历史语境下的 `观察 / 决策 / 管理 / 复盘`
- 也可向 `回测` 提供市场语境和验证集支持

### 需要详细设计

- 回放从什么时间点开始
- 回放 如何锁住未来信息并逐步推进时间
- 回放 如何驱动 `观察 / 决策 / 管理 / 复盘` 的历史循环
- 回放 向回测输出什么
- 回放验证集如何与用户历史交易并存
- 回放会话如何与真实主线状态隔离
