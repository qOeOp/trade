# PRD

## 文档定位

本文件基于 `docs/vision.md`，只回答当前阶段产品“做什么”。

当前先产出 PRD 骨架，用来承接后续逐模块细化，不提前写成设计文档、技术方案或长期制度。

## 当前阶段原则

- 只围绕 `vision` 已明确的方向下钻
- 只写产品能力与范围，不写实现方式
- 不预先固定 skill 数量、skill 职责、系统模块和记录模型
- 不把临时想法提前固化成长期规则

## 产品目标

产品要解决的核心问题是：

- 让一次交易从观察、决策、执行、管理、复盘到验证、迭代保持连续
- 让用户下次回到同一标的或同一策略时，不需要从零重建思路
- 让策略可以被持续生成、验证和迭代，而不是停留在零散想法

## 目标用户

当前核心用户是长期生活在 agent 工作区里的自用型交易者：

- 日常通过 Claude Code、Codex、Gemini CLI 等 agent 工作
- 盯少数标的，持有真实仓位
- 有分析能力，但缺少连续跟踪判断的产品形态

## 产品范围

### 当前范围

- 加密交易作为主战场
- 产品主体是一组运行在 agent 工作区里的高级 skill
- 产品核心先围绕交易链不断裂展开
- 在核心能力之上，后续可以承接统一持久化资产与看板视图

### 当前不展开

- skill 具体拆分方式
- 系统架构与模块边界
- 数据结构、接口协议、存储方案
- 自动化调度实现
- 面向泛用户的 SaaS 化产品设计

## 产品结构总览

当前 PRD 按三层来讨论：

1. 产品主线：围绕真实交易持续运转的主链路
2. 产品支线：`r&d agent` 与 `backtest`
3. 资产与视图：长期沉淀的内容，以及后续可视化查看方式

## 一、产品主线

### 1.1 主线目标

主线要解决的问题是：一笔真实交易从开始形成判断，到最后复盘结束，中间不能断。

### 1.2 主线阶段

当前主线按以下阶段理解：

1. 观察
2. 决策
3. 执行
4. 管理
5. 复盘
6. 验证
7. 迭代

补充说明：

- 这七个阶段是产品主线，不代表用户行为一定线性前进
- 观察与决策之间会反复来回
- 持仓后的管理仍会继续观察、继续决策，并产生新的执行动作
- 订单本身也有独立生命周期，会影响主线当前所处阶段

### 1.3 主线最小闭环

PRD 细化时，主线至少需要回答以下问题：

- 用户如何发起一次新的观察或决策
- 系统如何承接当前判断，而不是只接住一段聊天
- 执行动作如何挂回同一条链路
- 管理过程中的变化如何持续记录
- 一条链何时可以进入复盘
- 复盘产出的内容如何进入验证
- 验证通过后的内容如何进入迭代

### 1.4 主线模块骨架

后续主线可按以下模块继续细化：

#### 模块 A：观察

- 要回答什么：用户正在看什么，为什么值得继续看
- 当前先不展开：观察对象结构、筛选规则、信息源接入方式
- 后续要细化：观察的进入条件、退出条件、与决策的连接方式

#### 模块 B：决策

- 要回答什么：当前判断是什么，是否值得行动，为什么
- 当前先不展开：判断模板、字段结构、置信度表达方式
- 后续要细化：正式判断的最小要素、观望结论的表达方式

#### 模块 C：执行

- 要回答什么：判断如何变成真实下单、改单、撤单、成交等动作
- 当前先不展开：订单模型、交易所集成、执行接口
- 后续要细化：执行动作如何挂回链路、订单状态如何影响主线

#### 模块 D：管理

- 要回答什么：持仓后如何持续跟踪，而不是放着不管
- 当前先不展开：提醒策略、自动回拉机制、监控规则
- 后续要细化：管理中的观察、决策、执行如何循环发生

#### 模块 E：复盘

- 要回答什么：问题出在判断、执行还是运气
- 当前先不展开：复盘模版、归因框架、评分规则
- 后续要细化：复盘触发条件、复盘最小输出、复盘与主线关系

#### 模块 F：验证

- 要回答什么：复盘或新生成的策略是否真的成立
- 当前先不展开：验证方法、样本标准、验证门槛
- 后续要细化：不同阶段分别依赖什么验证来源

#### 模块 G：迭代

- 要回答什么：验证通过后的内容如何进入下一轮策略演化
- 当前先不展开：迭代周期、迭代制度、自动升级逻辑
- 后续要细化：什么可以进入迭代，什么仍停留在局部经验

## 二、产品支线

### 2.1 支线存在的原因

支线不是为了把系统做大，而是为了补足冷启动问题：

- 一开始没有足够多的真实交易记录
- 一开始也未必已经有成熟策略

所以主线之外，还需要两条可以独立运行、但最终能接回主线的支线。

### 2.2 支线一：`r&d agent`

`r&d agent` 的角色是生成策略。

当前骨架先明确三件事：

- 输入可以来自用户假设、agent 发散、社区策略、公开资料等来源
- 输出应该是可继续验证的策略，而不是零散灵感
- 它可以独立运行，但最终要接回统一的策略生成与验证过程

后续要细化的问题：

- 什么样的输出才算“一个策略”
- `r&d agent` 和主线复盘生成策略的关系如何定义
- 研究阶段与正式进入验证阶段的分界点是什么

### 2.3 支线二：`backtest`

`backtest` 的角色是为验证提供早期可用数据。

当前骨架先明确三件事：

- 当真实交易样本不足时，历史数据回测是验证来源之一
- `backtest` 不伪装成真实交易，只服务验证
- 它既可以承接主线复盘产出的策略，也可以承接 `r&d agent` 产出的策略

后续要细化的问题：

- 何时优先用回测，何时优先看真实交易记录
- 回测输出至少应包含哪些结论
- 回测结果如何回流到统一验证结论里

## 三、资产层骨架

### 3.1 需要沉淀的资产类型

基于 `vision`，当前至少可以先承认三类资产存在：

- 交易链资产：围绕一次次真实交易持续积累的判断、执行、管理、复盘内容
- 策略资产：可被继续验证和迭代的策略内容
- 验证资产：来自真实交易记录或历史测试数据的验证结果

### 3.2 当前只定义到这里

当前先不定义：

- 每类资产的正式字段
- 资产之间的具体引用关系
- 资产如何存储、同步、版本化

后续 PRD 细化时，需要分别回答：

- 每类资产的最小单位是什么
- 哪些资产属于短期上下文，哪些属于长期沉淀
- 用户修改、查看、删除这些资产时的基本动作是什么

## 四、视图层骨架

### 4.1 视图层目标

在 agent 能力层之上，后续可以有一个单独看板作为前端视图。

这个看板的目标不是替代 agent，而是：

- 展示长期积累的资产
- 支持不同视图查看
- 支持围绕资产做基本增删改查

### 4.2 当前只保留占位

当前先不展开：

- 看板信息架构
- 页面结构与交互方式
- 移动端和桌面端的具体形态

后续 PRD 细化时，可以按以下角度继续拆：

- 看什么
- 怎么切换视图
- 哪些操作必须在看板完成，哪些仍留在 agent 内完成

## 五、关键关系骨架

### 5.1 主线与支线的关系

当前先固定以下关系：

- 主线解决真实交易怎么连续跟下去
- `r&d agent` 解决没有当前持仓时，怎么继续生成策略
- `backtest` 解决实盘样本不足时，怎么先让验证跑起来
- 三者最终汇入同一个策略生成、策略验证、策略迭代过程

### 5.2 本阶段不固定的关系

当前先不固定：

- 三者之间的精确流转规则
- 谁先触发谁、谁依赖谁的制度化流程
- 每一步是否必须经过人工确认

## 六、范围边界

### 6.1 当前明确纳入

- 加密交易主链路
- 围绕真实仓位与真实判断的连续跟踪
- 从策略生成到验证、迭代的最小产品闭环

### 6.2 当前明确不纳入

- 其他市场的完整正式交易链路
- 完整独立 app 替代 agent 工作区
- 提前写死的长期组织结构

## 七、成功标准骨架

当前先只保留指标讨论框架，具体口径后续再收敛。

后续至少要覆盖三类指标：

- 连续性指标：交易链是否真的没有断
- 产出指标：策略是否持续被生成与验证
- 演化指标：验证后的结论是否真的进入下一轮迭代

可继续细化的方向：

- 高意图问题被承接的比例
- 一条链从开始到复盘的完成比例
- 复盘结果进入验证的比例
- 验证结果进入后续迭代的比例

## 八、后续拆解顺序建议

为了避免一次把 PRD 写得过实，后续建议按下面顺序逐段细化：

1. 先细化产品主线的最小闭环
2. 再细化 `r&d agent`
3. 再细化 `backtest`
4. 然后细化资产层
5. 最后再讨论视图层

## 九、待确认问题

以下问题保留到后续模块讨论时再定：

- 主线里“最小可用的一条链”到底至少包含什么
- 一条链与一个订单、一次持仓、一个策略之间的关系怎么定义
- 复盘产出的内容，到什么程度才算进入验证
- 策略验证通过后，什么才算真正进入迭代
- 看板在当前阶段是必要项，还是后置项
