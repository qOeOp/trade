# PRD

## 1. 流程控制

### 1.1 固定主线

```text
================ ENTRY / RESUME ================

  [RESUME] -> [RECOVERY] -> [OBSERVE / PLAN]
                          -> [WAIT-CONDITION]
                          -> [WAIT-UNTIL-FILL]
                          -> [MANAGE]

================ PRE-TRADE =====================

  [OBSERVE] <-> [PLAN] <-> [ALIGN]
                              +--> revise / override -> [PLAN]
                              +--> hard block / unclear -> [RECOVERY]
                              +--> [WAIT-CONDITION] -> valid trigger -> [PLAN]
                              |                    -> ambiguous / missed / replace /
                              |                       delete / env change -> [OBSERVE / PLAN]
                              |                    -> continue wait
                              +--> [DROP]  -> [OBSERVE / PLAN]
                              +--> [MKT]   -> [IN-POS]
                              +--> [PLACE] -> [WAIT-UNTIL-FILL]
                                               +--> cancel / amend / expire -> [OBSERVE / PLAN]
                                               +--> partial fill            -> [IN-POS]
                                               +--> full fill               -> [IN-POS]

================ IN-POSITION ===================

  [MANAGE] -> [OBS-POS] <-> [PLAN-POS] <-> [ALIGN-POS]
                                           +--> revise / override -> [PLAN-POS]
                                           +--> hard block / unclear -> [RECOVERY]
                                           +--> [ADD / REDUCE / LEV / SL / TP]
                                           |    +--> [EXEC]  -> [MANAGE]
                                           |    +--> [PLACE] -> [WAIT-UNTIL-FILL]
                                           |                     +--> cancel / amend / expire
                                           |                     |    -> [OBS-POS / PLAN-POS]
                                           |                     +--> partial fill            -> [MANAGE]
                                           |                     +--> full fill               -> [MANAGE]
                                           +--> [CLOSE]
                                                +--> [EXEC]  -> [REVIEW-QUEUE]
                                                +--> [PLACE] -> [WAIT-UNTIL-FILL]
                                                                     +--> cancel / amend / expire
                                                                     |    -> [OBS-POS / PLAN-POS]
                                                                     +--> partial fill            -> [MANAGE]
                                                                     +--> full fill               -> [REVIEW-QUEUE]

  [CLOSED / PAUSED / NO-MORE-ACTION] -> [REVIEW-QUEUE]

================ POST-TRADE ====================

  [REVIEW-QUEUE] -> [REVIEW] -> [STRATEGY-POOL] -> [BACKTEST] -> [ITERATE]
                     |             |                           |
                     |             +--> [OBSERVE / PLAN]      +--> [STRATEGY-POOL]
                     |             +--> [MANAGE]              +--> [RECOVERY]
                     +--> [OBSERVE / PLAN]
                     +--> [RECOVERY]

  [RESEARCH] -> [STRATEGY-POOL]
      |
      +--> [OBSERVE / PLAN]
      +--> [MANAGE]

  [REPLAY] -> [OBSERVE / PLAN]
      |
      +--> [MANAGE]
      +--> [REVIEW]
      +--> [BACKTEST]

  [BACKTEST]
      +--> apply to user history
      +--> apply to replay
```

这里表达的不是“章节顺序”，而是固定流程骨架。读法也不要按章节往下读，而是先看阶段，再看阶段内怎么流转。

#### 先看阶段

- 图里节点改成英文，只是为了保证 ASCII 可读；具体业务含义以下面规则为准。
- `ENTRY / RESUME -> RECOVERY` 是统一入口。每次重新进入系统，都先对齐真实状态，再决定当前交易链落回 `OBSERVE / PLAN`、`WAIT-CONDITION`、`WAIT-UNTIL-FILL` 或 `MANAGE`。
- `PRE-TRADE` 对应开仓前阶段。核心不是线性往前走，而是在 `OBSERVE <-> PLAN <-> ALIGN` 之间反复收敛；计划成立但条件未到时进入 `WAIT-CONDITION`，放弃则回到 `OBSERVE / PLAN`，`MKT` 直接进入 `IN-POS`，只有 `PLACE` 才进入 `WAIT-UNTIL-FILL`。
- `IN-POSITION` 对应持仓中阶段。`MANAGE` 不是单点，而是持仓语境下重跑一轮 `OBS-POS <-> PLAN-POS <-> ALIGN-POS`；加仓、减仓、调杠杆、止损、止盈等动作执行后回到 `MANAGE`，完整平仓或阶段性闭合后进入 `REVIEW-QUEUE`。
- `POST-TRADE` 对应平仓后与离线演化阶段。`REVIEW-QUEUE -> REVIEW -> STRATEGY-POOL -> BACKTEST -> ITERATE` 负责把已闭链样本变成候选策略、验证结果和下一轮可用版本；其中 `STRATEGY-POOL` 是长期对象，不是主线阶段。
- `RESEARCH` 和 `REPLAY` 是两条可独立运行、也能回接主线的支线。`RESEARCH` 是横向输入源，`REPLAY` 是锁住未来信息后的历史语境重跑；两者都不等于 `BACKTEST`。

#### 再看阶段内流转

- 图里的主操作对象是 `交易链` 下的当前交易计划，不是单订单；订单只是执行层结果。`PLAN` / `PLAN-POS` 对应后文章节里的 `决策`，其输出应是当前观察语境和当前适用策略分支版本共同推导出的当前交易计划。
- `ALIGN` / `ALIGN-POS` 是正式分歧处理点，不是备注位。它必须承接 `接受建议`、`软分歧后改计划继续`、`硬性阻断`、`状态不清需恢复`，并同时保留 `系统建议`、`用户确认`、`最终执行计划`。
- `WAIT-CONDITION` 和 `WAIT-UNTIL-FILL` 是两种不同等待。前者表示“计划仍有效，但条件还没成熟，暂时还没有挂单”；后者表示“已知有挂单在等待成交”，并且必须显式覆盖 `cancel / amend / expire / partial fill / full fill`。
- `REVIEW-QUEUE` 承接已闭链但暂未复盘的交易链；`REVIEW` 既能把结论直接回接下一轮 `OBSERVE / PLAN`，也能产出候选策略。`REVIEW` 和 `RESEARCH` 产出的候选策略都先进入 `STRATEGY-POOL`，再统一进入 `BACKTEST`。
- `BACKTEST` 是统一验证口，既可作用于用户历史交易，也可作用于回放语境；`ITERATE` 负责把结果回写 `STRATEGY-POOL`，并明确下一轮应该直接回到 `OBSERVE / PLAN`，还是先经 `RECOVERY` 再进入主线。

先把这张图定下来，后面 2-11 章再分别定义自己在这张图里负责什么、接什么输入、给谁输出。

### 1.2 主线与支线接点

| 对象 / 支线 | 角色 | 接点规则 |
| --- | --- | --- |
| 主线 | 围绕真实交易持续推进 | 正式操作对象是 `交易计划`；必要时可表现为组合计划，并在执行层展开成订单组或多条腿 |
| 主线读取策略 | 给 `观察 / 决策 / 管理` 提供当前可用规则 | 正常都读取 `策略池` 中当前语境下适用的生效分支版本；当前先不展开更细策略格式 |
| `对齐 / 分歧处理` | 承接系统建议与用户确认不一致 | 至少区分 `接受建议`、`软分歧后改计划继续`、`硬性阻断`、`状态不清需恢复` |
| `恢复` | 对齐真实状态并决定回接落点 | 用于重新进入系统、对话中断、执行未决或状态不清；事实源优先取交易所 API，用户反馈只补充 API 看不到的意图、场外动作、状态解释和异常说明 |
| `等待条件成熟` | 承接计划已存在但尚未进入下单 | 属于计划级等待，不等于订单等待成交 |
| `待复盘池` | 承接已结束或阶段性闭合的交易链 | 先入池，不要求立即复盘 |
| `策略池` | 管理长期策略分支和版本 | 不是新的主线阶段；当前只确认至少支持按 `交易标的` 或 `市场环境` 分支，并在分支内管理版本 |
| 投研 | 横向研究输入源 | 不依赖当前持仓，也可被主线中的具体问题触发；输出至少包括服务 `观察` 的市场语境、服务 `决策` 的 thesis/风险提示、服务 `管理` 的持仓监控/反证，以及可进入 `策略池` 的候选策略 |
| `复盘` | 抽取闭链样本，生成复盘结论 | 可把即时复盘结论直接回接下一轮 `观察 / 决策`；若产出候选策略，则走 `候选策略 -> 策略池 -> 回测` |
| 回放 | 独立历史语境支线 | 把当前问题映射到历史时间锚点，并在锁住未来信息后重跑一轮历史语境下的 `观察 / 决策 / 管理 / 复盘`；也可向 `回测` 提供市场语境 |
| 回测 | 统一回测口 | 既可把策略作用于用户历史交易，也可作用于回放语境 |
| `迭代` | 回写策略版本并决定主线回接落点 | 只能接在 `回测` 之后；负责更新 `策略池` 中当前生效分支版本，并明确下一轮是继续观察、快速重入，还是先经 `恢复` |
| `infra` | 底层承载层 | 不定义业务流转，只承载前面已经确定的业务对象和状态 |

### 1.3 章节在工作流中的位置

| 章节 | 在整体工作流中的职责 | 主要输入来自 | 主要输出给 |
| --- | --- | --- | --- |
| `2. 观察` | 为当前交易链补充最新市场语境和候选机会基础；在回放中则负责生成历史时间点下的观察语境 | 市场信息、当前交易链、`STRATEGY-POOL` 中当前适用的生效分支版本、上个阶段回返状态、`10. 投研` 的研究结论、`11. 回放` 的历史语境 | `3. 决策` |
| `3. 决策` | 基于观察结果和当前适用策略分支形成当前交易计划；必要时把多个机会收敛成一个组合计划 | `2. 观察`、`STRATEGY-POOL` 中当前适用的生效分支版本、`6. 管理` 中的再判断需求、`10. 投研` 的 thesis/风险提示 | `对齐 / 分歧处理`，再进入 `4. 执行`、`WAIT-CONDITION` 或回退到 `2. 观察` |
| `4. 执行` | 把分歧处理后的交易计划转成用户要执行或已反馈的订单、订单组或组合部分状态，并持续接收交易所 API 回报 | `3. 决策` 经 `对齐 / 分歧处理` 确认后的计划、`6. 管理` 中经 `对齐 / 分歧处理` 确认后的再执行动作、交易所 API 返回的订单与成交状态 | `WAIT-UNTIL-FILL`、`5. 恢复`、`6. 管理` 或回返 `2/3` |
| `5. 恢复` | 在重新进入系统、状态不清、执行未决、硬性阻断或暂停后恢复时对齐真实状态，并决定当前链和当前计划从哪里继续 | 交易所 API 拉取的订单 / 持仓 / 余额 / 成交状态、已持久化上下文、用户补充反馈、当前市场与账户语境、`对齐 / 分歧处理` 里的阻断结果、`9. 迭代` 给出的恢复门槛 | `2. 观察`、`WAIT-CONDITION`、`WAIT-UNTIL-FILL` 或 `6. 管理` |
| `6. 管理` | 在持仓/已成交语境下持续推进链路和当前计划；在回放中则负责推进历史仓位的后续动作 | `4. 执行` 的成交结果、`5. 恢复` 的对齐结果、当前持仓、`STRATEGY-POOL` 中当前适用的生效分支版本、`10. 投研` 的管理提示、`11. 回放` 推进后的历史持仓语境 | `2/3/4`、`对齐 / 分歧处理` 或 `7. 复盘` |
| `7. 复盘` | 从待复盘池或回放中的阶段性闭合链路里拉起样本，产出策略结论和下一轮主线判断 | `6. 管理` 闭链结果、待复盘池、`11. 回放` 中的历史链路 | `STRATEGY-POOL`、`2. 观察` 或 `5. 恢复` |
| `8. 回测` | 统一接收从策略池中选定的候选策略，并把策略作用于用户历史交易或回放语境做回测 | `STRATEGY-POOL` 中待验证的候选策略分支版本、用户历史交易、`11. 回放` | `9. 迭代` |
| `9. 迭代` | 把已回测结果变成下一轮可用内容，并更新策略池里的分支和版本状态 | `8. 回测`、`STRATEGY-POOL` 中当前候选和当前版本关系 | `STRATEGY-POOL`、`2. 观察`、`5. 恢复` 或后续策略流程 |
| `10. 投研` | 把外部材料、长期线索和候选方法整理成研究结论；其中一部分再升格为候选策略 | 用户、agent、社区、公开资料中的想法，以及主线中的具体研究问题 | `2. 观察`、`3. 决策`、`6. 管理` 或 `STRATEGY-POOL` |
| `11. 回放` | 从历史市场数据中选取起始时间点，锁住未来信息，并生成可供历史判断继续推进的回放语境 | 历史市场数据、起始时间点、当前问题对应的历史锚点 | `2. 观察`、`6. 管理`、`7. 复盘` 或 `8. 回测` |

### 1.4 固定回返

这里只固定回返规则簇，不在这里穷举所有动作名。

#### 高频回返

| 场景组 | 回返关系 |
| --- | --- |
| 重新进入系统、对话中断、状态不清、执行未决 | `任意阶段 -> 恢复 -> 观察/条件等待/成交等待/管理` |
| 开仓前反复判断 | `观察 <-> 决策` |
| 开仓前分歧但仍继续 | `决策 -> 分歧处理 -> 决策/条件等待/执行` |
| 开仓前硬性阻断，或分歧暴露出状态不清 | `分歧处理 -> 恢复` |
| 计划继续等待触发 | `决策 -> 条件等待` |
| 放弃当前计划，或当前机会不再成立 | `决策/条件等待 -> 观察` |
| 条件变化、机会错过、计划被替代 | `条件等待 -> 观察/决策` |
| 撤单、改单、失效 | `执行/成交等待 -> 观察/决策` |
| API 事实、用户反馈与已记录状态不一致 | `执行/成交等待/管理 -> 恢复` |
| 持仓中继续判断或执行 | `管理 -> 观察/决策/执行` |
| 持仓中分歧但仍继续 | `管理 -> 分歧处理 -> 管理/执行` |
| 持仓中再次挂单 | `管理 -> 执行 -> 成交等待 -> 管理/观察` |
| 已结束或阶段性闭合 | `管理 -> 待复盘池 -> 复盘` |

#### 特殊回返

| 场景组 | 回返关系 |
| --- | --- |
| 投研补充开仓前判断 | `投研 -> 观察/决策` |
| 投研补充持仓中判断 | `投研 -> 管理 -> 观察/决策/执行` |
| 回放重跑开仓前判断 | `回放 -> 观察/决策` |
| 回放推进历史仓位 | `回放 -> 管理 -> 观察/决策/执行` |
| 回放中的历史链路阶段性结束 | `回放 -> 复盘` |
| 复盘后重新进入主线 | `复盘 -> 观察/决策` |
| 原方向结束后开启新方向 | `管理 -> 复盘 -> 观察/决策` |
| 复盘或投研产出候选策略 | `复盘/投研 -> 策略池 -> 回测` |
| 迭代确认新的当前生效分支版本 | `迭代 -> 策略池 -> 观察/决策/管理` |
| 暂停后的分级恢复 | `复盘/迭代 -> 恢复 -> 观察/管理` |

### 1.5 固定约束

| 约束主题 | 固定约束 |
| --- | --- |
| 连续性与事实源 | 用户下次回来必须能继续当前交易链；当前产品先按“辅助用户决策并记录下单”设计，不默认系统直接代用户下单；系统建议、用户确认、真实已生效状态是三层不同东西，未对齐前不能直接视为已执行完毕；只要交易所 API 可用，订单、持仓、余额、活跃挂单和成交回报默认先以 API 结果为事实源；对话中断、用户自行操作、API 回报延迟和持久化上下文缺口是常态，必须通过 `恢复` 重新对账 |
| 流程治理 | `对齐 / 分歧处理` 是正式流程点，不是备注；必须保留系统建议、用户确认和最终计划，并区分 `软分歧` 与 `硬性阻断`；用户偏离系统建议不会产生新的主线阶段；后续章节都要回答“谁触发、接到哪里、给谁输出” |
| 主对象与计划来源 | 主线判断、等待和管理的默认操作对象是 `交易链 + 当前交易计划`，不是单订单；`交易链` 代表同一个交易想法的连续处理过程，不等于单次订单，也不等于单次开平仓；主线里的交易计划默认来自“当前观察语境 + 策略池中当前适用的生效分支版本”，不是长期允许 `OBSERVE` 脱离策略单独产出计划 |
| 策略绑定与版本 | 主线任一时点读取的不是“全局唯一当前策略”，而是 `策略池` 中当前语境下适用的单一分支版本；若没有适用策略，也要显式表现为 `无策略绑定`；同一条交易链里的当前计划默认绑定单一策略分支版本；除非在链内显式改计划，否则不能因为 `策略池` 里出现新版本就自动切换；当前先只确认 `策略池` 至少允许按 `交易标的` 或 `市场环境` 分支，不提前固定更细树形结构 |
| 可追溯与链边界 | 当前交易计划及其后续更新必须可解释、可追溯；不能只保留方向或动作结论而不保留依据、反证、失效条件和变更原因；多次撤单、重挂、加减仓、回补、补保护、再对冲或重试，默认先表现为同一条链里的计划更新，而不是直接新开链；只要核心 thesis、方向、时间尺度和机会定义没有发生实质变化，默认仍在原链内推进；当 thesis、方向、时间尺度或机会定义发生实质变化，或原链已经正式闭合后再次发起新的交易想法时，应视为新链 |
| 开仓前与等待状态 | 开仓前需要显式区分“机会本身是否成立”和“当前是否适合参与”；`不参与 / 只观察 / 缩仓 / 换载体 / 延后` 不能与“机会不成立”混成同一种结论；订单变化会反向影响主线所处阶段；`等待条件成熟` 和 `订单等待成交` 是两种不同等待，不能混成一个状态；`结束旧方向` 和 `开启新方向` 必须是两个判断，不能在流程上被写成单一“反手动作” |
| 支线边界 | 投研 是横向能力，不替代主线阶段；它回接主线时只能补充研究结论、风险提示和候选动作，不能直接改写真实持仓、订单和阶段状态；回放 必须锁住未来信息；回放中的判断链只服务历史验证和学习，不直接改写真实主线中的状态 |
| 复盘到迭代链路 | 已结束或阶段性闭合的交易链必须先进入 `待复盘池`；`复盘` 可以即时触发，也可以后续触发；`复盘` 和 投研 产出的候选策略先进入 `策略池`，再进入回测；`迭代` 负责回写分支状态和版本关系；复盘不会直接变长期规则，必须先进入回测；迭代只能发生在回测之后；`复盘` 可以直接回接下一轮主线，但直接回接的是复盘结论，不是绕过 `回测 -> 迭代` 直接切换长期策略版本 |

### 1.6 仍需继续回答的前置问题

| 问题主题 | 仍需继续回答的问题 |
| --- | --- |
| 链与版本粒度 | 原链、计划版本和单次尝试各需要记录到什么粒度，才能支撑同一 thesis 的多次重试与后续复盘 |
| 策略版本粒度 | 当前适用的策略分支和版本要记录到什么粒度；系统在没有显式策略版本时如何提供默认策略 |
| 等待状态粒度 | `条件等待` 与 `成交等待` 各需要记录到什么粒度，才能支撑回返和恢复 |
| 计划复杂度 | 一个交易计划何时只是单机会计划，何时需要升格为组合计划 |
| 分歧记录粒度 | `分歧处理` 需要记录到什么粒度，才能支撑后续复盘和策略迭代 |
| 策略池分支方式 | `策略池` 的分支到底只按标的、只按市场环境，还是允许两者交叉 |
| 执行层粒度 | 组合交易计划、对冲和订单组在执行层需要记录到什么粒度，才能支撑 `WAIT-UNTIL-FILL` 和 `RECOVERY` |
| 用户补充反馈粒度 | 在已有 API 事实源时，用户补充反馈还需要细到什么粒度，才能支撑恢复时的可信解释与意图还原 |
| 管理结束条件 | `管理` 何时算阶段性结束，何时进入 `复盘` |
| 投研输出分流 | 投研 结论如何区分“直接服务主线”与“升格进入 `策略池 -> 回测`” |
| 投研与复盘关系 | 除 `观察 / 决策 / 管理 / 回测` 外，投研 是否还需要直接服务 `复盘`，先不固定 |
| 回放与真实主线并存 | 回放 中的历史链路与真实主线链路如何并存又不混淆 |

## 2. 观察

### 输入

- 当前市场信息
- 当前交易链上下文
- `策略池` 中当前适用的生效分支版本
- 当前交易计划上下文
- 上一个阶段返回的状态
- 来自 `投研` 的市场语境、观察重点或风险提示
- 来自 `回放` 的历史时间锚点和锁住未来信息后的市场语境

### 输出

- 观察记录
- 进入决策的观察基础
- 候选机会列表或需要纳入同一计划判断的机会集合

### 管理对象

- 交易链
- 策略池中的当前分支版本
- 交易计划
- 阶段状态
- 判断记录

### 流程交互

- 在主线中负责给 `决策` 提供当前语境
- 来自流程控制定义的起点或回返点
- 可接入 `投研` 回接的市场语境和观察重点
- 可在 `回放` 中以历史语境方式再次发生
- 进入 `决策`
- 可被 `决策` 打回
- 可在 `管理` 中再次发生

### 需要详细设计

- 观察记录记录什么
- 观察如何挂到当前交易链
- 观察阶段如何读取当前适用的策略分支版本
- 多个候选机会如何挂到同一条链和同一个计划语境
- 如何恢复到当前观察语境
- 投研 结论如何映射成当前观察语境
- 回放 里的历史观察语境如何与真实观察记录区分

## 3. 决策

### 输入

- 观察记录
- 当前交易链上下文
- `策略池` 中当前适用的生效分支版本
- 当前交易计划上下文
- 持仓中的再判断需求
- 来自 `投研` 的 thesis、反证和风险提示
- 来自 `回放` 的历史语境下观察结果

### 输出

- 系统建议
- 不操作建议
- 放弃计划
- 进入分歧处理的计划建议
- 进入条件等待的最终计划
- 进入执行的最终动作决定
- 当前交易计划或组合计划

### 管理对象

- 交易链
- 策略池中的当前分支版本
- 交易计划
- 阶段状态
- 判断记录

### 流程交互

- 来自 `观察`
- 可直接消费 `投研` 提供的研究结论
- 可在 `回放` 中基于历史语境形成动作判断
- 先进入 `对齐 / 分歧处理`，再进入 `执行` 或 `WAIT-CONDITION`
- 可回退到 `观察`
- 可在 `管理` 中再次发生

### 需要详细设计

- 决策记录记录什么
- 决策与观察的往返如何表达
- 当前适用的策略分支版本如何约束计划生成
- 如何把“机会成立 / 机会不成立”和“现在适合参与 / 现在不适合参与”拆成两层判断，并让 `不参与 / 只观察 / 缩仓 / 换载体 / 延后` 成为正式输出
- 交易计划如何同时保留支持证据、反方证据、失效条件、`不做` 分支，以及入场、止损、止盈、分批和移动规则各自依据
- 决策与 `WAIT-CONDITION` 的往返如何表达
- 分歧处理点如何记录系统原建议、用户确认、风险提示、最终计划，以及何时回到 `PLAN`
- 不同决策结果如何影响后续流转
- 多机会何时收敛成一个组合计划，何时拆成多个互斥计划
- 投研 结论与观察记录冲突时如何处理
- 回放 中的历史决策如何与真实主线决策区分

## 4. 执行

### 输入

- 经 `对齐 / 分歧处理` 确认后的决策结果
- 当前交易计划
- 用户确认的执行意图
- 管理中的再执行动作
- 交易所 API 返回的订单、成交和撤改状态

### 输出

- 订单记录
- 订单组或组合部分状态记录
- 执行动作记录
- 等待成交态、恢复所需状态、API 对账结果或成交结果

### 管理对象

- 交易链
- 交易计划
- 阶段状态
- 订单

### 流程交互

- 来自 `决策` 或 `管理`
- 进入等待态、`恢复`、`管理` 或返回 `观察/决策`
- 可在 `管理` 中重复发生

### 需要详细设计

- 执行建议、用户反馈和订单生命周期如何区分
- API 回报、用户反馈和本地持久化状态冲突时谁优先、如何留痕
- 执行只负责订单层等待，不负责条件等待
- 组合交易计划如何展开成订单组、执行顺序和部分成交状态
- 撤单、改单、失效如何回写主线
- 执行结果如何保持链不断开

## 5. 恢复

### 输入

- 交易所 API 拉取的订单、持仓、余额、活跃挂单和成交回报
- 系统已持久化但可能不完整的上下文
- 用户补充反馈的当前真实状态与主观意图
- 当前市场信息
- 当前交易计划或订单组上下文

### 输出

- 已对齐的当前状态
- 待确认项、可信度判断或 API 与本地状态分歧说明
- 继续进入 `观察`、条件等待、等待成交或 `管理` 的落点
- 分级恢复下的当前恢复层级

### 管理对象

- 交易链
- 交易计划
- 阶段状态
- 持仓
- 订单
- 判断记录

### 流程交互

- 来自重新进入系统、对话中断后的继续、条件等待未决、等待成交未决或管理态失配
- 可承接 `迭代` 给出的暂停后恢复路径
- 进入 `观察`、条件等待、等待成交或 `管理`
- 可在任意状态不清时再次发生

### 需要详细设计

- 哪些状态能直接由 API 自动确认，哪些仍必须重新向用户确认
- API 事实源、用户补充反馈和旧上下文发生冲突时如何判优与留痕
- 组合计划失配、部分成交和对冲未补齐时如何恢复到当前计划
- `继续观察 / 影子恢复 / 小仓恢复 / 恢复常规` 如何映射到恢复后的落点
- 恢复后如何判定下一步落点

## 6. 管理

### 输入

- 已成交订单
- 当前持仓
- `策略池` 中当前适用的生效分支版本
- 当前交易计划
- `恢复` 后对齐出来的当前状态
- 持仓中的新增判断
- 来自 `投研` 的持仓监控点、反证或调整提示
- 来自 `回放` 推进后的历史持仓和历史订单语境

### 输出

- 管理建议
- 新订单
- 更新后的交易计划
- 进入待复盘池的交易结果

### 管理对象

- 交易链
- 策略池中的当前分支版本
- 交易计划
- 阶段状态
- 持仓
- 订单
- 判断记录

### 流程交互

- 来自 `执行` 或 `恢复`
- 可接入 `投研` 回接的持仓中研究结论
- 可在 `回放` 中继续推进历史持仓管理
- 内部可再次触发 `观察`、`决策`、`对齐 / 分歧处理`、`执行`
- 结束后进入 `待复盘池`

### 需要详细设计

- 持仓未结束但动作持续发生时如何表达
- 加仓、减仓、调杠杆、止损、止盈如何挂回同一交易链
- 当前适用的策略分支版本如何约束持仓中的计划更新
- 持仓中的当前计划何时允许切换到同分支新版本，何时必须继续沿用原版本
- 持仓中的计划更新如何保留原计划、变更后计划、触发信息和变更理由，并区分 thesis 变化、风险约束变化与执行方式变化
- 管理建议如何进入 `ALIGN-POS`，并记录系统原建议、用户确认和最终动作
- 计划内分批建仓、对冲补齐和情绪化摊平如何区分
- 管理何时算阶段性结束
- 投研 产出的风险提示和反证如何挂回当前持仓
- 同一条链里的计划更新、不同计划版本和应切成新链的情况如何判断与记录
- 回放 中的历史管理动作如何与真实管理动作区分

## 7. 复盘

### 输入

- 来自待复盘池的已结束或阶段性闭合的交易链
- 来自 `回放` 的阶段性闭合历史链路
- 交易记录
- 判断记录

### 输出

- 复盘记录
- 进入 `策略池` 的候选策略或分支修订建议
- 可回接下一轮主线的复盘结论

### 管理对象

- 交易链
- 待复盘池
- 交易记录
- 判断记录
- 候选策略

### 流程交互

- 来自 `管理` 进入的待复盘池
- 可由 `回放` 中阶段性结束的历史链路触发
- 从待复盘池中被拉起
- 进入 `策略池`，再进入 `回测`
- 可直接回接下一轮 `观察 / 计划`
- 与 投研 一起构成两个候选策略入池入口

### 需要详细设计

- 哪些结果进入待复盘池
- 待复盘池如何筛选和拉起
- 复盘记录保留什么
- 如何沿同一交易链回看计划从生成到多次改动的过程，并把动作变化映射回具体证据、反证、事件、约束和用户确认
- 如何持续对照系统原建议、用户最终执行和真实结果，并区分系统判断、用户判断与执行偏离的影响
- 如何把 edge 从观察、计划、下单、成交、仓位、持有、保护、恢复衔接和成本载体这些环节逐段拆开，区分“看对了但没赚到”到底丢在哪一段
- 复盘产物如何进入 `策略池`，并挂到具体分支和版本
- 哪些复盘结论只回接下一轮主线，哪些要继续推动策略迭代
- 回放 里的复盘样本如何与真实闭链样本区分

## 8. 回测

### 输入

- 来自 `策略池` 的候选策略分支版本
- 来自 回放 或用户历史交易的验证集

### 输出

- 回测结果
- 进入迭代的候选内容
- 继续回测所需状态

### 管理对象

- 策略池中的候选分支版本
- 验证集
- 回测结果

### 流程交互

- 来自 `策略池`
- 使用 回放 或用户历史交易
- 进入 `迭代`

### 需要详细设计

- 两个策略入口如何先进入 `策略池` 再共用同一回测口
- 每次回测如何绑定验证集来源
- 每次回测如何绑定具体分支、版本和回测前的策略状态
- 回测结果如何沉淀进数据库

## 9. 迭代

### 输入

- 回测结果
- `策略池` 中相关分支和版本的当前状态

### 输出

- 下一轮可使用的策略结果
- 回写到 `策略池` 的分支、版本和状态决议
- 下一轮主线回接落点

### 管理对象

- 策略池
- 策略分支与版本
- 回测结果

### 流程交互

- 来自 `回测`
- 回写 `策略池` 后重新进入后续主线使用或后续策略流程
- 为后续 `观察 / 计划 / 管理` 提供当前适用的生效分支版本
- 可显式回接 `观察 / 计划` 或 `恢复`

### 需要详细设计

- 哪些回测结果能进入迭代
- 迭代后的内容如何回写 `策略池` 并重新被主线使用
- 当前生效分支版本如何切换、回退或保留旧版对照
- `继续观察 / 影子恢复 / 小仓恢复 / 恢复常规` 如何由迭代结果决定

## 10. 投研

### 输入

- 用户提出的研究问题
- 用户提出的假设
- agent 发散出的假设
- 社区流传的策略
- 网络、论坛、论文里的公开策略
- 宏观、跨资产、监管、地缘或市场结构材料
- 当前主线里需要补充解释的标的、thesis 或风险点

### 输出

- 可直接服务 `观察` 的市场语境和观察重点
- 可直接服务 `决策` 的 thesis、触发条件、风险提示和反证
- 可直接服务 `管理` 的持仓监控点、失效线索和调整提示
- 可进入 `策略池` 的候选策略

### 管理对象

- 研究任务
- 研究结论
- 候选策略

### 流程交互

- 不依赖当前持仓，也可由主线中的具体问题触发
- 可直接回接 `观察`、`决策` 和 `管理`
- 其中一部分研究结论继续整理成候选策略后进入 `策略池` 再进入 `回测`
- 与 `复盘` 共享“候选策略 -> `策略池` -> 回测”入口

### 需要详细设计

- 零散想法如何进入系统
- 研究任务如何绑定到具体标的、thesis 或交易链
- 想法如何区分成“直接服务主线的研究结论”和“可回测策略”
- 来自不同标的或不同市场环境的候选策略如何映射到 `策略池` 分支
- 主线使用研究结论时如何保留来源、时点和失效条件
- 如何与复盘产物共用同一 `策略池 -> 回测` 入口

## 11. 回放

### 输入

- 历史数据
- 起始时间点
- 当前问题映射到的历史案例或时间锚点

### 输出

- 提供给 `观察 / 决策` 使用的历史市场语境
- 提供给 `管理` 使用的历史持仓推进语境
- 提供给 `复盘` 使用的历史链路样本
- 提供给回测阶段使用的回放语境和验证集

### 管理对象

- 回放会话
- 历史时间锚点
- 回放语境
- 验证集

### 流程交互

- 不生成策略
- 不直接产生回测结果
- 可在锁住未来信息后驱动一轮历史语境下的 `观察 / 决策 / 管理 / 复盘`
- 也可向 `回测` 提供市场语境和验证集支持

### 需要详细设计

- 回放从什么时间点开始
- 回放 如何锁住未来信息并逐步推进时间
- 回放 如何驱动 `观察 / 决策 / 管理 / 复盘` 的历史循环
- 回放 向回测输出什么
- 回放验证集如何与用户历史交易并存
- 回放会话如何与真实主线状态隔离

## 12. Infra 层

### 12.1 作用

在 1-11 章的业务流转收敛之后，提供统一承载；不反向定义业务链路本身。

### 12.2 数据库

#### 输入

- 流程控制确定下来的当前状态
- 2-11 章最终确定的长期对象和事件

#### 输出

- 持久化状态
- 历史记录
- 上下文恢复所需数据
- UI 界面读取的数据

#### 管理对象

- 交易链
- 阶段状态
- 订单
- 持仓
- 交易记录
- 判断记录
- 策略池
- 策略分支与版本
- 验证集
- 回测结果

#### 需要详细设计

- 哪些对象长期保存
- 各对象如何挂接
- 用户回来时最少恢复哪些内容

### 12.3 UI 界面

#### 输入

- 数据库中的长期数据
- 流程控制中的当前状态

#### 输出

- 查看结果
- 基础 CRUD 结果

#### 管理对象

- 被允许展示和编辑的长期数据对象

#### 需要详细设计

- 第一批展示哪些对象
- 哪些对象允许 CRUD
- 如何展示当前链路、历史链路、策略池分支版本、回测
