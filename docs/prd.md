这里表达的不是“章节顺序”，而是一个更完整的 user story。

其中最关键的是：

- 图里节点改成英文，是为了避免中文双宽字符把 ASCII 结构撑歪
- 右侧逐行中文注释负责解释每一步的业务含义
- `RECOVERY` 是正式阶段，不只是“重新加载上下文”，而是先对齐真实状态，再决定回到哪里继续
- 要先按 `开仓前 / 持仓中 / 平仓后` 三段来理解，而不是按线性章节来理解
- 开仓前不是 `OBSERVE -> DECIDE -> PLAN`，而是 `OBSERVE <-> PLAN`
- 开仓前的交易计划会分成 `WAIT-CONDITION`、`DROP`、`MKT`、`PLACE` 四条路
- `WAIT-CONDITION` 表达的是“计划仍有效，但条件还没成熟，暂时还没有挂单”
- `MKT` 会直接进入 `IN-POS`，只有 `PLACE` 才进入 `WAIT-UNTIL-FILL`
- `WAIT-UNTIL-FILL` 必须显式包含 `cancel / amend / expire / partial fill / full fill`
- 当前产品先按“辅助用户决策并记录下单”设计；`执行建议` 不等于 `真实已生效`
- `WAIT-CONDITION` 和 `WAIT-UNTIL-FILL` 是两种不同等待：前者等计划触发，后者等订单成交
- `WAIT-UNTIL-FILL` 表达的是“已知有挂单在等待成交”，`RECOVERY` 表达的是“当前真实状态还需要重新对齐”
- `MANAGE` 不是单点，而是持仓语境下再次运行一轮 `OBS-POS <-> PLAN-POS`
- 持仓中的动作要区分成“继续持仓动作”和“平仓动作”
- 继续持仓动作成交后回到 `MANAGE`，平仓全部成交或阶段性闭合后先进入 `REVIEW-QUEUE`
- `REVIEW-QUEUE` 用来承接已闭链但暂未复盘的交易链，`REVIEW` 可以立刻触发，也可以后续再拉起
- `REVIEW` 和 投研 都产出策略，统一进入 `BACKTEST`
- 主线第 8 阶段现在是 回测，不再叫 验证
- 回放 不是回测本身，而是给 回测 提供可作用策略的市场语境
- `BACKTEST` 既可以把策略作用于用户历史交易，也可以把策略作用于回放语境

### 1.2 主线与支线接点

- 主线负责围绕真实交易持续推进
- `恢复` 负责在重新进入系统、对话中断、执行未决或状态不清时先对齐真实状态
- `等待条件成熟` 负责承接计划已存在但尚未进入下单的状态
- 已结束或阶段性闭合的交易链先进入 `待复盘池`，不要求立即复盘
- 投研 不依赖当前持仓，负责从外部想法生成可回测策略
- `复盘` 和 投研 是两个策略入口
- 回放 是独立支线，负责提供可供回测使用的市场语境
- 回测 是统一回测口，既可以把策略作用于用户历史交易，也可以把策略作用于回放语境
- `迭代` 只能接在 `回测` 之后
- `infra` 不定义业务流转，只承载前面已经确定的业务对象和状态

### 1.3 章节在工作流中的位置

| 章节 | 在整体工作流中的职责 | 主要输入来自 | 主要输出给 |
| --- | --- | --- | --- |
| `2. 观察` | 为当前交易链补充最新市场语境和观察基础 | 市场信息、当前交易链、上个阶段回返状态 | `3. 决策` |
| `3. 决策` | 基于观察和上下文形成动作判断 | `2. 观察`、`6. 管理` 中的再判断需求 | `4. 执行`、`WAIT-CONDITION` 或回退到 `2. 观察` |
| `4. 执行` | 把决策转成用户要执行或已反馈的订单/动作结果 | `3. 决策`、`6. 管理` 中的再执行动作 | `WAIT-UNTIL-FILL`、`5. 恢复`、`6. 管理` 或回返 `2/3` |
| `5. 恢复` | 在重新进入系统、状态不清或执行未决时对齐真实状态，并决定当前链从哪里继续 | 用户口头反馈、已持久化上下文、当前市场与账户语境 | `2. 观察`、`WAIT-CONDITION`、`WAIT-UNTIL-FILL` 或 `6. 管理` |
| `6. 管理` | 在持仓/已成交语境下持续推进链路 | `4. 执行` 的成交结果、`5. 恢复` 的对齐结果、当前持仓 | `2/3/4` 或 `7. 复盘` |
| `7. 复盘` | 从待复盘池中拉起已结束或阶段性闭合的交易链，产出策略结论 | `6. 管理` 闭链结果、待复盘池 | `8. 回测` |
| `8. 回测` | 统一接收策略，并把策略作用于用户历史交易或回放语境做回测 | `7. 复盘`、`10. 投研`、用户历史交易、`11. 回放` | `9. 迭代` |
| `9. 迭代` | 把已回测结果变成下一轮可用内容 | `8. 回测` | 后续主线使用或后续策略流程 |
| `10. 投研` | 把零散想法整理成可回测策略 | 用户、agent、社区、公开资料中的想法 | `8. 回测` |
| `11. 回放` | 从历史市场数据中选取起始时间点，生成可供回测使用的市场语境 | 历史市场数据、起始时间点 | `8. 回测` |

### 1.4 固定回返

| 场景 | 回返关系 |
| --- | --- |
| 重新进入系统、对话中断、状态不清 | `任意阶段 -> 恢复 -> 观察/条件等待/成交等待/管理` |
| 开仓前反复判断 | `观察 <-> 决策` |
| 开仓前继续等待计划触发 | `决策 -> 条件等待` |
| 不建议操作、放弃计划 | `决策 -> 观察` |
| 条件触发、歧义触发、机会错过、被替代 | `条件等待 -> 观察/决策` |
| 撤单、改单、失效 | `执行/等待 -> 观察/决策` |
| 用户反馈与已记录状态不一致 | `执行/等待/管理 -> 恢复` |
| 持仓中继续判断 | `管理 -> 观察/决策/执行` |
| 持仓中再次挂单 | `管理 -> 执行 -> 等待 -> 管理/观察` |
| 已结束或阶段性闭合 | `管理 -> 待复盘池 -> 复盘` |

### 1.5 固定约束

- 用户下次回来必须能继续当前交易链
- 当前产品先按“辅助用户决策并记录下单”设计，不默认系统直接代用户下单
- 系统建议、用户确认、真实已生效状态是三层不同东西；未对齐前不能直接视为已执行完毕
- 对话中断、用户自行操作和持久化上下文缺口是常态，必须通过 `恢复` 重新对账
- 后续章节设计都要回答“谁触发、接到哪里、给谁输出”
- 订单变化会反向影响主线所处阶段
- `等待条件成熟` 和 `订单等待成交` 是两种不同等待，不能混成一个状态
- 已结束或阶段性闭合的交易链必须先进入 `待复盘池`；`复盘` 可以即时触发，也可以后续触发
- 复盘不会直接变长期规则，必须先进入回测
- 迭代只能发生在回测之后

### 1.6 仍需继续回答的前置问题

- 一条交易链何时开始、何时闭合
- `条件等待` 与 `成交等待` 各需要记录到什么粒度，才能支撑回返和恢复
- 用户口头反馈需要细到什么粒度，才能支撑恢复时的可信对账
- `管理` 何时算阶段性结束，何时进入 `复盘`

## 2. 观察

### 输入

- 当前市场信息
- 当前交易链上下文
- 上一个阶段返回的状态

### 输出

- 观察记录
- 进入决策的观察基础

### 管理对象

- 交易链
- 阶段状态
- 判断记录

### 流程交互

- 在主线中负责给 `决策` 提供当前语境
- 来自流程控制定义的起点或回返点
- 进入 `决策`
- 可被 `决策` 打回
- 可在 `管理` 中再次发生

### 需要详细设计

- 观察记录记录什么
- 观察如何挂到当前交易链
- 如何恢复到当前观察语境

## 3. 决策

### 输入

- 观察记录
- 当前交易链上下文
- 持仓中的再判断需求

### 输出

- 操作建议
- 不操作建议
- 放弃计划
- 进入条件等待的计划决定
- 进入执行的动作决定

### 管理对象

- 交易链
- 阶段状态
- 判断记录

### 流程交互

- 来自 `观察`
- 进入 `执行` 或 `WAIT-CONDITION`
- 可回退到 `观察`
- 可在 `管理` 中再次发生

### 需要详细设计

- 决策记录记录什么
- 决策与观察的往返如何表达
- 决策与 `WAIT-CONDITION` 的往返如何表达
- 不同决策结果如何影响后续流转

## 4. 执行

### 输入

- 决策结果
- 用户确认的执行意图
- 管理中的再执行动作

### 输出

- 订单记录
- 执行动作记录
- 等待成交态、恢复所需状态或成交结果

### 管理对象

- 交易链
- 阶段状态
- 订单

### 流程交互

- 来自 `决策` 或 `管理`
- 进入等待态、`恢复`、`管理` 或返回 `观察/决策`
- 可在 `管理` 中重复发生

### 需要详细设计

- 执行建议、用户反馈和订单生命周期如何区分
- 执行只负责订单层等待，不负责条件等待
- 撤单、改单、失效如何回写主线
- 执行结果如何保持链不断开

## 5. 恢复

### 输入

- 用户口头反馈的当前真实状态
- 系统已持久化但可能不完整的上下文
- 当前市场信息

### 输出

- 已对齐的当前状态
- 待确认项或可信度判断
- 继续进入 `观察`、条件等待、等待成交或 `管理` 的落点

### 管理对象

- 交易链
- 阶段状态
- 持仓
- 订单
- 判断记录

### 流程交互

- 来自重新进入系统、对话中断后的继续、条件等待未决、等待成交未决或管理态失配
- 进入 `观察`、条件等待、等待成交或 `管理`
- 可在任意状态不清时再次发生

### 需要详细设计

- 哪些状态必须重新向用户确认
- 用户口头反馈如何覆盖或修正旧上下文
- 恢复后如何判定下一步落点

## 6. 管理

### 输入

- 已成交订单
- 当前持仓
- `恢复` 后对齐出来的当前状态
- 持仓中的新增判断

### 输出

- 管理动作
- 新订单
- 进入待复盘池的交易结果

### 管理对象

- 交易链
- 阶段状态
- 持仓
- 订单
- 判断记录

### 流程交互

- 来自 `执行` 或 `恢复`
- 内部可再次触发 `观察`、`决策`、`执行`
- 结束后进入 `待复盘池`

### 需要详细设计

- 持仓未结束但动作持续发生时如何表达
- 加仓、减仓、调杠杆、止损、止盈如何挂回同一交易链
- 管理何时算阶段性结束

## 7. 复盘

### 输入

- 来自待复盘池的已结束或阶段性闭合的交易链
- 交易记录
- 判断记录

### 输出

- 复盘记录
- 新策略或策略修正

### 管理对象

- 交易链
- 待复盘池
- 交易记录
- 判断记录
- 策略

### 流程交互

- 来自 `管理` 进入的待复盘池
- 从待复盘池中被拉起
- 进入 `回测`
- 与 投研 一起构成两个策略入口

### 需要详细设计

- 哪些结果进入待复盘池
- 待复盘池如何筛选和拉起
- 复盘记录保留什么
- 复盘产物如何进入回测

## 8. 回测

### 输入

- 来自 `复盘` 的策略
- 来自 投研 的策略
- 来自 回放 或用户历史交易的验证集

### 输出

- 回测结果
- 进入迭代的候选内容
- 继续回测所需状态

### 管理对象

- 策略
- 验证集
- 回测结果

### 流程交互

- 来自 `复盘` 和 投研
- 使用 回放 或用户历史交易
- 进入 `迭代`

### 需要详细设计

- 两个策略入口如何共用同一回测口
- 每次回测如何绑定验证集来源
- 回测结果如何沉淀进数据库

## 9. 迭代

### 输入

- 回测结果
- 已进入后续使用的策略内容

### 输出

- 下一轮可使用的策略结果

### 管理对象

- 策略
- 回测结果

### 流程交互

- 来自 `回测`
- 重新进入后续主线使用或后续策略流程

### 需要详细设计

- 哪些回测结果能进入迭代
- 迭代后的内容如何重新被主线使用
