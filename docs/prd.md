# PRD

## 1. 流程控制

### 流程总图

这一章把在线交易推进、复盘后的验证闭环和策略沉淀放在同一张图里描述，不再拆成主流程和附属流程两套骨架。

这里默认只保留三种概念：

- `阶段`：`OBSERVE / PLAN / EXECUTE / REVIEW / BACKTEST / ITERATE`
- `状态`：`WAIT-CONDITION / WAIT-UNTIL-FILL`
- `动作`：`挂单 / 市价进入 / 撤单 / 改单 / 加仓 / 减仓 / 止损 / 止盈 / 平仓`

动作不是阶段，状态也不是阶段；`STRATEGY-POOL` 是长期容器，也不是阶段。

这里的阶段和状态更像 agent 编排时使用的工作视角，不是要求系统实现一套复杂状态机。系统实际长期维护的核心对象更应该是一组按时间戳推进的决策流；每条决策流记录当前决策状态、当前生效 plan 版本、最近一次账户信息拉取结果、系统建议、用户反馈和真实执行事实。验证链则额外维护研究目标、候选分支、最近一次验证结论、主要失败原因、下一轮待验证问题和停止条件。

agent 的实际运行方式不是机械按阶段前进，而是“收到用户消息后先看当前决策流状态与最新事实，再跑 checklist，再决定本轮推进到哪里”；当进入研究验证一侧时，再由主 agent 以研究调度器身份继续推进。

```text
[USER MESSAGE] --> [INTENT ROUTER] --> 看盘 / 追问 plan / 监控条件 ----------> [OBSERVE]
                          |                                                      |
                          |                                                      +--> 检测 observe checklist
                          |                                                      +--> checklist 未完成：
                          |                                                      |    补市场 / 宏观 / 新闻 / 用户上下文
                          |                                                      |    获取持仓 / 挂单 / 成交 / 余额
                          |                                                      |    必要时 web search
                          |                                                      +--> checklist 全完成 ----------> [PLAN]
                          |                                                                                     |
                          |                                                                                     +--> 写入当前 plan 版本及其属性
                          |                                                                                     |    `noop / continue-scan / draft-closed`
                          |                                                                                     |    `wait-condition / ready-execute / abandon`
                          |                                                                                     +--> 回复用户
                          |
                          +--> 执行已有 plan / 挂单 / 市价进入 / 撤单 / 改单
                          |    / 加仓 / 减仓 / 止损 / 止盈 / 平仓 ------------------------------> [EXECUTE]
                          |                                                                      |
                          |                                                                      +--> 执行动作、读取订单 / 成交 / 持仓变化
                          |                                                                      +--> 收到执行确认后调用币安 API 核实持仓 / 挂单 / 成交变化
                          |                                                                      +--> 回复用户
                          |
                          +--> 复盘 / 解释 / 总结 ----------------------------------------------> [REVIEW]
                                                                                                 |
                                                                                                 +--> 输出复盘 / 解释 / 阶段性结论
                                                                                                 +--> 回复用户
                                                                                                 
[外部研究输入] -----------------------------------------------------------------------------------+--> candidate hypothesis/rule --> [主 agent 研究调度]
                                                                                                |
                                                                                                +--> 派发 subagent:
                                                                                                |    抽假设 / 收敛规则 / 局部自测
                                                                                                +--> 派发 subagent [BACKTEST]:
                                                                                                |    +--> 长样本回测 / sibling 对照 / 失败聚类
                                                                                                |    +--> 历史回放（方法）
                                                                                                |
                                                                                                +--> 汇总结果 -----------------> [ITERATE]
                                                                                                                                    |
                                                                                                                                    +--> 升格 / 影子 / 分叉 / 归档 --> [STRATEGY-POOL]
                                                                                                                                    |                                     |
                                                                                                                                    |                                     +--> 当前语境下适用的单一生效分支版本 --> [OBSERVE / PLAN]
                                                                                                                                    |
                                                                                                                                    +--> 继续研究 -----------------------> [主 agent 研究调度]
```

这张图表达的是统一流程骨架，不是要求系统把所有节点实现成硬编码固定状态机。

- 所有在线交互默认都从 `USER MESSAGE` 进入，再由 `INTENT ROUTER` 判断本轮优先进入 `OBSERVE`、`EXECUTE` 还是 `REVIEW`。
- `OBSERVE` 内部有自己的 checklist；只要 checklist 还没全部打勾，就继续停留在 `OBSERVE` 内部循环补上下文，直到本轮判断所需语境齐全。
- `OBSERVE` 的 checklist 默认可以包含“账户事实”这一项；当该项命中时，只调用一次币安 API 获取当前持仓 / 挂单 / 成交 / 余额，后续同一轮 checklist 不再重复获取。
- `EXECUTE` 在收到用户执行确认消息后，也只调用一次币安 API 用来确认用户是否正确执行。
- 不是所有用户消息都默认先走 `OBSERVE`；例如执行类消息应优先进入 `EXECUTE`，复盘类消息应优先进入 `REVIEW`，而“看盘 / 继续追问计划 / 问条件是否满足”这类消息通常才进入 `OBSERVE`。
- `OBSERVE` 不是一次性取数，而是围绕当前用户消息、当前市场状态和策略所需证据持续补齐上下文的内部循环；它可以继续读取 `ohlcv`、结构、波动、宏观信息、新闻消息面，以及必要的 web search 结果，直到当前轮观察 checklist 完成。
- `PLAN` 默认只消费已经补齐的观察上下文，不再承担“上下文不够再去补”的职责；它负责结合当前适用策略、观察结果、上一版 `plan` 和用户反馈，生成本轮最终计划版本。
- `PLAN` 的 `noop / continue-scan / draft-closed / wait-condition / ready-execute / abandon` 更适合被视为当前 `plan` 的属性或状态标签，而不是生成后立刻自动跳到其他阶段。
- `PLAN`、`EXECUTE` 和 `REVIEW` 本轮产出都先回给用户；后续是否继续观察、执行、复盘，统一由下一条 `USER MESSAGE` 再次进入 router 决定。
- 同一条链上只要出现新的用户追问、用户反馈、条件重查需求，或市场时间推进导致原计划证据变旧，就重新从 `USER MESSAGE -> INTENT ROUTER` 进入下一轮；新一轮会带着上一版 `plan`、用户反馈和已知持仓 / 挂单语境继续收敛。
- `REVIEW` 是桥接节点，不是每次都为了发展新策略；只有复盘或外部研究真的形成了候选假设 / 候选规则，才进入验证与沉淀一侧。
- `BACKTEST / ITERATE / STRATEGY-POOL` 仍然属于验证与沉淀链，不属于在线交易推进本身；验证链默认由主 agent 调度，由 subagent 承接边界清楚的单轮任务。
- `历史回放` 当前更适合被视为 `BACKTEST` 内部调用的方法，不单列为独立流程节点。
- `STRATEGY-POOL` 在这里主要表达“研究产物进入长期容器”和“主线从中读取当前生效分支版本”；它不负责驱动研究循环。

### 流程项角色

| 项 | 角色 | 核心输出 | 不是什么 |
| --- | --- | --- | --- |
| `OBSERVE` | 结合用户消息、市场状态和当前适用策略，持续补齐本轮判断所需的市场 / 宏观 / 新闻 / 研究上下文，并渐进披露策略所需证据 | 已补齐上下文的观察记录、观测结果、候选机会、进入计划的依据 | 不是一次性取数动作 |
| `PLAN` | 消费已经补齐的观察上下文，结合当前适用策略、上一版 plan 和用户反馈，形成当前交易计划版本，回答“机会是否成立”和“现在是否适合参与” | 当前交易计划版本及其属性、进入执行的动作建议、等待条件标签、不参与结论 | 不负责补上下文，也不负责确认真实订单状态 |
| `WAIT-CONDITION` | 当前 `plan` 的状态标签之一，表示计划条件未成熟，后续通常由新的用户消息触发重查 | 等待中的计划状态、重查条件、作废或替代判断 | 不是独立阶段 |
| `EXECUTE` | 把已确认计划转成真实动作或真实订单状态 | 订单、成交结果、执行记录 | 不是重新做策略判断 |
| `WAIT-UNTIL-FILL` | 当前订单 / 执行链的状态标签之一，表示执行已开始但挂单仍未最终对齐 | 当前活跃订单、部分成交、撤改单、最坏风险 | 不是独立阶段 |
| `REVIEW` | 对已结束或阶段性闭合的交易链做解释、归因和提炼，并决定是否产出候选假设 / 候选规则 | 复盘记录、候选假设、进入验证链的研究输入 | 不是每次都要推进长期策略 |
| `BACKTEST` | 统一验证候选假设或候选规则 | 长样本验证结果、样本统计、失败簇、sibling 对照结果 | 不是复盘，也不是实时交易流程 |
| `ITERATE` | 根据验证结果决定保留、分叉、影子使用、升级、降级或归档 | 当前轮迭代结论、下一轮待验证问题、是否继续研究 | 不是自动改写在线流程 |
| `STRATEGY-POOL` | 存放长期策略分支与版本，供主流程读取 | 当前分支、版本、状态和可供主流程读取的结果 | 不是流程阶段，也不是研究驱动器 |
| `外部研究输入` | 提供市场语境、thesis、风险提示和候选假设来源 | 观察侧补充语境，或进入验证链的候选研究输入 | 不是独立阶段 |
| `历史回放` | `BACKTEST` 可调用的一种方法，用于按时间窗口滑动验证集、统计候选策略在完整样本周期里的触发与结果分布 | 历史样本与验证语境 | 不是独立流程节点，也不是回测结果本身 |

这里固定一个关键边界：

- `PLAN` 解决的是“做不做、怎么做、为什么这么做”
- `EXECUTE` 解决的是“这件事是否真的下出去了、成交了多少、还剩多少真实风险”

两者不能混成一个阶段。

### 常见回返

| 场景组 | 回返关系 |
| --- | --- |
| 刚收到用户消息、状态不清、执行未决 | `任意点 -> USER MESSAGE -> INTENT ROUTER -> OBSERVE / EXECUTE / REVIEW` |
| 决策间疑似发生场外执行 | `OBSERVE / EXECUTE` 的 checklist 命中账户项后调用币安 API 校验当前账户事实 |
| 开仓前反复判断 | `同一条链上的新用户消息 / 用户反馈 / 市场证据更新 -> USER MESSAGE -> INTENT ROUTER -> OBSERVE(checklist 循环) -> PLAN(生成新版本)` |
| 计划已形成但条件未到 | `PLAN 写入 wait-condition；后续通常由用户消息触发 OBSERVE 重查条件` |
| 挂单后未最终成交 | `EXECUTE 写入 WAIT-UNTIL-FILL；后续通常由用户消息触发获取账户信息 / EXECUTE / OBSERVE` |
| API 事实与记录冲突 | `OBSERVE / EXECUTE` 再次调用币安 API 校验账户事实 |
| 持仓后继续推进 | `持仓语境下仍使用 USER MESSAGE -> INTENT ROUTER -> OBSERVE / EXECUTE` |
| 一条链结束或阶段性闭合 | `用户要求解释 / 总结时，由 USER MESSAGE 路由到 REVIEW` |
| `REVIEW` 发现值得沉淀的方法、假设或模板 | `REVIEW -> 主 agent 研究调度 -> BACKTEST -> ITERATE` |
| 外部研究已足够像候选规则 | `外部研究输入 -> 主 agent 研究调度 / BACKTEST` |
| 当前轮研究仍有明确未解问题 | `ITERATE -> 主 agent 研究调度 -> 下一轮验证` |
| 当前轮研究得到可保留结果 | `ITERATE -> STRATEGY-POOL -> OBSERVE / PLAN 读取当前适用版本` |

### 固定约束

| 约束主题 | 固定约束 |
| --- | --- |
| 阶段与状态 | `WAIT-CONDITION` 和 `WAIT-UNTIL-FILL` 是状态，不升格为主阶段 |
| 流程性质 | 流程由当前决策流状态和研究状态共同驱动，不要求实现复杂状态机 |
| 在线入口 | 所有在线交互默认都从 `USER MESSAGE` 进入，再由 `INTENT ROUTER` 做意图分发 |
| router 职责 | router 负责根据用户消息和当前链状态决定本轮优先进入 `OBSERVE`、`EXECUTE` 还是 `REVIEW`，不负责补市场上下文 |
| observe checklist | checklist 默认是 `OBSERVE` 内部组件；只要 checklist 未完成，就继续停留在 `OBSERVE` 内部循环 |
| 账户信息获取 | 获取持仓 / 挂单 / 成交 / 余额是 checklist 内部按需触发的动作，不是独立流程节点 |
| agent 运行方式 | agent 先读当前决策流状态，再结合用户消息意图进入 router 分发；进入 `OBSERVE` 后再运行该轮观察 checklist；进入验证链后再由主 agent 维护研究状态并调度单轮任务 |
| 事实源 | 只要交易所 API 可用，订单、持仓、余额、活跃挂单和成交默认先以 API 为事实源 |
| 连续性 | 用户下次回来必须能继续当前交易链，不能因为对话中断就丢失上下文 |
| 观察与计划边界 | `OBSERVE` 负责把当前轮判断所需上下文补齐，`PLAN` 默认消费完整上下文，不承担补上下文职责 |
| 隐性步骤 | 必要确认、计划分歧处理、风险重算都是 `PLAN` 内动作，不单独升格为主阶段 |
| 计划与执行 | 系统建议、用户确认、真实已生效状态是三层不同东西，未对齐前不能当成已经执行完毕 |
| 计划状态 | `valid / noop / continue-scan / draft-closed / wait-condition / ready-execute / abandon` 默认挂在当前 `plan` 版本上，作为计划属性或状态标签保存 |
| 计划版本 | 同一交易链上的每轮新 `plan` 默认 append 为新版本，保留系统原建议、用户反馈、改判原因和历史版本链 |
| 主对象 | 在线推进默认操作对象是 `交易链 + 当前活动决策流 + 当前持仓上下文`，不是单订单 |
| 链边界 | 多次撤单、重挂、加减仓、补保护、再尝试，默认先看作同一条链里的计划更新；只有 thesis、方向、时间尺度或机会定义实质变化时，才切新链 |
| 持仓语境 | 已持仓不是独立阶段；只是让 `OBSERVE / PLAN / EXECUTE` 读取更多上下文并继续推进 |
| review 角色 | `REVIEW` 是桥接节点，不是每次都默认进入长期策略发展 |
| 验证链角色 | `BACKTEST / ITERATE / STRATEGY-POOL` 属于验证与沉淀链，不属于在线交易推进阶段 |
| 研究驱动 | 验证闭环真正的驱动器是 `主 agent + 研究状态`，不是策略池本身 |
| subagent 边界 | subagent 默认只负责边界清楚的单轮任务，例如抽假设、收敛规则、局部自测、长样本回测、失败聚类和 sibling 对照 |
| 持续推进条件 | 是否继续下一轮研究，不只看有没有新想法，还要看这轮是否真正减少了不确定性；若只是重复同类结论、样本短期无法补齐或连续几轮都没有更可用规则，应暂停闭环 |
| 历史回放 | `历史回放` 是 `BACKTEST` 内部可调用的方法，不单列为独立阶段 |
| 策略池角色 | `STRATEGY-POOL` 只保存当前分支、版本、状态和可供主流程读取的结果，不负责驱动研究 |

### 仍需继续回答的前置问题

| 问题主题 | 仍需继续回答的问题 |
| --- | --- |
| 链与版本粒度 | 原链、计划版本和单次尝试各需要记录到什么粒度，才能支撑同一 thesis 的多次重试与后续复盘 |
| 策略版本粒度 | 当前适用的策略分支和版本要记录到什么粒度；系统在没有显式策略版本时如何提供默认策略 |
| 等待状态粒度 | `条件等待` 与 `成交等待` 各需要记录到什么粒度，才能支撑回返和恢复 |
| 计划复杂度 | 一个交易计划何时只是单机会计划，何时需要升格为组合计划 |
| 分歧记录粒度 | 用户偏离建议、系统重算风险和最终采用版本需要记录到什么粒度 |
| 策略池分支方式 | `策略池` 的分支到底只按标的、只按市场环境，还是允许两者交叉 |
| 执行层粒度 | 组合交易计划、对冲和订单组在执行层需要记录到什么粒度，才能支撑 `WAIT-UNTIL-FILL` 和账户信息获取 |
| 用户补充反馈粒度 | 在已有 API 事实源时，用户补充反馈还需要细到什么粒度，才能支撑恢复时的可信解释与意图还原 |
| 持仓语境闭合条件 | 持仓中的观察、决策、执行链何时算阶段性闭合，何时进入 `REVIEW` |
| 研究输入分流 | 外部研究结论如何区分“直接服务主流程”与“升格进入验证链” |
| 历史回放口径 | 历史回放作为滑动窗口验证方法时，样本、触发、去重和统计口径如何固定 |
| 研究状态最小集 | 主 agent 至少要长期维护哪些最小研究状态，才能在压缩上下文后继续推进下一轮 |
| subagent 产物格式 | subagent 至少要回传哪些结构化字段，才能避免主 agent 反复重读原始研究过程 |
| 持续推进边界 | 研究闭环在什么条件下继续、暂停、归档或允许影子进入主线，才能避免空转 |

### 策略接口与匹配

这里只先固定“策略如何被主流程读取和匹配”的最小思路，不在这里提前展开成类图、表结构或完整 DSL。

#### 基本理解

- 主流程里的策略读取不应被理解成“一个统一策略直接下结论”，而应至少分三层：`基础分析层 -> 默认判定层 -> 个性策略层`。
- 第一层负责把原始市场数据整理成公共语境；第二层负责给出默认开启的公共判定信号；第三层才是策略池里那些需要被召回、筛选和比较的个性策略。
- 因此主流程里的策略读取不是“把几十个指标全跑完再让每个策略自己解释”，而是 `原始数据 -> 基础公共语境 -> 默认判定信号矩阵 -> 个性策略召回与筛选 -> 当前 plan 收敛`。

#### 三层角色

| 层级 | 默认是否执行 | 主要职责 | 直接产物 |
| --- | --- | --- | --- |
| `基础分析层` | 是 | 把原始市场数据整理成跨策略可复用的公共语境 | 公共语境槽位 |
| `默认判定层` | 是 | 基于公共语境产出一组默认开启的判定信号，帮助主线快速判断机会质量、参与质量和风险级别 | 判定信号矩阵 |
| `个性策略层` | 否；需先召回再筛选 | 承接真正带 edge 的具体策略，用私有特征进一步深判、比较和解释 | 候选策略及其证据包 |

这里固定的是“层级职责”，不是提前固定每一层里必须有哪些具体策略。

#### 第一层：基础分析层的最小公共接口

基础分析层至少应能输出这些公共语境槽位；它们不是某个单一指标，而是供后两层复用的高层特征类：

| 公共接口 | 作用 |
| --- | --- |
| `market_regime` | 当前更像趋势、震荡、事件驱动、高波动、低流动性等哪类环境 |
| `directional_bias` | 当前方向偏置是多、空、中性还是冲突 |
| `structure_state` | 当前关键结构状态，例如是否临近支撑/压力/共振区、结构是否破坏、是否接近突破边缘 |
| `momentum_volatility_state` | 当前动量与波动状态，例如扩张、压缩、放量、衰减、贴压整理等 |
| `execution_constraints` | 当前执行约束，例如离关键位是否过远、止损是否过大、流动性/滑点/事件窗口是否不利 |
| `risk_context` | 当前风险语境，例如高周期逆风强弱、上下方空间、结构拥挤度 |
| `feature_conflicts` | 当前哪些公共特征相互冲突，例如低周期偏强但高周期逆风强 |
| `missing_context` | 当前还缺什么公共语境，导致不宜直接推进计划 |

这里要固定的是“公共槽位”，不是固定“必须使用哪几十个指标”。指标、结构算法、趋势线算法、事件输入都只是生成这些公共槽位的手段。

#### 第二层：默认判定层的最小接口

默认判定层不是用户私有 edge，也不是只做底层描述。它位于 `基础分析层` 和 `个性策略层` 之间，负责把公共语境进一步变成主线默认会参考的一组判定信号。

这一层可以包含很多默认开启的公共判定策略，例如多周期关系、结构位置质量、趋势 / 震荡适配度、执行质量、风险收益空间、事件窗口约束等；这里先固定接口，不提前固定清单。

| 接口块 | 作用 |
| --- | --- |
| `name` | 默认判定项名称，例如某种结构质量或多周期关系判定 |
| `version` | 该判定项当前使用的版本 |
| `depends_on` | 依赖哪些基础分析层输出的公共语境 |
| `judgement_scope` | 这个判定在评估什么，例如机会质量、参与质量、执行质量、风险升降级 |
| `signal` | 输出的标准化信号，例如 `support / weak-support / neutral / warning / block` |
| `reason` | 给出当前信号的主要依据、冲突项和触发原因 |
| `downgrade_path` | 当不是直接支持时，应该进入只观察、降级参与、等待、换载体还是直接阻断 |

默认判定层的输出更适合表现为矩阵，而不是一条拍板式结论。它回答的是“默认公共判断现在怎么看”，不是直接替代最终计划。

#### 第三层：个性策略层的接口

个性策略层才是策略池里那些真正需要被召回、比较和筛选的具体策略。它们建立在前两层之上，但不要求与前两层 `1:1` 完整对齐。

| 接口块 | 作用 |
| --- | --- |
| `family` | 策略族，例如突破、趋势跟随、回归、事件驱动 |
| `branch` | 同一策略族下的适用分支，例如适用标的、市场环境、周期 |
| `version` | 同一分支下逐步迭代的规则版本 |
| `status` | 当前处于 `candidate / shadow / active / deprecated` 等什么状态 |
| `base_requirements` | 依赖哪些基础分析层输出的公共语境，用于召回前置过滤 |
| `default_judgement_requirements` | 依赖哪些默认判定层信号，用于判断当前是否值得进入本策略深判 |
| `private_feature_requirements` | 该策略自己的私有特征，不要求基础分析层统一产出 |
| `hard_blocks` | 明确否决条件 |
| `soft_signals` | 加分项、弱匹配项或需要解释的信号 |
| `evaluation_mode` | 规则如何被解释，例如硬条件、软条件、需要 LLM 二次判断的部分 |

#### 默认收敛顺序

- 第一步是 `基础分析层产出公共语境`：先把原始数据整理成稳定、可复用的公共槽位。
- 第二步是 `默认判定层产出信号矩阵`：默认开启的公共判定策略基于公共语境给出 `support / weak-support / neutral / warning / block` 等标准化信号。
- 第三步是 `个性策略召回与筛选`：只有在基础语境和默认判定层支持继续看的前提下，才从策略池里召回值得进一步检查的个性策略。
- 第四步是 `策略私有深判与 plan 收敛`：被召回的个性策略再补看自己的私有特征，结果允许是 `match / weak-match / reject / missing-info`，再与前两层结果一起收敛到当前 plan。
- `weak-support` 和 `weak-match` 都应被视为正式信号，而不是随手备注。它们表达的不是“错”，而是“可以继续看，但应降级、比较、等待或缩小参与方式”。

#### LLM 在匹配中的位置

- LLM 更适合做 `候选内二次软匹配`，不适合直接面对全市场零散指标和零散材料裸判策略。
- 因此应先给 LLM 一份 `策略证据包`，至少包含：基础分析层的公共语境、默认判定层的信号矩阵、个性策略的私有特征已知情况、未满足项、冲突项和当前主要风险。
- LLM 的输出重点是解释：为什么匹配、为什么只是弱匹配、还缺什么信息、是否值得进入当前 `PLAN` 比较；不是跳过前两层直接替代全部策略匹配逻辑。

#### 当前不预先固定的内容

- 不预先固定必须有哪些指标，也不要求每次主线都把所有指标完整平铺给策略层
- 不预先固定类继承树、数据库表或规则 DSL
- 不预先固定默认判定层里究竟有多少条公共判定策略，也不提前把它们固化成长期不变的制度
- 不预先要求所有个性策略都能被公共接口完整表达；目标是先让基础分析层和默认判定层稳定工作，再让个性策略在其上做召回、比较和私有扩展

## 2. 观察

### 输入

- 当前轮用户消息、追问和反馈
- 当前市场信息
- 当前轮按需获取的持仓 / 挂单 / 成交 / 余额信息
- 当前交易链上下文
- 当前活动决策流
- `策略池` 中当前适用的生效分支版本
- 当前交易计划上下文
- 最近讨论摘要、未闭合追问链、当前生效 plan 版本、用户偏离建议记录和上一轮改判原因
- 上一个阶段返回的状态
- 来自外部研究输入的市场语境、观察重点或风险提示
- 宏观信息、新闻消息面和必要的 web search 结果

### 输出

- 观察记录
- 已补齐上下文、可供 `PLAN` 直接消费的观测结果
- 候选机会列表或需要纳入同一计划判断的机会集合

### 管理对象

- 交易链
- 当前活动决策流
- 策略池中的当前分支版本
- 交易计划
- 阶段状态
- 判断记录

### 流程交互

- 在主流程中负责围绕当前用户消息拉取市场数据、运行策略、补齐所需上下文，并给 `PLAN` 提供可直接消费的观测结果
- 来自流程控制定义的起点或回返点
- 当观察 checklist 命中账户项时，可调用币安 API 拉取当前持仓 / 挂单 / 成交 / 余额
- 可接入外部研究输入给出的市场语境和观察重点
- 可主动补读宏观信息、新闻消息面和必要的 web search 上下文
- 进入 `PLAN`
- 在上下文未齐时可在本阶段内自循环
- 同一条链上的新用户追问、计划证据过期或用户反馈变化时，会带着上一版 `plan` 和历史反馈再次发生
- 可在持仓语境下再次发生

### 需要详细设计

- 观察记录记录什么
- 观察如何挂到当前交易链
- `OBSERVE` 的观测结果以什么结构交给 `PLAN`
- `OBSERVE` 如何判断“当前轮上下文已经足够完整，可以退出 while 循环进入 `PLAN`”
- 观察阶段如何读取当前适用的策略分支版本
- 原始市场数据、结构算法、指标和研究输入如何先收敛成基础分析层的公共语境，而不是把全部底层输出直接平铺给决策层
- 哪些公共特征由基础分析层稳定产出，哪些进入默认判定层，哪些只是临时研究特征或个性策略私有特征
- 多个候选机会如何挂到同一条链和同一个计划语境
- 观察 checklist 里的账户项何时应命中并调用币安 API 获取当前持仓 / 挂单 / 成交 / 余额
- 外部研究结论如何映射成当前观察语境
- 用户消息、宏观信息、新闻消息面和 web search 结果如何并入同一轮观察上下文

## 3. 决策

### 输入

- 观察记录
- `OBSERVE` 产出的、上下文已补齐的观测结果
- 当前交易链上下文
- 当前活动决策流
- `策略池` 中当前适用的生效分支版本
- 当前交易计划上下文
- 最近讨论摘要、未闭合追问链、系统原建议、用户反馈、当前生效版本和上一轮改判原因
- 持仓中的再判断需求
- 来自外部研究输入的 thesis、反证和风险提示

### 输出

- 系统建议
- 不操作建议
- 放弃计划
- 需要在 `PLAN` 内继续确认或处理分歧的计划建议
- 带有 `wait-condition`、`ready-execute`、`noop` 等属性的当前计划版本
- 面向执行阶段的动作建议
- 当前交易计划版本或组合计划版本

### 管理对象

- 交易链
- 当前活动决策流
- 策略池中的当前分支版本
- 交易计划
- 阶段状态
- 判断记录

### 流程交互

- 来自 `OBSERVE`
- 消费的外部研究结论默认应已在 `OBSERVE` 中被收敛进当前轮上下文
- 在 `PLAN` 内完成必要确认、计划分歧处理和风险重算后，写回当前 `plan` 版本及其状态属性，等待后续用户消息决定下一轮路由
- `PLAN` 本身不负责账户事实拉取；若下一轮需要账户事实，应回到 `USER MESSAGE -> INTENT ROUTER -> OBSERVE / EXECUTE`，由对应 checklist 决定是否调用币安 API
- 不负责补上下文；若收到新的用户追问、用户反馈或发现原证据已过期，应带着当前 `plan` 版本重新进入下一轮 `USER MESSAGE -> INTENT ROUTER -> OBSERVE`
- 可在持仓语境下再次发生

### 需要详细设计

- 决策记录记录什么
- `PLAN` 如何消费 `OBSERVE` 的观测结果
- 同一交易链上的多轮 `plan` 版本如何 append 保存，并保留原建议、用户反馈和改判原因
- 当前适用的策略分支版本如何约束计划生成
- 如何基于基础分析层和默认判定层先形成默认判定矩阵，再决定召回哪些个性策略进入深判
- 当默认判定层与个性策略层给出冲突信号时，如何输出 `support / weak-support / neutral / warning / block / match / weak-match / reject / missing-info`
- 如何把“机会成立 / 机会不成立”和“现在适合参与 / 现在不适合参与”拆成两层判断，并让 `不参与 / 只观察 / 缩仓 / 换载体 / 延后` 成为正式输出
- 交易计划如何同时保留支持证据、反方证据、失效条件、`不做` 分支，以及入场、止损、止盈、分批和移动规则各自依据
- `wait-condition / ready-execute / noop / abandon` 这些计划属性如何结构化保存，并驱动下一轮用户消息路由
- 分歧处理点如何记录系统原建议、用户确认、风险提示、最终计划，以及何时回到 `PLAN`
- 不同决策结果如何影响后续流转
- 多机会何时收敛成一个组合计划，何时拆成多个互斥计划
- 外部研究结论与观察记录冲突时如何处理

## 4. 执行

### 输入

- 经 `PLAN` 内必要确认、计划分歧处理和风险重算后确认的决策结果
- 当前活动决策流
- 当前交易计划
- 用户确认的执行意图
- 持仓语境下的再执行动作
- 交易所 API 返回的订单、成交和撤改状态

### 输出

- 订单记录
- 订单组或组合部分状态记录
- 执行动作记录
- 等待成交态、恢复所需状态、API 对账结果或成交结果

### 管理对象

- 交易链
- 当前活动决策流
- 交易计划
- 阶段状态
- 订单

### 流程交互

- 来自 `决策`
- 在收到用户执行确认消息后，可调用币安 API 核实执行结果
- 进入等待态、获取账户信息、`OBSERVE / PLAN` 或 `复盘`
- 可在持仓语境中重复发生

### 需要详细设计

- 执行建议、用户反馈和订单生命周期如何区分
- API 回报、用户反馈和本地持久化状态冲突时谁优先、如何留痕
- 执行只负责订单层等待，不负责条件等待
- 组合交易计划如何展开成订单组、执行顺序和部分成交状态
- 撤单、改单、失效如何回写主线
- 执行结果如何保持链不断开

## 5. 复盘

### 固定要求

- `复盘` 默认先还原时间顺序：`当时能看到什么`、`当时为什么会这么判断`、`后来发生了什么异变`、`事后验证结果如何`；不能直接用后验结果覆盖当时判断。
- `复盘` 不能只复述“最后涨了 / 跌了”。至少要把 `高周期静态结构`、`盘中动态异变`、`执行位置与方向优势`、`最终结果` 分层拆开。
- 当用户用真实盘中结果 challenge 系统先前判断时，复盘要能显式回答：是原判断本身错了，还是判断成立但盘中状态已经变了，只是当时没有及时切换解读。
- `复盘` 要支持把一次长对话里的追问链保留下来，例如从“做多还是做空更有优势”一路推进到“趋势线 + 水平线共振”“刚才为什么没看出异变”“这类异变后面怎么提前识别”；这些追问本身就是候选假设来源。
- `复盘` 产物至少分三层：`复盘解释记录`、`待验证假设`、`可进入策略池的候选策略`；不能把一次复盘里的临时想法直接当长期规则，也不默认把未验证结论跨窗口回接主流程。
- `复盘` 输出除了给人读，还要足够让主 agent 直接拿去派发下一轮 subagent；也就是说，至少要能显式提炼 `候选假设`、`假设依据`、`反例或疑点`、`优先验证问题`，避免下一轮研究又从整段聊天重新读起。

### 输入

- 来自待复盘池的已结束或阶段性闭合的交易链
- 交易记录
- 判断记录

### 输出

- 复盘记录
- 待验证假设
- 进入 `策略池` 的候选策略或分支修订建议

### 管理对象

- 交易链
- 待复盘池
- 交易记录
- 判断记录
- 候选策略

### 流程交互

- 来自主流程阶段性闭合后进入的待复盘池
- 从待复盘池中被拉起
- 进入 `策略池`，再进入 `回测`
- 与外部研究输入共享候选策略入池入口

### 需要详细设计

- 哪些结果进入待复盘池
- 待复盘池如何筛选和拉起
- 复盘记录保留什么
- 如何在同一条链里同时保留 `当时判断版本` 和 `事后验证版本`，避免复盘时把后验信息倒灌回当时语境
- 如何沿同一交易链回看计划从生成到多次改动的过程，并把动作变化映射回具体证据、反证、事件、约束和用户确认
- 如何持续对照系统原建议、用户最终执行和真实结果，并区分系统判断、用户判断与执行偏离的影响
- 如何把 edge 从观察、计划、下单、成交、仓位、持有、保护、恢复衔接和成本载体这些环节逐段拆开，区分“看对了但没赚到”到底丢在哪一段
- 如何把 `静态结构判断失效`、`盘中异变漏判`、`方向判断对但位置不好`、`位置好但不是趋势单` 这些不同问题显式分开，而不是都混成“判断错了”
- 如何从聊天式复盘里抽取可复用模式，例如 `扩张阳线先行收复 -> 贴压横盘 -> 浅回踩不破 -> 低级别放量突破`
- 如何把一次复盘里的“第一次异变信号”和“后续加速延伸”区分开，避免把延续段误记成首发信号
- 复盘产物如何进入 `策略池`，并挂到具体分支和版本
- 哪些复盘产物只保留为当前窗口解释，哪些应继续推动策略迭代

## 6. 回测

### 固定要求

- `回测` 的对象不只是一整套长期策略，也要支持从复盘里抽出来的局部模板或候选假设，例如“盘中异变模板”“突破前吸收模板”“旧压力翻转支撑模板”。
- 当前 `历史回放` 默认指按时间窗口滑动验证集、统计触发次数和结果分布的回放方式，不默认指带模拟用户交互的历史主流程陪练。
- 每次 `回测` 都必须绑定清楚：验证对象、数据范围、标的、周期、入场定义、止损定义、止盈或持有窗口、去重规则；没有这些前提，就不要给出“成功率”。
- `回测` 结果不能只报胜率，至少要同时给出 `样本数`、`触发频率`、`1R / 2R 命中`、`先止损`、`未完成样本`；样本过少时必须显式说“样本不足”，不能硬说有效。
- `回测` 要允许同一候选假设并行存在 `严格版` 和 `实战版` 等 sibling 版本，用来回答“为什么严格版样本太少，但实战版仍值得观察”。
- `回测` 要支持回答这类问题：这次盘中异变究竟是 `第一次有效信号`，还是更早已有首发信号、当前只是延续加速；否则很容易把回测对象定义错。
- 当一个策略是从单次复盘里被发现时，`回测 / 自测` 应允许先围绕该发现点做局部时间窗口验证，例如检查发现点前后若干根 K 线、若干小时或若干个同类局部窗口，确认它究竟是 `首发信号`、`弱延续` 还是纯噪音，再决定是否进入更长样本回测。
- 上述局部时间窗口验证不替代正式长样本回测，但它是很重要的前置方法论：先回答“这套接口在发现当下能不能识别到该策略，以及能识别多久”，再回答“长期样本里总共有效多少次”。
- `回测` 不只要产出统计结果，还要产出下一轮研究可直接消费的结论，例如 `继续放宽定义`、`增加延续过滤`、`拆成 sibling 版`、`只保留影子观察`、`样本不足先暂停`；否则主 agent 无法稳定驱动下一轮。

### 输入

- 来自 `策略池` 的候选策略分支版本
- 来自历史回放或用户历史交易的验证集

### 输出

- 回测结果
- 进入迭代的候选内容
- 继续回测所需状态

### 管理对象

- 策略池中的候选分支版本
- 验证集
- 回测结果

### 流程交互

- 来自 `策略池`
- 使用历史回放或用户历史交易
- 进入 `迭代`

### 需要详细设计

- 两个策略入口如何先进入 `策略池` 再共用同一回测口
- 每次回测如何绑定验证集来源
- 每次回测如何绑定具体分支、版本和回测前的策略状态
- 如何把聊天里的自然语言假设收敛成最小可检验规则，而不是要求用户先写成完整策略
- 如何支持 `严格版 / 实战版 / sibling 版` 并排验证，并保留它们之间的差异说明
- 如何对一次回测显式标注 `样本过少`、`定义过严`、`定义更贴近实盘` 这类研究结论
- 如何在回测结果里区分 `首发信号` 与 `延续信号`
- 如何围绕策略被发现的时间点，做局部窗口自测，并记录该策略在附近窗口内是 `match`、`weak-match` 还是 `reject`
- 如何把失败样本按原因聚类，例如 `突破后立刻失守`、`没有放量只有穿越`、`只是局部噪音不是结构异变`
- 回测结果如何沉淀进数据库

## 7. 迭代

### 固定要求

- `迭代` 不能因为一次复盘或一次漂亮案例就直接升格长期规则；默认先回答“这只是复盘结论、待验证假设，还是已通过验证的候选版本”。
- 当 `回测` 显示样本不足、定义过严或定义偏离实盘时，`迭代` 应支持继续保留候选版本并分叉 sibling 版本，而不是强行二选一地 `采纳 / 废弃`。
- `迭代` 产物要能落回不同层面：`观察模板`、`决策模板`、`执行模板`、`持仓语境约束`；不必把所有改动都升格成同一层的长期策略规则。
- `迭代` 要能明确给出下一轮动作：`继续观察收样本`、`仅影子使用`、`限定语境小范围使用`、`升格为当前分支版本`，以及回主线时是否需要额外读取账户事实。
- `迭代` 同时也是研究闭环的闸口；它必须明确回答当前候选版本是 `继续派发下一轮 subagent`、`暂停等待新样本`、`回到主流程影子使用`、`正式升级` 还是 `归档`，不能只给一段模糊总结。

### 输入

- 回测结果
- `策略池` 中相关分支和版本的当前状态

### 输出

- 下一轮可使用的策略结果
- 回写到 `策略池` 的分支、版本和状态决议
- 下一轮主线回接落点

### 管理对象

- 策略池
- 策略分支与版本
- 回测结果

### 流程交互

- 来自 `回测`
- 回写 `策略池` 后重新进入后续主线使用或后续策略流程
- 为后续 `观察 / 决策 / 执行` 提供当前适用的生效分支版本
- 可显式回接 `观察 / 决策`，并由后续 checklist 决定是否获取账户信息

### 需要详细设计

- 哪些回测结果能进入迭代
- 迭代后的内容如何回写 `策略池` 并重新被主线使用
- 当前生效分支版本如何切换、回退或保留旧版对照
- 如何把一次复盘里抽出来的局部模式回写成 `观察层提示`、`盘中异变 checklist` 或 `执行层约束`，而不是只能回写成整套策略
- 当严格版样本太少、实战版样本够但质量一般时，如何决定是继续收样本、只做影子版本，还是允许有限进入主线参考
- 如何记录“为什么这次不直接升级为长期规则”，避免后面重复讨论同一个候选模式
- `继续观察 / 影子恢复 / 小仓恢复 / 恢复常规` 如何由迭代结果决定

## 8. Infra 层

### 8.1 作用

在 0-7 章的业务流转收敛之后，提供统一承载；不反向定义业务链路本身。

### 8.2 数据库

#### 输入

- 流程控制确定下来的当前状态
- 2-7 章最终确定的长期对象和事件

#### 输出

- 持久化状态
- 历史记录
- 上下文恢复所需数据
- 供主流程直接读取的结构化上下文，以及账户事实快照缓存
- UI 界面读取的数据

#### 管理对象

- 交易链
- 阶段状态
- 订单
- 持仓
- 交易记录
- 判断记录
- 恢复上下文快照
- 最近讨论摘要与未闭合追问链
- 决策流、plan 版本与分歧记录
- 策略池
- 策略分支与版本
- 验证集
- 回测结果

#### 需要详细设计

- 哪些对象长期保存
- 各对象如何挂接
- 用户回来时最少恢复哪些内容
- 聊天线程、压缩摘要、未闭合追问、决策流、plan 版本和判断记录如何结构化挂到同一条交易链

### 8.3 UI 界面

#### 输入

- 数据库中的长期数据
- 流程控制中的当前状态

#### 输出

- 查看结果
- 基础 CRUD 结果

#### 管理对象

- 被允许展示和编辑的长期数据对象

#### 需要详细设计

- 第一批展示哪些对象
- 哪些对象允许 CRUD
- 如何展示当前链路、历史样本、策略池分支版本、回测结果
