# PRD

## 文档定位

本文件只回答“做什么”，不回答“怎么做”。

当前 PRD 以 `vision` 为上层输入，先收敛产品需要具备的核心能力、关键流程与约束，不展开系统结构、技术实现、接口协议或数据模型。

当前范围也需要明确区分两层：

- 主系统当前服务于加密货币交易协作
- 系统可以读取更广泛的跨市场信息，作为判断背景与上下文输入
- 其他市场的信息可以参与分析，但现阶段不默认进入和加密交易同等完整的正式判断、持续跟踪、复盘与方法演化链路

## 当前目标

产品需要支持一个最小但完整的加密交易协作闭环：

1. 用户发起问题
2. 系统补齐判断所需上下文
3. 系统给出正式判断或明确观望
4. 被采纳的判断进入持续跟踪
5. 已结束或已失效的判断进入复盘
6. 必要时把复盘结论反馈回后续方法

## 核心能力

### 1. 短入口提问

用户应该可以直接用很短的话发起市场或交易问题，而不需要每次先重新描述完整背景。

要求：

- 用户入口可以非常短
- 只要问题已经指向真实资金决策，系统就必须按正式判断标准处理
- 系统不能把“入口短”理解为“可以降低分析严谨度”
- 用户问题既可以直接来自加密市场，也可以由其他市场背景触发对加密交易的再判断

### 2. 上下文补齐与共享底板

在给出单笔正式判断之前，系统需要先形成一层共享的决策底板，用来回答当前大环境下是否值得继续承接风险。

这一层至少要覆盖：

- 当前市场状态
- 当前作战主线
- 当前最值得跟踪的关注对象
- 当前策略优先级与风险预算
- 当前校准与风控状态

要求：

- 系统应先回答“当前整体应该怎么参与”，再回答“这一笔好不好”
- 共享底板可以吸收加密市场之外的宏观、跨市场信息，但当前主要服务于加密交易判断
- 当关键前提变化时，系统应允许整组底板被更新
- 正式判断应能看出自己依附于哪一轮共享底板

### 3. 正式判断输出

当问题已经足以进入正式判断时，系统应输出结构化结论，而不是只有模糊观点。

正式判断至少需要覆盖：

- 当前立场
- 主要依据
- 反向证据或失效前提
- 还在等待什么催化剂或条件
- 是否值得行动，还是应继续观望
- 所属时间尺度
- 与当前组合预算的关系

要求：

- 系统必须区分“有观点”和“值得出手”
- “不做”或“继续等”必须被视为正式、有效的结论
- 判断质量、执行质量与最终结果必须允许被分开讨论
- 当前默认被体系化承接的正式判断对象，是加密货币交易机会

### 4. 决策链管理

当一条判断被用户采纳、进入执行、进入持仓管理，或被明确要求持续跟踪时，系统需要把它纳入正式决策链。

当前决策链节点包括：

1. 观察
2. 判断
3. 计划
4. 触发
5. 执行
6. 持仓更新
7. 结束

要求：

- 系统应允许多个并行判断共存
- 同一条链不要求每次都写满所有节点
- 计划可以直接结束
- 持仓更新可以反复发生
- 系统要能区分“想法”“计划”“真实动作”“结果”
- 当前默认进入正式决策链持续管理的，主要是加密货币交易判断

### 5. 复盘

当计划失效、判断被否定、仓位结束、止损止盈触发，或验证窗口走完时，系统应支持把该链路转入复盘。

要求：

- 用户可以显式要求复盘
- 系统也应能识别哪些历史判断已经到达可复盘状态
- 复盘输出不能只看盈亏，还要区分判断质量、执行质量与结果归因

### 6. 反思入策

当复盘结果已经足以影响后续方法时，系统应支持把复盘结论反馈回后续策略层。

要求：

- 不是每次复盘都必须入策
- 只有结论具有可迁移、可重复、值得改变行为的意义时，才进入这一步
- 入策时需要能说明它修正了什么、适用于什么条件、证据强度如何

### 7. 三类核心对象

产品当前需要围绕三类对象组织信息：

- 市场快照：保存当时看到了什么
- 决策链：保存一次判断如何演化
- 策略链：保存方法如何延续和修正

要求：

- 决策节点应基于“当时可见”的上下文
- 历史判断应允许影响后续策略修正
- 并非每次临时分析都必须生成完整对象链路
- 跨市场信息可以进入市场快照或判断背景，但当前链路主对象仍以加密交易相关判断为主

## 关键用户流程

### 流程一：新问题进入

1. 用户发起市场或交易问题
2. 系统判断这是不是一个真实资金决策问题
3. 如果是，系统先补齐上下文并建立共享底板
4. 系统输出正式判断，或输出继续观望的正式结论

### 流程二：判断进入持续跟踪

1. 用户采纳判断，或该判断已进入执行 / 持仓 / 明确跟踪
2. 系统把它纳入正式决策链
3. 后续更新围绕同一条链继续发生

### 流程三：链路结束与复盘

1. 判断、计划或仓位到达结束条件
2. 系统把该链路标记为可复盘
3. 用户或系统发起复盘
4. 若结论足够强，再进入反思入策

## 非功能要求

- 对真实资金决策保持正式判断标准
- 不因用户输入简短而降低严谨度
- 支持长期上下文连续性
- 支持并行判断，而不是只保留单一路径
- 保持时间尺度清晰，避免不同周期观点混淆
- 不把记录数量当成目标，只在需要持续管理时进入存储
