# Product Vision

## 一句话定位

面向开发者自用、由高级 skill 包组成并运行在 agent 工作区里的加密交易工作台，把零散的观察、决策、执行与复盘，沉淀为可验证、可迭代的策略资产。

其中，`策略资产` 是核心资产；观察、决策、执行、复盘、验证等链路负责承接真实交易过程，并持续把经验沉淀回策略资产。

## 真正缺什么

我不缺信息，也不缺观点。缺的是：真实资金决策发生时，没有一套能把观察、决策、执行与复盘接成连续链路、并持续沉淀为策略资产的工作台。

断裂最常见的三个时刻：

- 手上有仓位，但已经记不清当初为什么那样判断
- 交易结束后说不清是判断错了、执行乱了，还是运气差
- 想跟进一个标的，却要从头重建上次的思路

今天这些内容散落在 TradingView + 笔记 + 实时消息流之间，信息很多，但交易过程与策略迭代过程是断的。这不是使用习惯问题——现有工具本来就没有被设计成能承接连续判断链路的工作台，它们各自解决了信息获取或记录的局部问题，但没有一个环节负责把观察、决策、执行与复盘接成一条不断裂的链。

现有工具各自解决了局部问题，但没有一个负责把链路接起来：

| 工具 | 解决了什么 | 断在哪里 |
|------|-----------|---------|
| TradingView | 图表、价格数据、提醒 | 没有"为什么这样判断"的记录，也没有判断演变的追踪 |
| 笔记（Notion / Obsidian 等） | 可以记录思路 | 被动存储，不能主动召回相关上下文，也无法和市场数据联动 |
| 实时消息流（新闻 / 快讯 / 社媒 / 群聊） | 实时新闻、观点、讨论与市场异动线索 | 高度噪声、按时间直列、无个人链路归属，判断散在时间流里无法回溯 |
| TradeNote / Edgewonk 等交易日志工具 | 结构化记录历史交易，提供 P&L 统计与事后复盘 | 设计范式是"事后记录"而非"实时承接"——要求用户在交易结束后主动填写，而不是在判断发生时就介入；没有判断链的概念，只有交易结果；不能在新判断时主动召回历史上下文；距离高意图时刻最远 |

这个表里缺什么是值得注意的：没有任何一个工具被设计成在"高意图时刻"主动介入、承接判断并把它接入后续跟踪链路。交易日志工具是目前最接近"链路记录"的尝试，但它们要求用户先完成交易、再回头填写，本质上仍是被动归档，而不是连续承接。判断链路最容易断裂的时刻——"为什么当时那样看"——恰恰是这类工具最不在场的时刻。

## 为什么是现在

这个方向成立，依赖三个条件同时成立：

**工作区能承接历史上下文。** 这是最晚成立的一个。在 2025 年底之前，包括 Cursor 在内的主流 agent 工具每次会话结束后都会重置项目上下文。Claude Code 的跨会话持久化记忆能力在 2025 年底才进入可用阶段。在这之前，"把判断链路沉淀下来"这件事没有合适的基础设施。

**工具调用可靠到可以用于真实决策。** 2024 年之前，LLM 的 function calling 幻觉率过高，不适合在真实资金场景下调用结构化数据。2024 年主流模型（GPT-4o、Claude 3.5、Gemini 1.5）在 Berkeley Function Calling Leaderboard 上进入生产可用区间，GPT-4.1 的幻觉率降至 6.2%，较上一代减少约 40%。

**加密市场的连续性本身是结构性压力。** 24/7 不停盘不只是交易时间更长，而是一种有文献支撑的认知负担来源：研究发现，持续的价格监控会直接导致决策疲劳、情绪性交易和睡眠中断，高杠杆持仓者尤为显著。这不是个人习惯问题，是结构性的——没有持续承接链路的工作台，在这个市场里所有判断都会随波动频率一起散掉。

这三个条件在 2025 年底到 2026 年初同时成立。

## 它最小长什么样

当前阶段的最小成立形态，不是一个完整交易操作系统，也不是一个覆盖所有链路细节的全能助手，而是一个能把高意图判断正式承接下来、并让这次判断以后不用从零重建的最小闭环。

这个最小闭环当前必须成立的范围只有四件事：

- 能承接高意图输入，而不是把它当普通聊天直接回答完就结束
- 能判断这次输入应挂回旧链还是开启新链，并补齐形成正式判断所需的最小上下文
- 能把正式判断保留下来，并在后续变化时继续追加更新，而不是覆盖原判断
- 能在这轮管理需求结束后进入复盘，并让复盘结论有机会继续沉淀为策略资产

当前阶段不要求一开始就把所有链路都展开到同等完整度，也不要求先做出独立界面、复杂运营能力或面向多市场的完整闭环。只要高意图判断能够被承接、延续、回看、复盘，这个方向就已经以最小方式成立。

在这个范围下，用户打开时，第一眼看到的不是"今天想问什么"，而是当前有哪些活跃链路、最近发生了什么变化、下一次为什么值得回来。

比如丢一句"ETH 现在还拿吗"，系统不该像第一次见到我一样从头回答，而应该先检索相关旧链路与对应策略上下文，判断这是旧决策链的延续、一次执行偏离、一次复盘触发，还是一条新链开始；如果是旧链延续，就先把上次观察、正式决策、之后执行与当前缺口接起来，再只问最少一个真正影响结论的问题；如果是新链，就立刻把它接入之后还能继续更新、验证和复盘的链路，而不是让这次判断再次散掉。

**冷启动阶段**（没有任何历史链路时）也一样成立：系统的价值不是"回看历史"，而是"这次交易过程不再散掉"。从第一条观察链或决策链起，这次判断就被正式承接、可以被下次召回、可以复盘，并有机会进一步沉淀为策略资产。这个价值从第一天就存在，不需要等历史积累才成立。

## 产品不能只有主张

这不是一个只会聊天的分析工具，也不是先把差异化路线讲清楚、再把基础底盘留到以后补的产品。要让主线真正成立，产品还必须具备足够支撑正式判断、持续跟踪、复盘验证的基础能力。

即使当前主战场是加密交易，正式决策也不能只看单一标的的局部信息。系统至少需要能够读取跨资产市场背景，拿到多资产的最新价格与必要上下文，用来形成更完整的判断底板。

同样，执行事实不能长期依赖用户事后手工转述；验证与复盘也不能只有主观总结。系统必须以低摩擦方式承接真实执行结果，并提供最小但可信的验证与统计支撑，让用户看见相近情境下过去发生过什么、常见错误是否在收敛、哪些经验只是偶发、哪些已经值得进入策略资产。

这些基础能力不是为了追求"大而全"，也不是为了在功能列表上与成熟竞品逐项对齐；它们存在的意义，是让这个产品不是悬空的理念，而是真能承接真实资金决策的工作台。

## 什么叫高意图

高意图的标志不是用词正式，而是输入背后有真实资金在等待判断。

以下信号单独出现就可以直接判定为高意图：
- 有明确的仓位或计划（"我现在拿着 ETH"/"准备进 SOL"）
- 有明确的行动意向动词（"还拿吗"/"该减吗"/"要进吗"/"止损位怎么设"）
- 输入指向已有活跃链的对象（"上次说的 ETH 现在怎么样了"）

以下信号单独出现通常不是高意图：
- 纯好奇性提问（"ETH 的技术路线图是什么"）
- 他人视角的观点征集（"大家怎么看 BTC"）
- 不涉及具体标的的宏观讨论

边界模糊时（有标的、有看法，但没有明确仓位或行动意向），系统默认按高意图承接，并在重建上下文后加入一个最小探针——"你现在有仓位，还是在看入场机会？"——作为判断底板的一部分，而不是入口门槛。这个探针只在没有现存相关链路且输入缺乏仓位上下文时才出现；如果已有相关活跃链，就直接挂接、不再额外问。

宁可误判为高意图，也不要漏掉。

## 核心对象

这个系统的核心不是单条 `决策链`，而是持续演化的 `策略资产`，以及围绕真实交易过程展开的多条工作链。

其中最重要的几类链路是：

- `观察链`：现在在看什么，为什么值得看，什么条件会触发正式判断
- `决策链`：当前判断是什么，建立在什么依据上，什么变化会触发更新或失效
- `执行链`：实际怎么进、怎么拿、怎么减、怎么出，哪里偏离了原计划
- `复盘链`：结果该归因到策略、执行、情绪还是运气，下一步要修什么
- `验证链`：一个经验是否值得升级为更稳定的策略规则，还是只是一段情境性有效经验

这些链路不是彼此孤立的记录，而是共同服务于 `策略资产` 的形成、分叉、验证、修正与淘汰。

如果要理解长期使用后的复杂度，可以看这张单独整理的示意图：[复杂链关系图](./complex-chain-map.md)。它展示的不是一条无限变长的链，而是多条链、多个快照、执行事件、偏离事件与策略版本之间的关联网络。

`观察链` 是入口最宽、门槛最低的链路，也是最容易在没有结构支撑的情况下流失的链路。一条观察链至少需要记录：当前在看什么对象、是什么触发了这次关注（价格行为、宏观变化、他人观点，还是自己的直觉）、什么条件会让它升级为正式决策、什么条件会让它直接终止。观察链可以长期挂起，不需要立即进入决策阶段；但它必须被正式承接——而不是散落在笔记或对话里等待被遗忘。

`执行链` 附着在决策链上，记录计划如何落地：何时进场、以什么价格、多大仓位、中途有没有调整（加仓、减仓、移动止损），以及最终如何离场。执行链最重要的信息不是结果本身，而是实际执行相比原决策计划的偏离点，以及偏离背后的原因——是市场条件变化、是执行犹豫，还是情绪干扰。没有执行链，复盘时就无法区分"判断错了"和"执行乱了"，归因会一直是模糊的。

`复盘链` 附着在已结束的决策链上，不是独立漂浮的事后总结。它必须回答四个归因问题：结果主要由什么决定、当时判断质量如何、执行是否偏离计划、是否存在明显情绪干扰。复盘的结论不是"这次赚了"或"这次亏了"，而是把结果分配到策略、执行、情绪、运气各自承担的部分，并判断有没有值得提议进入策略资产的经验。

`验证链` 负责回答：一条复盘里得到的经验，到底只是一次情境性有效，还是已经值得升级为稳定策略规则。它不是用户手动新建的一条独立链，而是系统对相近复盘结论的跨链验证与聚合；只有来源证据积累到足够一致时，系统才应该提议升级，用户确认后再正式进入策略资产。

在这些链路里，`决策链` 最直接承接高意图输入。一条决策链由一串按时间延续的决策快照组成。每个决策快照至少包含当时的市场快照、所采用的策略上下文，以及最终形成的交易决策。

一次高意图问题被正式承接时，相关链路开始形成；真实管理需求结束后，这一轮决策与执行才算结束。结束后可以进入复盘与验证，但这不改变"结束"的定义。

一条决策链不是一份会被反复改写的当前答案，而是一串按时间延续的判断快照。过去时间点的正式判断一旦进入链路，就不应被事后覆盖；之后无论因为市场变化、执行变化，还是情绪触发了重新审视，都只能作为新的判断快照继续附着在原链上。

这也意味着，情绪可以触发复核，但不能单独构成改写决策的依据。系统必须先回到原判断所属的时间尺度、持仓目的和失效条件，再决定这次变化究竟是合理更新，还是一次应该被识别出来的情绪扰动。

这些链路共同服务于两层积累：

`交易链路资产`：历史观察链、决策链、执行结果、复盘结论，以及每个关键时间点留下来的决策快照。默认保留和承接，优先保证链路不丢。

`策略资产`：会直接影响后续默认行为的稳定规则集合。默认保守演化，优先避免把一次性有效经验过早升级成长期策略。

两者的分界不能是黑盒：明显带有情境性的经验继续留在交易链路资产；只有那些已经在多个相近情境里反复显示出稳定作用、值得影响后续默认判断方式的东西，才应该进入策略资产。策略的升级、分叉、切换与回退都必须慎重、可见、可确认。

`策略资产` 内部分三类，各自服务于不同的判断层次：

- `signal_rule`：特定市场形态下的入场或出场触发条件，例如"在什么结构下值得建仓"或"什么信号出现后应该离场"
- `risk_rule`：仓位、止损、加减仓的边界约束，例如"单标的最大仓位比例"或"亏损达到多少时强制止损"
- `cognitive_rule`：判断偏差或情绪模式的行为约束，例如"价格剧烈波动时不做仓位决策"或"已连续两次绕开系统判断自主开仓亏损时，第三次必须先经过正式决策链再执行"

`cognitive_rule` 是三类规则里最独特的一类，也是其他交易工具基本缺席的一层。这不是技术限制，而是这类规则要真正有约束力，必须同时满足两个条件——在其他工具里都无法同时成立：

**规则必须来自你自己的行为证据，而不是通用建议。** "不要情绪化交易"对任何人都没有约束力，因为处于偏差状态的人总会认为自己的这次不算。有约束力的 cognitive_rule 来自具体链路：第 3、7、12 条复盘都显示，在市场剧烈波动的窗口内我自主改变了仓位，归因都是情绪干扰而非市场结构变化——这条规则来自你自己记录在案的 source_chains，你没法辩驳。

**规则必须在高意图时刻被主动带入，而不是靠你自己想起来。** 交易日志也可以记录行为模式，但它是被动的——处于高压决策时刻的人不会先去翻笔记确认自己是否正在偏差。这个系统在承接每一次高意图输入时，相关 cognitive_rule 已经进入了判断底板，不依赖你主动触发。

`cognitive_rule` 处理的不是"判断方向对不对"，而是"我知道该怎么做，但还是做了另一件事"——这是加密市场 24/7 持续压力下极其常见的失效模式，也是绝大多数交易工具设计视野里根本不存在的一层。

三类规则适用不同的升级标准：`signal_rule` 和 `risk_rule` 需要在多个相近市场情境里验证；`cognitive_rule` 更多来自复盘归因，只要自己的行为模式被识别出来且有足够的一致性，就值得进入。

## 核心立场

- 只要问题明显指向真实资金决策，系统就先承接
- 系统先承担大部分上下文重建成本，不把整理负担压回给用户
- 只有真正影响判断质量的关键歧义，才值得再抛回确认

这套系统不是把我当下的风险冲动直接翻译成建议，而是先理解我此刻真正想要的交易形态，再判断这种要求是否仍然值得被严肃承接。即使我明确想要更高杠杆、更短周期、更激进的打法，系统也不能因为迎合我而降低判断标准；它可以拒绝、劝退，或者改写为一个更合理的新决策。

同一个我，在不同时间尺度、持仓目的和风险偏好下，可能同时存在多套并行但不应混淆的策略模式。系统要承接的不是一个抽象统一的"我怎么看市场"，而是不同模式下各自独立的链路集合、执行约束与复盘标准，避免用相近但并不相同的策略彼此污染。

回拉不是可有可无的提醒，而是正式链路的一部分。只要链仍处于活跃管理状态，系统就应该对"下一次为什么值得回来"有默认承接。

## 关键承接挑战

关键不在于系统自动判对一切，而在于：当已经积累了大量历史链路后，这次高意图输入进来，系统能不能先把复杂历史压缩成足够小、足够清楚的候选后续动作。

这些候选动作通常不是抽象地回答"属于哪条链"，而是更贴近真实承接方式：是在某条旧链的末尾追加一次更新、在原链上记录一次执行或偏离，还是开启一条新链。

这个方向的押注不是"一开始就能完美自动挂接"，而是：只要系统能在大多数高意图场景里，先重建出足够小的候选链路上下文、候选策略上下文与候选挂接动作，再通过一次低摩擦确认让用户完成选择，就已经足够可用。

如果长期做不好，最小可接受降级是：系统仍然负责召回和重建候选上下文，但把最终挂接动作的确认明确交还给用户。如果连这种"候选收敛 + 用户确认"的半自动承接都不成立，这个方向就不再是"策略资产工作台"，只剩下一次性分析工具。

## 产品成立依赖的关键假设

以下假设如果不成立，方向需要重新评估：

1. **行为改变可以发生**：用户在真实高压决策时刻仍会先进入这个工作台，而不是直接操作。如果这个习惯建立不起来，入口不成立。
2. **检索质量足够可用**：新链/旧链归属依赖检索，如果候选结果长期不准、每次都需要大量纠正，"低摩擦"就是假的。
3. **早期价值足以支撑坚持**：系统价值随历史沉淀增长，但如果冷启动阶段价值不足，用户会在积累起来之前放弃。

## 成功长什么样

最终要帮助我在长周期里更稳定地赚钱：

- 是否比过去更少重复犯同一种判断错误
- 是否能更稳定地区分判断问题、执行问题和情绪问题
- 结果是否更稳，而不是只靠几次运气好的收益

当前阶段的可观测 proxy：在一个可比的 30 天窗口里，高意图决策问题中有多少被承接成了可回看的交易链路，并至少发生过一次有效更新、执行记录、复盘回看或验证沉淀。更关键的是，这种承接是否真的减少了从零重建上次判断的成本，并逐步把有用经验推进成更稳定的策略资产——如果链路虽然建起来了，但我仍然经常需要重新解释自己，或者始终沉淀不出稳定策略，这个方向依然不成立。

## 使用场景与用户形态

这是一套运行在 agent 工作区里的 skill 集合，没有独立界面。用户在 Claude Code、Codex、Gemini CLI 等 agent 环境中配置好 skill 集合后，直接通过这些工具自己的聊天窗口、命令行或移动端入口使用。

### 为什么是 skill，而不是独立 app

这是一个有意识的选择，不是默认值。

独立 app 的问题不是技术上做不到，而是它天然要求用户在"需要判断"的时刻，先离开当前上下文、打开另一个工具、重建语境，然后才能开始分析。这个切换成本在高压决策时刻是真实的摩擦，而高压决策时刻恰恰是最需要减少摩擦的时刻。

更根本的问题是：独立 app 需要自己解决 LLM 接入、跨会话上下文持久化、工具调用、结构化数据存储等基础设施问题。这些能力在 2025 年底的主流 agent 工作区里已经基本具备。在这些基础设施已经存在的情况下，选择重建一遍是在为已经解决的问题付出重复成本。

这个选择的代价是真实的：
- 没有独立界面，信息呈现完全受限于 agent 工具的输出格式
- 发现路径更长，无法通过应用市场分发，用户需要主动配置
- 对使用习惯有要求——用户必须已经在用某个 agent 工具

在当前阶段，这些代价是可以接受的：目标用户是开发者自用，本来就生活在 agent 工具里；省去自建基础设施，意味着可以更早把精力集中在判断链路本身的设计上。

这个选择未来是可以重新评估的。如果使用规模超出 agent 工具的承载范围，或者界面需求变得不可回避，独立 app 仍然是合理的演化路径。

### 设备与场景

电脑和手机都是正式入口，不存在"电脑维护主状态、手机只做补充输入"的分工。

关键不在于设备本身，而在于是否进入同一个远程工作区、读取同一份链路状态，并调用同一套 skill。只要宿主 agent 工具支持在手机端选择仓库、发起任务并进入远程环境，手机上的这次输入就和电脑上一样，是对同一套链路资产、策略资产与数据库状态的正式访问，而不是一个降级副本。

这点很重要：手机不是"先记一下，回头再分析"的临时入口，而是一等入口。高压时刻发生在手机上时，系统仍然要按同样标准完成正式承接、上下文重建、判断输出、链路更新与后续回拉，而不是把核心能力留给电脑。

具体来说：
- 电脑和手机访问的是同一个远程工作区与同一份持久化状态，不维护两套分叉链路
- 手机端可以完成完整分析，不只是快速记一笔或占坑
- 链路、策略资产、执行记录与数据库读写在多端之间保持同一来源，不因设备切换而分裂
- 设备差异只影响当下交互便利性，不改变系统对这次输入的承接标准

在当前前提下，不需要把手机单独视为一个产品问题。真正需要关心的不是"手机能不能用"，而是无论用户此刻拿着什么设备，系统都必须承接到同一条正式链路上，避免状态分裂、上下文丢失或策略资产失配。

## 当前边界

当前主战场是加密交易协作，运行在 agent 工作区里的持久化 skill 集合。

其他市场信息只进入两种层级：

- 作为加密判断的背景输入
- 在明确要求时作为单次分析对象

默认不进入完整交易链路、策略资产和持续复盘闭环。
