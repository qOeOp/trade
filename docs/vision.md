# Product Vision

## 快速总览

产品主体是一组运行在 agent 工作区里的 skill，通过 Claude Code、Codex、Gemini CLI 直接调用，不做从头到尾包办的独立 app。

当前优先把判断记录、上下文召回、research、策略生成、回测、复盘这些核心能力跑通，再考虑更重的界面。

产品主线只覆盖在线交易推进：观察、决策、执行、复盘；回测、迭代属于离线演化链路。持仓后的推进仍属于主线，只是运行在持仓语境下。外部研究输入和历史回放分别为主线、复盘、回测提供补充材料与历史语境。

主战场是加密交易；其他市场只作为背景参考或单次分析，不进入完整交易链路。

## 产品形态

### 核心载体

产品主体是一组运行在 agent 工作区里的 skill，通过 Claude Code、Codex、Gemini CLI 直接调用，不做从头到尾包办的独立 app。

这些 skill 保持单一职责，由 agent 负责调度和编排。当前先把判断记录、上下文召回、research、策略生成、回测、复盘这些核心能力跑通，再考虑更重的界面。

### 数据与视图

在这层能力之上，可以有单独看板，展示长期积累的数据，并支持基本查看、增删改查。

长期数据更适合放在服务器侧统一存储；本地 agent 负责读写，看板读取同一份数据做展示。这样能力层、数据层和视图层可以协同，但不绑成单体。

### 终端与市场边界

电脑和手机共用同一个远程工作区和持久化状态，不分叉。

主战场是加密交易，其他市场只作为背景参考或单次分析，不进入完整交易链路。

## 产品主线

### 主线定义

产品主线只覆盖在线交易推进：观察 -> 决策 -> 执行 -> 复盘。

这里的观察、决策、执行、复盘说的是主线阶段。`WAIT-CONDITION` 和 `WAIT-UNTIL-FILL` 说的是当前 plan 或订单的等待状态，需要显式记录，但不单独升格为新的主线阶段，也不阻断其他 plan 继续流转。

决策不是单点结论，而是围绕当前机会反复运行 `OBSERVE <-> PLAN <-> ALIGN` 的收敛过程。若在 `PLAN <-> ALIGN` 里发现上下文不足，还会回到 `OBSERVE` 补信息，再重回 `PLAN <-> ALIGN` 继续收敛。

执行从真正开始挂单，或以市价等方式直接进入仓位时才算开始。等挂单条件但还没挂单，仍属于决策收敛后的条件等待状态，不算进入执行。

重点是保证链条不中断，用户下次回来还能继续。

在线主线里，观察、决策、执行会不断往返；复盘则承接阶段性闭合后的样本。

### 每次进入系统时要做的事

用户每次进入系统，都先基于现有持仓、挂单、既有判断、最新市场数据恢复并对账当前交易链。

完成恢复后，先判断当前处于哪一个在线阶段，或哪个 plan 仍处于条件等待 / 订单等待状态，再继续推进。

### 主线内的返回关系

开仓前，观察和决策会反复往返；没有操作建议、不建议操作、放弃计划，或等待条件失效、错过、被替代后，都回到观察或决策；等挂单条件但还没挂单时，plan 处于条件等待状态；订单挂出但未完成时，plan 处于执行后的订单等待状态。这两类等待都需要显式记录，但不单独升格为新的主线阶段。

持仓后，主线会继续观察、继续决策，并继续执行新的动作。常见动作包括加仓、减仓、调杠杆、止损、止盈；这些动作都属于持仓语境下的新一轮决策与执行。

### 订单状态

订单状态是执行里的附属对象，不是主线阶段。可先按创建、挂出、等待成交、部分成交、全部成交、撤销、改价、失效来理解。

订单状态变化时，会推动主线返回观察、进入新一轮决策，或在持仓语境下继续执行新的动作。

### 离线演化

复盘、回测、迭代不属于在线主线，而是策略演化链路：

- 复盘：整理已发生的交易与判断，产出问题、假设和候选改动
- 回测：验证候选改动或候选策略是否成立
- 迭代：把通过验证的改动沉淀为可继续使用的策略版本

已结束的交易，以及已经和用户完成对齐、但最终没有进入执行的详细 plan，都可以视为阶段性闭合样本，进入后续复盘和策略演化。

### 策略池

策略池不是流程阶段，而是承接离线演化结果的长期容器。

策略不应只是一份全局单体规则。更合理的形态是一个带分支和版本管理的策略池。

池中的对象是策略，但这里的策略不只是几句开仓和平仓条件，而是一个可版本化的规则组合，至少包含适用语境、标注规则、判定规则、执行规则。

不同分支可对应不同交易标的或不同市场环境。主线每次只读取当前语境下适用的生效分支版本。

外部研究输入和复盘产出的候选策略先进入策略池，再经回测和迭代决定是否升格。

### 策略迭代的层面

策略迭代可能发生在不同层面：标注规则，例如 box 用 wick 还是 body、先看高周期还是先看低周期；判定规则，例如触碰几次才有效、跌破还是收盘跌破才失效；执行规则，例如直接挂单还是等待更低周期确认。

现阶段先明确这种理解，不预先把它固化成过细的数据结构。

## 输入来源与方法

在主线和离线演化链路之外，还需要两类长期存在的能力：外部研究输入，以及历史回放方法。

外部研究输入负责把主线之外的材料和想法接入系统；历史回放负责在锁住未来信息的前提下重建历史语境，为复盘和回测提供样本与验证集。

外部研究输入有时会进入“候选策略 -> 回测 -> 迭代”，有时则直接回接主线；历史回放主要服务复盘和回测，也可在需要时为主线判断提供历史参考。

### 外部研究输入

外部研究输入不依赖实盘持仓触发，负责消化外部材料、整理不同来源的想法，并把它们转成可供主线使用的研究结论；其中一部分会继续沉淀成可回测的策略。

外部研究输入的产出不只有策略，还可以是：对某个标的当前语境的研究判断；对宏观、流动性、地缘政治、监管或市场报告的解读；对其他市场变化如何影响加密标的的解释；候选假设、观察重点或风险提示。

策略来源包括：用户自己提出的假设、agent 发散出的假设、社区流传的策略、网络、论坛、论文里的公开策略。

### 历史回放

历史回放不直接回测策略，也不只是给回测准备数据。它是在锁住未来信息的前提下，从历史市场数据中选取时间点开始，屏蔽后续 K 线，让用户和系统在当时语境里重跑观察、决策、执行与复盘，并用模拟单推进。

回测使用的策略来源主要是主线复盘与外部研究输入。

回测输入包括用户自己的历史交易记录，以及从历史市场数据中选取时间点开始的历史回放语境。
