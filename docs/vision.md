# Product Vision

把交易判断变成可持续演化的记录，而不是一次性的分析回答。

## 核心用户

长期生活在 agent 工作区里的自用型交易者：

- 日常在 Claude Code、Codex、Gemini CLI 里工作
- 盯少数标的，手里有真实仓位，判断直接影响资金
- 不缺信息和分析能力，缺的是判断能被连续跟踪

## 痛点

同一个断裂反复发生，只有三种形式：

- 仓位还在，已经说不清当初为什么这样看、为什么这样拿
- 交易结束，分不清问题出在判断、执行还是运气
- 再次跟进同一标的，要从头重建上次的思路

现有工具各管一截：TradingView 管图表不记判断，笔记只能存放不能在关键时刻主动召回，消息流噪声高且无个人归属，交易日志能做复盘但接不到判断形成的当下。缺的不是更多工具，而是在做决策的当下把观察、决策、执行、复盘接成一条持续的线。

## 产品形态

一组运行在 agent 工作区里的 skill，通过 Claude Code、Codex、Gemini CLI 直接调用，不做独立 app。先用现有模型能力把判断记录和上下文召回跑通，比过早做界面更有价值。

电脑和手机共用同一个远程工作区和持久化状态，不分叉。主战场是加密交易，其他市场只作为背景参考或单次分析，不进入完整交易链路。

## 产品主线

产品做一件事：让每一次判断不断裂、可追溯、能归因。完整链路覆盖观察、决策、执行、管理、复盘、验证、迭代七个阶段，但核心不是线性流程管理，而是判断在这条线上持续演化，而不是每次重来。

这七个阶段描述的是产品主线，不等于真实运行时只有一条线性流程。实际交易链路里，还会反复进入若干状态机，但当前不预先把它们固定成一套过细制度。

当前至少需要承认两类循环与中间状态：

- 执行前，观察与决策会反复来回。没有操作建议、不建议操作，仍停留在这个循环里；形成挂单建议但尚未成交，则进入待执行监控；撤单、改挂单或放弃计划后，又会回到观察与决策循环
- 进入持仓后，管理本质上是带着仓位、计划和约束的持续观察与持续决策，并在过程中不断分叉出新的执行事件，例如加仓、减仓、调节杠杆、止损、止盈；如果形成新的待成交动作，也会再次进入待执行监控，直到链路结束

除了整条判断链路的状态变化，系统还需要承认每笔订单自身也有生命周期管理。订单不是一次性的“下单”动作，而是会经历创建、挂出、等待成交、部分成交、完全成交、撤销、改价、失效等状态，并且这些变化会反过来影响链路当前所处的观察、决策、执行与管理状态。

复盘之后，不是自动把单次经验写成长期规则，而是先验证它是否真的重复成立，再决定是否进入后续迭代。

## 产品支线

除了围绕真实交易记录持续演化的产品主线，系统还需要两条可独立运行、但能够接入主线的产品支线：`r-and-d agent` 与 `backtest`。

增加产品支线，不是为了把系统做得更大，而是因为产品在冷启动阶段会天然遇到一个问题：用户一开始既没有成熟策略，也没有足够多的交易记录。此时，主线中的复盘即使产生了有价值的假想结论，也常常跑不通完整的验证与迭代流程，因为验证层还没有足够实盘样本支撑。

### 支线一：`r-and-d agent`

这条支线不依赖实盘持仓触发。它的作用不是代替主线做交易管理，而是在脱离当前仓位的情况下，围绕社区策略、公开论文、论坛讨论，或用户自己的发散性假设，生成新的候选判断、候选规则和候选方法。

这条支线的关键，不是“多做研究”，而是产物要和主线复盘阶段产出的内容尽可能同构。也就是说，`r-and-d agent` 产出的候选经验、候选规则、候选框架，应能够像复盘结论一样进入验证，并在验证成立后作用于后续迭代，而不是停留在漂浮的灵感或资料收藏。

### 支线二：`backtest`

这条支线解决的是验证层的数据不足问题。当真实交易记录还不够时，系统需要能把候选规则、候选方法、复盘结论或 `r-and-d agent` 生成的假设，放到历史数据里做模拟验证，而不是因为缺少实盘样本就让验证阶段空转。

`backtest` 的意义，不是伪装成真实交易，而是为验证阶段提供早期可用的证据来源。它让系统在冷启动时就能开始区分“看起来有道理”和“在历史条件下至少能初步成立”的差别。

因此，验证阶段的证据来源会随着使用过程逐步演化：

- 冷启动阶段，主要依赖历史数据上的模拟盘验证与回测
- 积累一部分真实交易后，可以同时参考最近一笔或最近几笔实盘交易的回看验证
- 形成更多正式链路后，验证会越来越多地依赖用户自己的历史交易记录

因此，后续迭代的输入至少有三类：

- 来自主线中的真实交易复盘结论
- 来自 `r-and-d agent` 生成的候选产物
- 来自 `backtest` 提供的历史验证证据

主线解决的是“真实判断如何不断裂、可追溯、能归因”；`r-and-d agent` 解决的是“没有当前持仓时，如何持续生成可验证的新候选”；`backtest` 解决的是“实盘样本不足时，验证层如何先跑起来”。三者可以独立存在，但在验证与迭代层应尽量兼容，而不是各自形成孤立体系。

最小闭环六步：

1. 用户提出真实资金决策问题
2. 系统识别为正式判断输入，不作为普通聊天消费
3. 召回相关历史，判断接续旧链还是开启新链
4. 判断正式记录，后续变化只能追加，不能覆盖
5. 至少发生一次有效延续：更新判断、记录执行或触发复盘
6. 用户下次回来，不需要重新交代上次的判断
