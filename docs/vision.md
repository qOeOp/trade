# Product Vision

## 产品形态

产品主体是一组运行在 agent 工作区里的 skill，通过 Claude Code、Codex、Gemini CLI 直接调用，不做从头到尾包办的独立 app。

这些 skill 保持单一职责，由 agent 负责调度和编排。当前先把判断记录、上下文召回、策略生成、验证、复盘这些核心能力跑通，再考虑更重的界面。

在这层能力之上，可以有单独看板，展示长期积累的数据，并支持基本查看、增删改查。

长期数据更适合放在服务器侧统一存储；本地 agent 负责读写，看板读取同一份数据做展示。这样能力层、数据层和视图层可以协同，但不绑成单体。

电脑和手机共用同一个远程工作区和持久化状态，不分叉。主战场是加密交易，其他市场只作为背景参考或单次分析，不进入完整交易链路。

## 产品主线

产品主线是一条连续的交易链：观察、决策、执行、管理、复盘、验证、迭代。重点是保证链条不中断，用户下次回来还能继续。

这七个阶段是主线，不表示真实交易一定线性推进。实际使用里，用户会在几个状态之间反复往返，先承认这种往返存在，不急着把它写成过细制度。

至少有两类常见循环：

- 开仓前，观察和决策会反复往返；没有操作建议、不建议操作、撤单、改单或放弃计划后，都回到观察和决策；有挂单但未成交时，进入监控
- 持仓后，管理意味着继续观察、继续决策，并继续执行新的动作，例如加仓、减仓、调杠杆、止损、止盈；如果又挂出新单但未成交，再次进入监控

订单也有自己的生命周期：创建、挂出、等待成交、部分成交、全部成交、撤销、改价、失效。订单状态变化时，主线所处阶段往往也会跟着变化。

复盘不会直接变成长期规则；先验证，验证通过后再进入迭代。

## 产品支线

除了围绕真实交易记录持续运转的主线，系统还需要两条能独立运行、也能接回主线的支线：`r&d agent` 和 `backtest`。

冷启动时，用户往往既没有成熟策略，也没有足够多的交易记录。主线里的复盘即使产出新策略，也可能因为样本太少，暂时无法完成验证和迭代。

### 支线一：`r&d agent`

`r&d agent` 不依赖实盘持仓触发，职责就是生成策略。

策略来源包括：

- 用户自己提出的假设
- agent 发散出的假设
- 社区流传的策略
- 网络、论坛、论文里的公开策略

它把这些来源整理成可验证的策略，不让它们停留在零散想法、收藏夹或聊天灵感里。

### 支线二：`backtest`

`backtest` 用来补足验证数据。真实交易记录还不够时，系统把主线复盘生成的策略，或 `r&d agent` 生成的策略，放到历史数据里做模拟验证。

它给验证阶段提供早期可用数据，让系统先区分一个策略只是听起来有道理，还是在历史条件下至少能初步跑通。

验证数据会随使用过程变化：

- 冷启动阶段，主要依赖历史数据回测
- 积累一部分真实交易后，可以同时参考最近几笔实盘交易的回看
- 形成更多正式链路后，验证会越来越多地依赖用户自己的历史交易记录

因此：

- 策略主要来自两个入口：主线复盘和 `r&d agent`
- 验证主要来自两类数据：用户自己的历史交易记录和 `backtest` 使用的历史数据

迭代发生在策略生成之后、验证之后。策略先被生成，再放到不同来源的数据里验证，验证通过后才进入迭代。

主线解决真实交易怎么连续跟下去；`r&d agent` 解决没有当前持仓时怎么继续产出策略；`backtest` 解决实盘样本不够时怎么先把验证跑起来。三者可以独立存在，但最后汇入同一个“策略生成 -> 验证 -> 迭代”的过程。
