# Product Vision

把交易从判断、执行到复盘接成一条不断裂的链，并让策略能够持续生成、验证和迭代。

## 核心用户

长期生活在 agent 工作区里的自用型交易者：

- 日常在 Claude Code、Codex、Gemini CLI 里工作
- 盯少数标的，手里有真实仓位，判断直接影响资金
- 不缺信息和分析能力，缺的是判断能被连续跟踪

## 痛点

同一个断裂反复发生，只有三种形式：

- 仓位还在，已经说不清当初为什么这样看、为什么这样拿
- 交易结束，分不清问题出在判断、执行还是运气
- 再次跟进同一标的，要从头重建上次的思路

现有工具都只解决一小段问题：TradingView 管图表，不管你的判断；笔记能存东西，但不会在关键时刻把相关内容拉回来；消息流信息很多，但不属于你自己的连续判断；交易日志能做复盘，但接不上判断形成的当下。缺的不是更多工具，而是把一次交易从开始想到最后复盘，接成一条不断裂的线。

## 产品形态

产品主体是一组运行在 agent 工作区里的高级 skill，通过 Claude Code、Codex、Gemini CLI 直接调用，不做一个从头到尾包办一切的独立 app。

这些 skill 应尽量保持单一职责，像一块块小砖。agent 本身负责调度和编排，用这些小砖搭出最后的完整能力。这样做的目的，是让系统更容易快速迭代，也更容易拆卸、替换和维护。

如果用传统软件的语言去理解，这一层更像后端能力层。它先把判断记录、上下文召回、策略生成、验证、复盘这些核心能力跑通，再考虑更重的界面形态。

在这层能力之上，可以有一个单独的看板作为前端视图，用来展示系统里长期积累的资产，并支持不同视图查看。用户不仅要能看到这些资产，还要能围绕它们做基本的查看、增删改查。

这些持久化资产未来更适合放在服务器侧统一存储。本地 agent 会和这层数据存储交互，看板也读取同一份数据做可视化展示。这样主能力、持久化资产和前端视图可以接在一起，但又不必绑死在一个庞大的单体里。

电脑和手机共用同一个远程工作区和持久化状态，不分叉。主战场是加密交易，其他市场只作为背景参考或单次分析，不进入完整交易链路。

## 产品主线

产品主线围绕一条交易链展开：观察、决策、执行、管理、复盘、验证、迭代。产品要解决的，不是把流程画得很漂亮，而是让这条链不断裂，下一次回来时还能接着往下走。

这七个阶段是主线，不代表真实交易过程是一条直线。实际使用里，用户会在几个状态之间反复来回，当前先承认这些状态存在，但不急着把它们写成过细的固定制度。

当前至少有两类循环：

- 开仓前，观察和决策会反复来回。没有操作建议，继续看；不建议操作，也继续看；有挂单但还没成交，就进入监控；撤单、改单或放弃计划后，再回到观察和决策
- 持仓后，管理不是“拿着不动”，而是继续观察、继续决策，并不断产生新的执行动作，比如加仓、减仓、调杠杆、止损、止盈。如果又挂了新的单但还没成交，就再次进入监控

除了整条交易链，订单本身也有自己的生命周期。订单不是一次性的“下单”动作，它会经历创建、挂出、等待成交、部分成交、全部成交、撤销、改价、失效等状态。订单状态一变，主线当前处在哪个阶段，往往也会跟着变。

复盘之后，不会自动把一次经验写成长期规则。还要先验证它是不是真的成立，验证通过后，才进入后续迭代。

## 产品支线

除了围绕真实交易记录持续运转的产品主线，系统还需要两条能独立运行、但又能接回主线的产品支线：`r&d agent` 和 `backtest`。

增加这两条支线，不是为了把系统做大，而是因为产品冷启动时会遇到一个很实际的问题：用户一开始既没有成熟策略，也没有足够多的交易记录。这样一来，主线里的复盘即使产出了新策略，也可能因为样本太少，暂时跑不通验证和迭代。

### 支线一：`r&d agent`

这条支线不依赖实盘持仓触发。它的本质很简单：`r&d agent` 就是用来生成策略的。

它可以从不同来源生成策略，例如：

- 用户自己提出的假设
- agent 自己思考后发散出的假设
- 社区里流传的策略
- 网络、论坛、论文里的公开策略

它的价值不在于“研究了很多内容”，而在于把这些来源里的东西真正整理成策略，能继续拿去验证，而不是停留在零散想法、收藏夹或聊天灵感里。

### 支线二：`backtest`

这条支线解决的是验证数据不够的问题。当真实交易记录还不够时，系统需要把主线复盘生成的策略，或 `r&d agent` 生成的策略，放到历史数据里做模拟验证，而不是因为缺少实盘样本就让验证阶段空转。

`backtest` 的意义，不是伪装成真实交易，而是给验证阶段提供早期能用的数据。它让系统在冷启动时就能先分清楚：一个策略只是听起来有道理，还是在历史条件下至少能初步跑通。

验证阶段用到的数据，会随着使用过程慢慢变化：

- 冷启动阶段，主要依赖历史数据回测
- 积累一部分真实交易后，可以同时参考最近几笔实盘交易的回看
- 形成更多正式链路后，验证会越来越多地依赖用户自己的历史交易记录

所以更准确地说：

- 策略主要有两个来源：主线复盘生成的策略，和 `r&d agent` 生成的策略
- 验证主要也有两个来源：用户自己的历史交易记录，和 `backtest` 使用的历史测试数据

迭代不是单独多出的一步，而是发生在“策略生成”之后、“验证完成”之后。也就是说，策略先被生成，再被放到不同来源的测试集合里验证，验证通过后，才进入后续迭代。

主线解决的是“真实交易怎么连续跟下去”；`r&d agent` 解决的是“没有当前持仓时，怎么继续产出新策略”；`backtest` 解决的是“实盘样本不够时，怎么先把验证跑起来”。三者可以独立存在，但最后都汇入同一个策略生成、策略验证、策略迭代过程，而不是各做各的。
