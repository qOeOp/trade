# Product Vision

## 一句话定位

这不是一个给我更多观点的 agent，而是一个把我临时、断裂、不可回看的交易判断，变成一条可延续、可回看、可修正的判断链的加密交易工作台。

更具体地说，它当前不是一个面向泛用户出售的 SaaS 产品，而是一组运行在 agent 工作区里的高级 skill 集合，给开发者自己长期使用。

## 我要解决的真实问题

这个产品当前的目标用户只有一个：我自己。

我在这里既是用户，也是开发者。它不是先为一类抽象用户画像设计，再拿我去套；而是先从“我作为一个持续管理仓位的主观加密交易者，到底缺什么”出发，再决定这个系统应该长成什么样。

这里说的长期积累，也只服务于同一个人随时间延续的判断过程。`决策资产` 和 `方法层` 不是为了多人协作、组织沉淀或平台化复用而设计，而是为了让我自己在下一次真实决策里不必重新解释自己。

我不缺信息，也不缺观点。

我真正缺的是：当真实资金决策发生时，我没有一条连续、可回看、可修正的判断链。

最常见的断裂时刻是：

- 我手上已经有仓位，现在该继续拿、减仓，还是重做计划
- 这个标的我前几天看过，但现在已经想不起来当时为什么那样判断
- 一笔交易结束后，我知道自己做得不稳，却说不清是判断错了、执行乱了，还是只是运气差

今天这些内容散落在 `TradingView + 笔记 + 群消息 + 零散对话` 之间。信息很多，但判断过程是断的。

## 为什么值得做

多数工具只能覆盖其中一小段：

- 给一个当下看起来聪明的回答
- 存一堆零散记录
- 展示市场数据

但真正稀缺的是把判断、更新和复盘接成同一条链。

这个系统存在的意义，不是替我做决定，而是把“看市场 -> 形成判断 -> 后续调整 -> 复盘修正”变成一条能延续的判断链。

这个产品的最小成立条件不是“分析更聪明”，也不是“记得更多”，而是：一次真实高意图决策问题进来后，系统能把它承接成一条之后还找得回、接得上、复得了盘的判断链。

所以当前最重要的产品定义，不是单次回答质量，而是承接率和延续率：在一个可比的 30 天窗口里，高意图决策问题里有多少能被承接成可回看的判断链，以及这些链里有多少没有在创建后立刻断掉。

## 为什么是现在

现在值得做，不是因为模型更会说话了，而是因为已经第一次具备了把这条链真正接起来的条件：

- agent 已经能主动调用工具拿市场数据
- 工作区已经能承接历史上下文，而不只是一次性对话
- 加密市场本身高波动、全天候、跨市场联动强，最需要连续判断，而不是临时灵感

这里真正成立的也不是“更强记忆”，而是“同一工作区里的连续协作”。前者更接近记得我说过什么，后者更接近围绕同一个判断对象持续推进、回看、修正，并且不绑定单一模型；只要工作区里的决策资产被保留下来，不同 agent 都应该能读取、延续和使用它。

## 它最小长什么样

它至少不应该从一个空白对话框开始，而应该运行在一个持久化的 agent 工作区里。用户打开它时，第一眼看到的不是“今天想问什么”，而是当前有哪些活跃判断链、每条链现在看到哪一步、最近发生了什么变化、下一次为什么值得回来。

它和 `TradingView + 备忘录` 的最小差异，不是多一段分析，而是默认承接方式不同。

比如我丢一句“ETH 现在还拿吗”，系统不该像第一次见到我一样从头回答，而应该先检索所有相关旧链，再判断这是旧链延续还是新链开始；如果旧链候选不止一条，它也应该直接反问我是在接哪一条，而不是假装自己必须一次判对。如果是旧链，它先把上次判断、之后变化和当前缺口接起来，再只问最少一个真正影响结论的问题；如果是新链，它也应该立刻形成一条之后还能继续更新和复盘的链，而不是让这次判断再次散掉。

也就是说，这套系统的最小解法不是“自动把新链旧链一次判对”，而是“先检索候选旧链，再用一次低摩擦确认完成挂接”。只要这个承接动作足够顺滑，它就已经比从零重建上下文前进了一大步。

## 核心对象

`判断链` 是这个系统的核心对象。

它不是任意一段市场聊天，也不只是一个带时间轴的笔记，而是围绕同一个判断对象持续维护的判断对象。任何时刻，一条链至少都要能回答四个问题：

- 现在到底在看什么
- 当前判断是什么
- 这个判断主要建立在什么依据上
- 什么变化会触发下一次更新或结束

一次高意图决策问题被系统正式承接时，这条链开始形成；只要这些问题还需要继续维护，它就还活着；当持仓、计划或观察条件已经结束，可以进入结果归因或复盘时，这条链才算结束。

这里的“结束”不是指复盘做完，而是指这条链对应的真实管理需求已经结束。复盘发生在结束之后；它决定这条链有没有被真正消化，但不改变“结束”本身的定义。

## 核心立场

这套系统不是先要求我记录，再开始工作；而是先由系统承接，再把我必须补充的部分压到最少。

它的基本立场只有三条：

- 只要问题已经明显指向真实资金决策，系统就先承接
- 系统先承担大部分上下文重建成本，而不是先把整理负担压回给我
- 只有真正影响判断质量的关键歧义，才值得再把问题抛回给我确认

这也意味着系统不能把“回拉”理解成一种可有可无的提醒，而要把它当成正式链路的一部分。只要一条链仍处于活跃管理状态，系统就应该对“下一次为什么值得回来”有默认承接，而不是把后续更新完全赌在我会自己想起。

## 最难的问题

最难的地方不是回答市场，而是判断这次输入到底是在延续旧链，还是在开启新链。

这不是一个已经解决的问题，而是这套系统最核心的未解风险。

但我对它的押注不是“系统一开始就能完美判对所有链路归属”，而是：只要系统能在大多数高意图场景里先检索出足够小的候选旧链集合，再用一次低摩擦确认完成挂接，这个问题就已经足够可用。在 agent 工作区里，这首先是一个“检索 + 交互确认”的问题，而不是一个必须脱离交互一次判对的分类问题。

如果这件事在真实使用中长期做不好，这个方向也不能硬撑成“全自动延续”。它的最小可接受降级形态，是把自动归属降级为半自动承接：系统仍然负责重建候选上下文，但把链路归属确认更明确地交还给我；如果连这种半自动承接都不能成立，那它就不再是我要做的“判断链工作台”，而只剩下一次性分析工具。

如果这一步做不好，判断链不会沉淀成资产，只会继续断裂成噪音。

## 系统真正积累什么

它真正积累的不是聊天记录，而是两层东西：

- `决策资产`：历史判断、执行结果、复盘结论，以及已经确认过的偏好边界
- `方法层`：会直接影响后续判断方式的默认规则，比如优先级、风险边界、常见错误和复盘框架

`决策资产` 默认保留和承接，优先保证链路不丢；`方法层` 默认保守升级，优先避免把偶然经验过早写成长期规则。

两者的分界不能是黑盒。凡是还带有明显情境性、仍然可能只属于某一条链的经验，默认继续留在 `决策资产`；只有那些足够稳定、值得影响后续默认行为的东西，才应该进入 `方法层`。在 vision 层面，我要锁住的不是具体升级算法，而是升级权必须慎重、可见、可回退。

## 成功长什么样

这套系统最终要帮助我在长周期里更稳定地赚钱。

更具体地说：

- 我是否比过去更少重复犯同一种判断错误
- 我是否能更稳定地区分判断问题、执行问题和情绪问题
- 我的结果是否更稳，而不是只靠几次运气好的收益
- 在可比周期里，使用这套系统的自己，是否优于没有这套系统时的自己

在当前阶段，至少要有一个可观测的 proxy 来判断方向有没有成立：在一个可比的 30 天窗口里，我发起的高意图决策问题中，有多少最终被承接成了可回看的判断链，并且之后至少发生过一次有效更新、执行记录或复盘回看。更关键的是，这种承接是否真的减少了我从零重建上次判断的成本；如果链虽然建起来了，但我仍然经常要重新解释自己，这个方向依然不成立。

如果在一个可比的 30 天周期里，我用了这套系统之后，仍然经常需要从零重建自己上次为什么那样判断，仍然很少真正完成“判断 -> 更新 -> 复盘”的闭环，那么这个 vision 就没有成立。

## 当前边界

当前主战场是加密交易协作。

这里说的“工作台”至少必须是一个运行在 agent 工作区里的持久化 skill 集合：它要能持续承接上下文、可查看活跃判断链、可回看历史链路与方法层；单次问完就散的对话，不足以承载这套产品的核心价值。

其他市场信息当前只进入两种层级：

- 作为加密判断的背景输入
- 在我明确要求时，作为单次分析对象

默认不进入完整判断链、方法层和持续复盘闭环。
