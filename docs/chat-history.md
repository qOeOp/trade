### Structured Notes

#### Session Goal Evolution

- 起点目标：分析 BTC 日线大阳后新一天盘面，要求用 skill 拉 K 线和指标，而非口头猜测。
- 目标扩展：从"看行情"转成"找交易位置"，映射 BTC `70256` 到 ETH `2156-2148` 承接区。

- 止盈迭代：多单从建立到分批成交（`LONG 0.18 → 0.047 → 0`），每次同步检查剩余止损是否匹配，并发现并修正止盈超覆盖问题。

#### Data And Tools Used

- `ohlcv-fetch`：持续拉取 `BTC/USDT:USDT` 与 `ETH/USDT:USDT` 的 `1d/4h/1h/15m` 及超短线 `5m/1m`。
- `tech-indicators`：输出各周期 bias、支撑、压力、趋势线、失效位，关键结论多次依赖多周期组合信号。
- `binance-account`：读取账户余额、持仓、普通挂单和保护单，把建议从通用观点升级成账户级方案。
- `web`：补充跨资产分析（美伊战争消息面、原油黄金表现）。
- Shell 级 CSV 检查：直接 `tail` 最新 K 线，不只依赖摘要；本地补算 `MACD(12,26,9)`、`EMA20/50/200`、`KDJ(9,3,3)` 翻译成交易者语言。
- 截图回退链路：API 超时时，用用户截图继续验单和风控判断。

#### Decision Log

- BTC 日线大阳后的第一结论是"高位整理"，而非继续加速，影响后续所有策略偏向等确认。
- BTC `70256` 是中等胜率支撑位（盲接 40-50%，确认后 55-60%），为 ETH 位置框架提供锚点。
- ETH `2156-2148` 叠加 `4h + 1h` 双支撑，比 `2202.4` 更适合作为承接带；方案从 `2x` 到 `50x`，再演化成总仓位 `6 ETH` 的睡觉版三张独立单。
- BTC 空单止损从裸露到 100% 覆盖（`0.18 @ 73054 / 0.15 @ 74113 / 0.242 @ 74200`），是整个线程最关键的账户风险状态变化。
- ETH 挂单从三张收敛到两张（撤掉 `2156`，只保留 `2152 / 2148`），基于最新 `1h` 支撑更靠近 `2148` 的结构更新。
- spot/usdm 口径纠偏：日内高点 `72761`（spot）vs `72743.4`（usdm），纠偏后所有关键位重算；这类错误对真实交易影响极大，产品层应默认在每个关键数字旁标注口径。
- 发现止盈超覆盖（三张 TP 总量超过 `LONG 0.18`），修正为 `0.043 @ 71920 / 0.090 @ 72152 / 0.047 @ 72678`，账户快照确认数量完全对齐。
- `LONG 0.18 → 0.047 → 0`：每次止盈成交后同步缩减保护单，避免留下过量止损。
- 从"高位追多锁仓"转向"SHORT 侧 BUY STOP_MARKET 灾难减仓"：最终保护结构为 `SHORT -0.572`，上方 `73054 / 74113` 两层，下方 `65000` 止盈。
- `72397.70` 双层评价：事中认可（落在 `72360-72410` 首回踩承接区内），事后承认不是最优（市场后来给到更深 `71592`）；首回踩单天然有"接早"风险，这不是逻辑错误。

#### Why Recommendations Changed

- BTC 早期偏"整理" → ETH 接多强调等确认，不盲追。
- 账户快照显示 BTC 空单浮亏+止损不完整 → ETH 多单降级为轻仓对冲，而非进攻仓。
- 50x 方案：只省保证金，不改变方向判断，但容错降低，仓位必须更小。
- 数据切换 spot→usdm：不是观点变了，而是数据口径变了，关键位全面微调。
- LONG 仍存在时允许分批止盈当反弹对冲；LONG 归零后立刻切回"只围绕空单主仓做风险管理"。
- 同一个 `72397.70`：事中认可、事后承认不是最优，产品需要同时保留"当时为什么下这个结论"和"事后复盘怎么看"。

#### Friction And Failure Points

- Python 环境依赖：skill 初次运行被本机版本限制，需通过 conda python3.10 绕过。
- Binance API 超时+SSL EOF：多次导致无法读取账户，迫使工作流从"API 自动核验"退回"截图人工核验"；产品必须设计重试、降级和可视化状态反馈。
- spot/usdm 口径错位：产品层应默认把口径标注在每个关键数字旁边，不能依赖用户来纠正。
- 保护单数量匹配是高频痛点：用户多次改 TP/SL、改单量、平仓，AI 需要像持续在线的助手一样逐步核验，而不是只给一次建议就结束。
- 高速行情导致建议过时：用户多次指出"你说的那个位置已经过掉了"，产品必须显式标记建议生成时间。

#### Product Insight

- 行情分析 + 账户状态 + 执行校验的闭环很有价值：同一个 `72000`，对空仓、净多仓、对冲仓的建议完全不同，单独的技术分析不够。
- 策略建议必须支持从"观点"到"交易参数"的过渡：杠杆、仓位、分批进场、统一止损、分层止盈、睡觉版挂单。
- 市场口径必须一等公民化：spot/usdm/coinm 不是实现细节，会直接改变用户对 AI 是否可信的判断。
- API 不稳定时截图回退链路非常重要：用户发一张挂单截图，AI 就能继续工作，这对真实产品是强需求。
- 高频场景下，系统应自动判断"上一条建议是否已经过时"，不能让用户来指出"这个位置已经没了"。
- 用户非常在意建议的可追责性：不只想知道现在该怎么做，还要回头问"你当时那个点位到底算不算认可"——需要产品沉淀决策上下文而非只留结论。
- 账户验单不是一次性的：用户会反复改单、平仓、留尾仓，AI 需要持续跟踪结构变化。

#### Candidate PRD Themes

- 主题 1：面向真实账户的行情分析助手。输入是"我账户里有什么仓位和挂单"，输出是结合当前风险敞口的建议。
- 主题 2：策略生成到执行校验闭环。AI 不只负责建议，还负责核对用户是否真的按建议挂对了单。
- 主题 3：截图 + API 双通道核验。任何一个通道失败，另一个通道都能兜底。
- 主题 4：建议有效期与口径管理。关键价位附带时间窗、市场口径（spot/usdm）和失效状态。
- 主题 5：多空对冲与保护单编排。管理主仓 + 保护单 + 临时对冲单的关系，重点是结构卫生而非方向判断。
- 主题 6：建议复盘机制。允许追问"当时为什么这样建议、事后是否仍成立"，自动生成决策链式历史。

#### Candidate User-Story Angles

- 作为一个持有真实仓位的用户，我希望 AI 在分析行情时自动结合我的账户风险，而不是只给通用观点。
- 作为一个用 Binance 合约交易的用户，我希望所有关键价位都明确标注口径，避免 spot/usdm 混淆。
- 作为一个会睡前挂单的用户，我希望 AI 给我的方案能直接落地成 Binance 可执行参数，并能帮我检查是否挂对。
- 作为一个会被 API 抖动影响的用户，我希望即使交易所接口失败，也能通过截图继续完成验单和风控判断。
- 作为一个会在快节奏行情里频繁改挂单的用户，我希望 AI 能自动检查"仓位和保护单是否仍匹配"，而不是让我手工核对。
- 作为一个会事后复盘的人，我希望看到"为什么当时能挂第一脚、为什么后来又说不是最优"的完整原因链。

#### Replay Value

- 关键节点（按时序）：第一次用 skill 拉行情并给结论、第一次把建议和账户绑定（BTC 空单裸露风险）、ETH 三张单收敛到两张、spot/usdm 口径纠偏、发现止盈超覆盖并修正、API 超时退化成截图验单、`LONG 0.18 → 0.047 → 0` 逐步变化、从"高位追多锁仓"切到"SHORT 侧灾难保护"、`72397.70` 事中/事后双层评价。

#### Product Insight
