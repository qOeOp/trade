# Chat History

## 2026-04-08 ~ 2026-04-09 BTC / ETH 行情、账户风险、挂单调整

### Structured Notes

#### Session Goal Evolution

- 起点目标：用户先想知道 BTC 在“日线大阳后的新一天”盘面到底是什么状态，要求用 skill 拉 K 线和指标，而不是口头猜测。
- 中段目标：问题很快从“看行情”转成“找交易位置”，先讨论 BTC `70256`，再映射到 ETH 的对应支撑位。
- 决策升级：在找到 ETH `2156-2148` 承接区后，用户不再只关心方向，而是开始要求可执行的仓位、杠杆、进场、止损、止盈方案。
- 风险升级：用户随后把问题切换到“结合真实 Binance 持仓”，目标变成既要接 ETH 短多，又要处理 BTC 空单带来的账户级风险。
- 产品意图显化：最后用户开始关注“如何把这些对话沉淀成可复用资产”，希望后续能用于 PRD、user-story、复盘和产品设计。

#### Data And Tools Used

- `ohlcv-fetch`：负责从 Binance USDT 永续拉 `1d/4h/1h/15m` K 线。这个 skill 是整个判断链路的基础，用户明确要求“记得用 skill 查看 K 线和指标”。
- `tech-indicators`：负责从本地 OHLCV 输出趋势、动量、支撑、压力、趋势线和失效位。AI 的结论多次依赖 `4h/1h/15m` 的组合信号，而不是单一价位。
- `binance-account`：负责只读拉取账户余额、持仓、普通挂单和保护单。它让建议从“泛行情建议”升级成“结合当前风险敞口的账户级方案”。
- `web`：用于补充跨资产和消息面，尤其是在用户要求结合“美伊战争消息面、原油黄金表现”分析 ETH 后续走势时使用。

#### Decision Log

- BTC 日线大阳后的第一结论不是“继续起飞”，而是“高位整理”。这影响了后续所有策略都偏向等确认，而不是盲追。
- BTC `70256` 被识别为一个可交易支撑，但不是高胜率盲接位。这为 ETH 映射位置提供了框架：寻找同样是“能做，但更适合确认”的位置。
- ETH `2202.4` 被识别为最像 BTC `70256` 的 `4h` 支撑，但进一步比较后，发现真正更舒服的承接带在 `2156-2148`，因为这里叠加了 `4h + 1h` 支撑。
- ETH 方案从轻仓、低杠杆，一路演化到 `50x`、再到总仓位 `6 ETH` 的“睡觉版”挂单。这反映了用户对“少占保证金、但保留操作感”的真实偏好。
- 用户挂好单后，AI 通过截图和 API 双重核验方案是否落地正确。这个阶段不再是“给方案”，而是“验执行质量”。
- BTC 空单风险一开始是裸露的，后来通过补齐三段止损，变成 100% 有止损覆盖。这里是整个线程里最关键的一次账户风险状态变化。
- ETH 挂单后来从三张变成两张，撤掉 `2156` 后，只保留 `2152 / 2148`。这是一次基于最新盘面重新收紧方案的迭代，不是简单复读原建议。

#### Why Recommendations Changed

- 早期 BTC 结论偏“整理”，所以接多方案都强调“等确认”，而不是提前挂高。
- 当账户快照显示 BTC 空单浮亏较大且止损覆盖不完整时，ETH 多单从“进攻仓”被降级为“轻仓尝试 / 对冲型仓位”。
- 当用户切换到 `50x` 方案时，AI 没有改变方向判断，但明确下调了可接受仓位，因为杠杆提高改变的是容错，不是胜率来源。
- 当用户提供截图表明单子已挂好时，AI 的职责从“策略设计”切到“验单与纠偏”，并且开始针对 Binance 实际下单界面做核验。
- 当 ETH 最新盘面显示 `4h` 仍 bullish、`1h` 支撑更靠近 `2148`，而 `15m` 处于偏弱回踩阶段时，AI 建议撤掉 `2156`，保留更深的 `2152 / 2148` 两张。

#### Friction And Failure Points

- 技术阻塞 1：`ohlcv-fetch` 和 `tech-indicators` 一开始被本机 Python 版本限制卡住，必须通过本地 `conda` 的 `python3.10` 环境绕过去。这个过程说明技能可用性和本地环境依赖会直接影响用户体验。
- 技术阻塞 2：Binance API 多次出现超时和 SSL EOF，导致无法稳定读取实时账户信息。这个问题迫使工作流从“API 自动核验”退回“让用户发截图人工核验”。
- 技术阻塞 3：即使在同一线程里，行情数据和账户数据的可用性也不是持续稳定的，说明产品如果要依赖交易所 API，必须设计重试、降级和可视化状态反馈。
- 决策阻塞：用户有时关心的是“方向判断”，有时其实更在意“睡觉前会不会出事”。这说明系统需要识别用户的真实目标状态，而不是只回复表面问题。

#### Product Insight

- 价值点 1：`行情分析 + 账户状态 + 执行校验` 的闭环很有价值。单独的技术分析并不够，用户真正想要的是“这对我现在这个账户意味着什么”。
- 价值点 2：策略建议必须支持从“观点”过渡到“交易参数”，包括杠杆、仓位、分批进场、统一止损、分层止盈和睡觉版挂单。
- 价值点 3：API 不稳定时，截图回退链路非常重要。用户发一张挂单截图，AI 就能继续工作，这对真实产品是一个强需求。
- 价值点 4：用户并不只想复盘“市场怎么走”，还想复盘“自己是怎么一步步做出决策的”。这说明聊天历史不只是日志，更是后续 PRD / user-story 的需求源。
- 价值点 5：历史记录不能只保留结论。必须保留“为什么改方案”“是什么信号触发了调整”“工具在哪一步失败”这些过程信息，否则后续很难反推出真正的产品需求。

#### Candidate PRD Themes

- 主题 1：面向真实账户的行情分析助手。输入不是抽象交易对，而是“我现在账户里有什么仓位和挂单”。
- 主题 2：策略生成到执行校验闭环。AI 不只负责建议，还负责核对用户是否真的按建议挂对了单。
- 主题 3：截图 + API 双通道核验。任何一个通道失败，另一个通道都能兜底。
- 主题 4：对话沉淀为结构化历史。聊天本身应能持续沉淀成后续产品文档素材，而不是事后再人工整理。

#### Candidate User-Story Angles

- 作为一个持有真实仓位的用户，我希望 AI 在分析行情时自动结合我的账户风险，而不是只给通用观点。
- 作为一个会睡前挂单的用户，我希望 AI 给我的方案能直接落地成 Binance 可执行参数，并能帮我检查是否挂对。
- 作为一个会被 API 抖动影响的用户，我希望即使交易所接口失败，也能通过截图继续完成验单和风控判断。
- 作为一个要做产品的人，我希望把高价值对话自动沉淀成结构化历史，用于后续写 PRD、user-story 和复盘。

#### Replay Value

- 这段对话可复盘的不是单一结论，而是“用户目标如何变化”和“AI 如何从市场分析一路下沉到账户决策”。
- 真正有复盘价值的节点包括：第一次用 skill 拉行情、第一次把建议和账户绑定、第一次发现 BTC 裸露风险、第一次通过截图代替 API 验单、ETH 从三张单收敛到两张单。
- 如果后续继续增量追加，建议每次都标记：`Goal Shift`、`Decision Change`、`Risk State Change`、`Tool Failure / Fallback`。

## 2026-04-08 ~ 2026-04-09 BTC / ETH 实时账户联动、挂单迭代、超短线复盘（续）

### Structured Notes

#### Session Goal Evolution

- 起点目标：用户从更直接的交易问题切入，不再只问“怎么看盘”，而是问“BTC 现在上破 `72000` 的概率大不大”“如果现在接多，挂单该怎么挂”。

- 变化 1：从 `spot` 改到 `usdm` 后，关键高点、阻力和入场位全部发生微调。AI 不是观点变了，而是数据口径变了。

- 变化 3：当盘面从 `72858.5` 冲高后快速回落到 `72000` 附近时，超短线判断从“行情已经启动”切成“1h 中性、4h 偏强、但更适合观察而不是加新单”。也就是说，AI 会随着结构从加速态回到消化态而降级建议强度。

#### Friction And Failure Points

- 账户读取偶发超时：在检查保护单时，摘要层一度出现 `openOrders handshake timed out`，但 JSON 快照仍成功返回完整订单列表。AI 因此采用“以 JSON 为准、摘要仅作辅助”的判断方式。
- 高速行情导致建议过时：用户多次直接指出“你说的那个位置已经过掉了”。这不是单次判断失败，而是说明在快节奏盘面里，产品必须显式标记建议生成时间，并提醒哪些价位已经失效。
- 口径错位风险：用户纠正 “日内高点到底是 `72743.4` 还是 `72761`” 后，AI 才发现之前混用了 `spot` 和 `usdm`。这类错误对真实交易影响很大，说明产品层应该默认把口径写在每个关键数字旁边。
- 保护单数量匹配是高频痛点：用户挂好多单 TP/SL 后，AI 多次需要检查“单量是否和剩余仓位一致”“多单平掉后是否还有遗留止损”。这不是边角问题，而是实际使用中的核心风控环节。
- `tech-indicators` 输出冗长且伴随 pandas warning。虽然不阻塞分析，但会污染终端阅读体验，提示后续产品层可能需要更稳定的“面向人”的摘要层。

#### Product Insight

- 价值点 1：用户真正需要的不是“看盘观点”，而是“账户当前结构下，这个观点意味着什么”。同一个 `72000`，对空仓、净多仓、对冲仓的建议完全不同。
- 价值点 2：市场口径必须一等公民化。`spot/usdm/coinm` 不是实现细节，而是会直接改变用户对 AI 是否可信的判断。
- 价值点 3：高频场景下，系统应该自动判断“上一条建议是否已经过时”，而不是让用户来指出“这个位置已经没了”。
- 价值点 4：用户非常在意“建议的可追责性”。他们不仅要知道现在该怎么做，还要回头问“你当时那个点位到底算不算认可”。这要求产品沉淀决策上下文，而不只是结论。
- 价值点 5：账户验单不是一次性的。用户会反复改 TP/SL、改单量、平仓、留尾仓，AI 需要像一个持续在线的交易助手，而不是一次性给完方案就结束。

#### Candidate PRD Themes

- 主题 1：按账户状态解释行情。输入是 `positions + openOrders + latest market data`，输出是“当前结构下的建议”，而不是脱离账户的通用观点。
- 主题 2：建议有效期管理。任何关键价位都应附带时间窗、市场口径和“已失效/仍有效”状态。
- 主题 3：多空对冲与保护单编排。重点不是帮用户下方向判断，而是管理 `主仓 + 保护单 + 临时对冲单` 的关系。
- 主题 4：建议复盘机制。允许用户追问“当时为什么这样建议，事后是否仍成立”，并自动生成决策链式历史。

#### Candidate User-Story Angles

- 作为一个同时持有 `LONG` 和 `SHORT` 的用户，我希望 AI 不只给我方向，还能告诉我这两边仓位在当前盘面里分别该怎么处理。
- 作为一个用 Binance 合约交易的用户，我希望所有关键价位都明确标注口径，避免 `spot/usdm` 混淆。
- 作为一个会在快节奏行情里频繁改挂单的用户，我希望 AI 能自动检查“仓位和保护单是否仍匹配”，而不是让我手工核对。
- 作为一个会事后复盘的人，我希望看到“为什么当时能挂第一脚、为什么后来又说不是最优”的完整原因链。

#### Replay Value

- 这段续篇最有复盘价值的，不是某个单点位本身，而是“同一套建议如何随着账户状态、市场口径和时间推移而被不断重估”。
- 值得复盘的节点包括：`spot/usdm` 口径纠偏、第一次发现止盈超覆盖、`LONG 0.18 -> 0.047 -> 0` 的逐步变化、从“开多对冲”切到“只留空单灾难保护”、以及 `72397.70` 的事中/事后双重评价。
- 如果后续继续追加，建议保持这几个标记维度：`Market Context Shift`、`Account State Change`、`Order Coverage Check`、`Advice Expired Or Still Valid`、`Post-Mortem`。

#### Friction And Failure Points

- 最明显的一次失败点是自动化创建流程：用户先要“每 `5` 分钟分析一次”的 Codex 自动化，AI 起初只给了 prompt，没有真正创建；用户明确指出“你没帮我创建自动化”，之后才补做 `automation_update` 创建。

#### Product Insight

- 用户真正需要的不是“技术分析摘要”，而是“和我的持仓结构绑定的执行建议”。同一套 `RAVEUSDT` 结构，对空仓、锁仓、净空、净多的建议完全不同。

- 用户强烈偏好简洁、可执行、带推荐度的答案，而不是长篇分析；但同时又会追问“为什么这样判断”。说明产品层需要把“执行版”和“推理骨架版”分层输出，而不是二选一。
- Heartbeat 自动化很适合这类交易场景，因为用户并不是想看每次都很长的研究报告，而是希望在新的 `5m/15m` 触发出现时被一句高价值结论唤醒。
- “主力操盘手法学习”其实是对 `15m` 微观结构模式识别的需求。产品可以不声称懂主力，但可以沉淀“常见急拉急砸模板、真假突破模板、插针后收盘确认模板”。

- `symbol-snapshot`：先后两次读取 `BNBUSDT` 合约快照，确认 `last_price` 从约 `612.7` 到 `611.7`，并补 `24h` 高低点、`open interest`、`funding` 语境。
- `ohlcv-fetch`：抓取 `BNBUSDT` 的 `1w/1d/4h/1h`，后续又重拉 `1d/4h/1h`，避免只凭第一次快照做延伸建议。
- `tech-indicators`：产出多周期 `trend`、`support`、`resistance`、`bullish/bearish invalidation`，核心信号包括 `1d bearish`、`4h neutral`、`1h bullish-leaning`、总偏向 `slightly-bearish`。
- `binance-account`：读取 `BNBUSDT` 账户快照，确认没有现货和合约持仓，只有一张 `BUY LIMIT 16.36 @ 611.2` 的 `LONG` 向 `OTOCO` 母单。
- Shell 级数据透视：直接查看最新 `1d/4h/1h` CSV 尾部，确认 `625.74` 冲高回落、随后在 `611-620` 区间震荡的真实 K 线轨迹。
- `web.search_query/open`：补看 `BNB Chain` 周报与 `Binance Research` 月报，用于给出“生态不差，但价格仍受更大 risk-off 约束”的背景判断，而不是只看图形。

#### Decision Log

- 第一版技术结论不是“旧多单仍然强有效”，而是多周期偏弱：周线中性、日线偏空、`4h` 卡阻力、`1h` 只是短反。AI 因此把结论落在“不是完全失效，但已进入防守阶段”。
- 在第一版回答里，AI 用条件句区分“如果是多单 / 如果是空单”，核心理由是账户状态尚未恢复，只能先从盘面给方向性框架：多头防守位看 `607` 与 `604.2`，上方要先收复 `614.5` 再看 `620-625.4`。
- 用户质疑“你看不到我的持仓吗”后，AI 立即切换到 `position-monitor` 的正确顺序，先用 `binance-account` 查账户，再回到交易建议。
- 账户快照把关键事实纠正得很明确：用户并没有 `BNB` 持仓，只有一张尚未成交的 `BUY LIMIT 16.36 @ 611.2`，并且它是 `OTOCO` 母单，不是裸限价单。
- AI 因此把判断从“仓位是否继续拿”改写成“这张挂单是否还值得保留”。结论变成：这不是顺风追涨单，而是短线支撑区的防守型低吸单，逻辑还成立，但只能算中等质量。
- 在无法读到附带 TP/SL 的前提下，AI 先给出“只看入场位”的评分：`611.2` 这张单约 `6.5/10`；若止损放在 `604` 下方、第一止盈放 `624-626`，可上调到 `6.5-7/10`；若止损过宽或根本不清楚，则降到约 `5/10`。
- 当用户进一步说明挂单已经撤掉未成交，AI 不再围绕旧单打分，而是重拉最新 `BNB` 数据后重新计算更优挂单区。新的判断是：`611` 附近不再算便宜，真正更值得挂的主支撑在 `605.5-606.5`，更深次支撑在 `596.5-599.5`。

#### Why Recommendations Changed

- 变化 1：最初只看盘面时，AI 无法确定用户到底是已持仓还是未成交挂单，所以只能给条件化结论；一旦账户状态明确，建议就从“仓位还能不能拿”切换成“挂单本身值不值”。
- 变化 2：读取到账户里没有 `BNB` 持仓后，原本关于“多单已进入防守期”的表述被重构成“低吸逻辑还在，但你还没真正承担持仓风险”，风险性质从已暴露仓位变成候选入场单。
- 变化 3：挂单评分之所以先给入场分、后再给完整方案分，是因为 Binance 公开 API 读不到这张 `OTOCO` 母单附带的 TP/SL 明细；在缺少执行侧约束时，AI 只能先评价 entry，而不能完整评价整套交易。
- 变化 4：用户撤单后，AI 再次刷新到最新 `4h/1h` 结构，发现 `BNB` 仍处在 `611-614` 的拥挤区，继续挂 `611.2` 已不具优势，因此把推荐迁移到更深的 `606` 与 `598` 支撑带。

#### Friction And Failure Points

- 本轮没有出现明显的行情 API 超时，也没有需要截图回退的场景；主要阻塞来自分析顺序的第一次偏差，以及 Binance 公共 API 的信息盲区。
- 最关键的降级点是：`binance-account` 能识别这是一张 `OTOCO` 母单，也能确认“存在附带 TP/SL 策略”，但无法通过公开 API 读取具体 TP/SL 价格。系统因此必须把判断降级为“只看入场位/请用户手动补充 TP/SL”。
- 这轮也暴露了另一个交互摩擦：用户说“之前你让我开单的 bnb”时，语义上既可能是已成交仓位，也可能是挂单。若系统不先恢复账户暴露，就容易误把未成交计划当成已持仓事实。

#### Product Insight

- 同一条“交易建议”在产品里应该明确区分三种状态：`已持仓`、`挂单未成交`、`只是候选计划`。这三者的风险和后续问答路径完全不同。
- 对 Binance 这类存在 `OTO/OTOCO` 母单的场景，产品需要显式展示“母单可见 / 子 TP-SL 不可见”的能力边界，否则用户会自然预期系统应该读到完整保护单细节。
- 技术分析输出不应只给方向判断，还应支持“入场位评分”与“整套交易评分”分离。当前这轮对话里，正是因为看不到 TP/SL，AI 才必须把评分拆成 entry-only 与 full-setup 两层。
- 当用户问“之前那单还有效吗”时，产品应默认先查账户，再决定是复盘旧仓位、旧挂单还是旧观点，避免人类和系统对“这单”指代对象不一致。

#### Candidate PRD Themes

- 主题 1：交易对象状态识别。系统自动判断用户说的“这单”是持仓、挂单还是历史计划，并走不同分析路径。
- 主题 2：母单与附带保护单可见性提示。对 `OTO/OTOCO` 结构给出清晰的读取能力边界和降级说明。
- 主题 3：挂单质量评分。把 `entry quality`、`stop placement quality`、`tp alignment quality` 拆开评分，而不是只给一个总分。
- 主题 4：旧建议复活机制。用户再次提起旧推荐时，系统自动刷新市场和账户上下文，再判定旧建议是否仍成立。

#### Candidate User-Story Angles

- 作为一个会反复问“这单还有效吗”的交易者，我希望系统先自动确认我现在是已持仓、已挂单还是已经撤单，再给建议。
- 作为一个使用 Binance App 的用户，我希望 AI 能明确告诉我它是否真的看到了附带的 TP/SL，而不是让我误以为保护单已经被完整纳入判断。
- 作为一个低吸型交易者，我希望系统不仅告诉我“能不能挂”，还告诉我“这个挂单位置值几分，为什么不是更高分”。
- 作为一个会撤单重挂的用户，我希望系统在我说“单撤了”后自动重算更优价位，而不是继续沿用旧 entry。

#### Replay Value

- 这轮最值得复盘的不是单纯 `BNB` 的涨跌判断，而是建议如何随着“账户事实”被补齐而被重写：先是盘面条件句，再到真实挂单识别，再到 entry-only 评分，最后到新的挂单位。
- 关键复盘节点包括：第一次多周期判断给出 `607 / 604.2 / 614.5 / 625.4` 结构、账户查询发现“无持仓仅有 `BUY LIMIT 611.2`”、识别 `OTOCO` 但读不到附带 TP/SL、把旧单评为 `6.5/10`、撤单后把新主挂单下移到 `606.2`。
- 如果后续继续追加，建议保留这些标签：`State Misread Corrected`、`Pending Order Not Position`、`OTOCO Visibility Gap`、`Entry-Only Scoring`、`Repriced Limit Plan`。

### Structured Notes

#### Session Goal Evolution

- 起点目标：用户先否定 `15m` 级别交易，明确要找 `BTC` 的“周内单”机会，核心诉求从短线噪音切到 `4h / 1d` 级别的可持有交易。

- `symbol-snapshot`：先读取 `BTCUSDT` 即时盘面、资金费率和持仓量，确认现价大致在 `74000` 一带、资金费率接近中性略负，不是明显拥挤多头。

#### Decision Log

- 高周期判断的核心结论是：`1d bearish-leaning`、`4h bullish`、现价约 `74000` 处于中间位置，因此不适合现价追。`4h` 关键支撑带落在 `70380-71227`，近端压力在 `75153-76845`。

- `A` 计划的执行骨架被进一步参数化为：挂多区 `70520-71180`，失效用 `4h 收盘跌破 70380` 定义，而不是用分钟级插针；止盈分层看 `75150 / 75980 / 77480`。

#### Why Recommendations Changed

- 变化 1：一开始只是“找 BTC 周内机会”，所以结论停留在结构判断和方案排序；当用户要求“计划细节和推荐分对应起来”后，输出从方向结论升级成带评分的执行剧本。
- 变化 2：当用户把范围缩到 `A 计划` 并要求 Binance 格式时，AI 不再泛讲支撑逻辑，而是把方案压缩成具体挂单区、分批数量、统一止损和分层止盈。
- 变化 3：当用户要求 AI 直接代下单时，建议层必须切换到执行安全层。AI 先核验“有没有下单能力、账户模式是什么、旧挂单会不会冲突”，而不是直接盲发委托。
- 变化 4：统一止损最初想用 `closePosition=true`，但 Binance 规则要求“先有仓位再能挂整仓保护”，于是方案从整仓保护降级为数量型 `reduceOnly` 保护。这不是交易观点变化，而是执行规则逼出的实现调整。
- 变化 5：在订单清理阶段，动作优先级又从“继续优化计划”切换为“只保留新计划、去除历史噪音订单”，说明账户执行期的重点已经不再是重新看行情，而是保持订单集合整洁且可解释。

#### Friction And Failure Points

- 最大阻塞不是行情判断，而是能力边界识别：`binance-account` 的名字很容易让人误以为“能账户操作”，但其 `SKILL.md` 明确只读。AI 先停下来确认边界，再决定是否绕过 skill 直接走底层 API。
- Binance 条件单规则存在隐藏复杂度：`closePosition=true` 的 `STOP_MARKET` 不能对零仓位预先挂出，导致“睡觉版统一保护”在未成交前不能直接按最直觉方式实现。
- 删除订单时暴露出另一个执行细节：极大的期货 `orderId` 不能默认安全地当作 JavaScript 数字处理，否则会出现“Unknown order sent”。这逼出用 `origClientOrderId` 做撤单兜底。
- 本轮没有出现前几次那种 Binance API 持续超时，但真实执行里仍有“仓库无现成下单 skill”“交易所规则限制某种保护写法”“数值精度导致撤单失败”这类非网络型阻塞。

#### Product Insight

- “能读账户”与“能改账户”必须被产品显式区分。只靠 skill 名称或模糊描述，用户和 AI 都容易误判能力边界。
- 交易执行产品不能只输出一个理想计划，还要知道交易所允许的表达方式。例如“统一整仓止损”在理论上成立，但在实际 API 规则下可能必须拆成数量型条件单。
- 订单集合清理本身是独立价值点。用户在意的不只是“新单挂上了没有”，还在意“旧单有没有残留、会不会和新计划冲突、账户现在是否只剩当前意图”。
- 真实执行里，`clientOrderId` 非常重要。它不仅帮助 AI 区分“哪些单是我刚挂的”，也能在大整数 `orderId` 不可靠时充当稳定的撤单和追踪锚点。

#### Candidate PRD Themes

#### Candidate User-Story Angles

### Structured Notes

#### Session Goal Evolution

- 起点目标：用户先问 `BTC 4小时周期` 现在有没有好的做单机会，需求是“直接给交易判断”，不是写策略文章。

- `symbol-snapshot`：多次读取 `BTCUSDT`、`TAOUSDT` 及其它候选标的的最新 `24h ticker`、资金费率和未平仓量，用于确认“当前盘面语境”。

- `tech-indicators`：输出各标的 `bias`、`trend`、`support`、`resistance`、`EMA50 / EMA200`、`bullish/bearish invalidation`，为“是否存在周期共振”“更适合追还是等”“一周内目标空间是否足够”提供结构证据。
- `binance-account`：先读技能说明确认它只读，再实际读取账户状态、`TAOUSDT` 现有挂单、可用余额，用于判断是否能安全自动化下单。

#### Why Recommendations Changed

- 对 `BTC` 的建议变化，不是因为方向被推翻，而是因为持有周期被重写。最初问的是“4h 有没有机会”，所以答案偏向当前位置能不能做；当用户说可以拿几周时，评价标准改成“`4h/1d` 是否共振”；当用户又说最晚下周一前清仓时，标准再次变成“这几天能否走完一段明确波动”。
- 多标的筛选里，`XAU/XAG` 从“偏多还不错”降级成“不是这组里最优先”，不是因为结构突然变差，而是因为用户从看方向改成看“周期共振 + 周内执行空间”，于是 `TAO/SOL` 这类偏空标的的相对优势被放大。
- `TAO` 的入场价从一段区间逐步收敛到 `254.8`，是因为用户不仅要计划，还要问“这个点这周成不成交”。AI 因而把“最优价格”和“高成交率”分开，最终把 `254.8` 作为平衡点，`257` 则保留为更甜但低成交率的备选。
- 执行建议从“可以直接帮你下整套”改成“只能先挂主单”，不是因为计划失效，而是因为真实账户模式和 Binance 条件单 API 语义让保护单无法安全同步挂上。AI 把安全性优先级放在便利性之上，哪怕因此让自动化体验变差。

#### Friction And Failure Points

- 最大阻塞不是行情分析，而是交易执行链条的口径不一致。用户相信自己已经切到别的模式，但 Binance API 连续多次返回 `dualSidePosition: true`，说明 UI 状态、用户认知和 API 真实状态之间存在错位。
- 现有 `binance-account` skill 只读，无法直接承担“查询 -> 判断 -> 执行”的闭环。真正要下单时，AI 必须绕过 skill，直接操作底层依赖与 API，这会抬高风险与复杂度。
- Binance USDⓈ-M 的保护性条件单已迁移到 `Algo Order API`，导致“普通挂单”和“保护单”不再能用同一个入口一致处理；如果产品层不显式抽象这点，用户会以为“下单”是一件事，实际上却是两套接口。
- `Hedge Mode` 下的 `positionSide`、`reduceOnly`、`closePosition`、`Algo` 规则组合非常容易踩坑。即使主单能过，也不代表保护单能跟着过；这类接口级失败不能靠经验口头推断，必须先做测试或具备安全回滚。
- 有一个特别关键的降级节点：AI 在保护单失败后没有让主单留在市场上，而是自动撤销。这个“原子化失败回滚”对真实交易代理非常关键，否则产品会在用户以为“整套已挂好”时留下裸单风险。

#### Product Insight

- 交易代理产品必须把“市场分析能力”和“交易执行能力”明确拆层。前者可以是只读 skill，后者需要具备专门的交易接口抽象、模式探测、预检、测试单与回滚，不应临时拼装。
- “执行前预检”本身值得产品化：账户模式、精度、步进、可用余额、已有挂单、条件单入口、`positionSide` 语义，都应在真正发单前自动过一遍，并给出人类可读的阻塞原因。
- 条件单接口迁移说明交易所 API 在持续演化，产品不能把“止损就是普通单的一种”写死。保护单需要一层独立模型来承接 `Algo Order API`、`closePosition`、触发价与工作价类型。
- 当用户说“我已经改好了”，产品不能只信用户输入，而应优先相信真实 API 状态；同时要把“不一致”明确显示给用户，而不是只在底层报错。
- 在策略层，`257` 这类问题显示出用户对“点位是否足够甜”和“本周是否容易成交”有明确区分。产品层适合把挂单方案天然拆成 `高成交 / 平衡 / 高盈亏比` 三档，而不是只给一个点。

#### Candidate PRD Themes

- 主题 1：读写分离的交易代理架构。只读账户/行情 skill 与真实下单 skill 分离，后者具备专门的预检、测试、风控与回滚。
- 主题 2：交易执行预检面板。自动展示 `position mode`、`margin mode`、精度、步进、`min notional`、当前挂单与可用余额，并在失败时给出可执行修复建议。
- 主题 3：保护单抽象层。统一建模普通限价单、条件止损单、止盈单、`Algo` 单与 `closePosition` 语义，避免每次都让 AI 自己拼交易所细节。
- 主题 4：挂单方案分层。针对同一标的，自动输出 `激进 / 平衡 / 保守` 三档挂单，并把“成交概率”和“盈亏比”的权衡显式呈现。

#### Candidate User-Story Angles

- 作为一个会让 AI 直接代我下单的用户，我希望系统在真实发单前自动检查账户模式与接口兼容性，而不是事后才发现保护单挂不上。
- 作为一个做短波段的用户，我希望看到的不只是“做多还是做空”，而是“这周内更值得盯哪几个、为什么、到哪里挂更容易成交”。
- 作为一个会在 Binance App 里手动改模式的用户，我希望产品能明确告诉我“UI 已改但 API 还没生效”，避免我误以为可以安全自动挂整套。
- 作为一个对价格和成交率都敏感的用户，我希望系统把“最优价”“平衡价”“高成交价”分开，而不是只给单一价位。
