# 用户故事

## 1. 使用方式

- 只记录真实使用过程，用来校验流程；不在这里定义 UI、字段、表结构和长期制度。
- 如果某个流程节点找不到对应故事，或某个故事走不通、无法恢复，先补故事或改流程，再谈章节设计。
- 当前目标是覆盖高频和高风险场景，不追求一次写尽所有边角情况。

## 2. 开仓前

### US-01 明确标的问机会

- 触发：用户带着明确标的进入，例如 BTC、ETH、TRX。
- 核心问题：现在能不能做；偏多、偏空还是不做。
- 流程要求：开仓前起点不只来自全市场扫描，也可以是“带标的进入观察”；`暂不操作` 必须是正式输出。

### US-02 系统提示不建议但用户接受风险后仍决定做单

- 触发：系统给出完整计划，但明确提示盈亏比、胜率、位置或条件不理想。
- 核心问题：在知情前提下，用户仍然要不要继续执行。
- 流程要求：区分“风险提示”和“硬性阻断”；记录系统提醒与用户确认继续执行。

### US-03 系统建议的执行方式与用户最终选择不一致

- 触发：用户认可方向，但不认可系统建议的执行方式。
- 核心问题：系统建议 `MKT`，用户想改 `PLACE`；或系统建议 `PLACE`，用户想改 `MKT`。
- 流程要求：记录“系统原建议”和“用户最终执行方式”的差异，后续复盘要能把方向问题和执行问题拆开。

### US-04 挂单等待过程中再次询问是否要继续等或调整挂单

- 触发：订单未成交、快成交、行情波动放大，或用户主动怀疑原挂单位置。
- 核心问题：继续等、撤单、改价，还是重新做计划。
- 流程要求：`WAIT-UNTIL-FILL` 不是静态状态，而是活跃判断阶段；必须能回到 `OBSERVE / PLAN`。

### US-10 用户不指定币种而是要系统找出当前最值得做的一单

- 触发：用户不先给标的，而是要求系统从全市场找机会。
- 核心问题：现在哪些品种值得做；如果只做一单，优先谁。
- 流程要求：`OBSERVE` 需要支持“先扫描、再收敛到标的”的起点。

### US-13 用户拿到挂单计划后要求系统详细解释思路

- 触发：用户拿到计划，但要追问压力位、支撑位、结构、周期依据。
- 核心问题：为什么是这个方向、这个位置、这个执行方式。
- 流程要求：`OBSERVE` 和 `PLAN` 的输出必须可解释、可追溯，而不是黑箱结论。

### US-19 错过原计划入场点后用户回来问还要不要追

- 触发：用户错过原本入场点，行情已经走了一段。
- 核心问题：现在还能不能追，还是原机会已经结束。
- 流程要求：支持“原计划已错过”这一状态；通常应回到新的 `OBSERVE / PLAN`。

### US-23 多个机会同时存在但用户只能做一个

- 触发：同时存在多个候选机会，但资金、保证金、精力只能支持一单。
- 核心问题：优先做哪一个，放弃哪一个。
- 流程要求：`OBSERVE` 和 `PLAN` 不只服务单一标的，也要支持候选机会排序。

### US-24 原计划结构未破坏但市场环境变了

- 触发：价格结构未破，但波动、成交量、情绪或宏观背景已明显变化。
- 核心问题：原计划还是否有效，还是应该整体作废重来。
- 流程要求：回返 `OBSERVE / PLAN` 的触发源不能只靠价格命中，也要覆盖环境变化。

### US-28 用户想确认现在是否应该空仓观望

- 触发：用户不是来找机会，而是来确认当前是否根本不值得做。
- 核心问题：现在是不是空仓更合理。
- 流程要求：`不做` 必须是正式、积极、可解释的输出，而不是默认空白结果。

### US-29 用户拿着条件触发型计划反复回来确认是否满足并询问能否左侧进场

- 触发：系统之前给的是“等条件满足后再建仓”的计划。
- 核心问题：条件现在满足了吗；能挂单了吗；没满足但概率大时要不要提前左侧进场。
- 流程要求：区分“等待条件成熟”和“订单等待成交”两种等待；允许条件型计划的周期性复查。

### US-30 用户希望系统提前给出快速行情下的应对预案

- 触发：用户担心行情太快，来不及等系统实时回复。
- 核心问题：如果快速拉升、急跌、假突破、瞬间反抽，各自怎么处理。
- 流程要求：`PLAN` 有时必须包含条件分支和应急动作，而不只是单一静态结论。

### US-32 用户准备离线一段时间前来问当前仓位和挂单该怎么布置

- 触发：用户要睡觉、开会、出门或一段时间无法盯盘。
- 核心问题：当前仓位和挂单是否需要提前调整；离线期间最坏和最好情况各自怎么处理。
- 流程要求：系统需要支持“离线前整理”这一类开仓前或持仓中的预案需求，帮助用户在无法实时交互时仍能执行既定处理。

## 3. 持仓中

### US-05 持仓发生较大回撤后用户带着恐慌来问是否继续扛单

- 触发：仓位出现较大回撤，用户心理压力明显上升。
- 核心问题：继续持有、立刻止损、调整止损，还是等反弹后减亏离场。
- 流程要求：`MANAGE` 必须覆盖恐慌下的再判断，而不只是常规优化。

### US-06 持仓回撤放大时用户考虑做反向短单或锁仓

- 触发：原仓位回撤持续放大，用户考虑反向波段或等量锁仓。
- 核心问题：能不能通过反向仓位缩小回撤或冻结风险。
- 流程要求：`MANAGE` 需要覆盖新增反向仓位这种管理动作；后续要能跟踪何时拆解、如何回到单边暴露。

### US-07 用户带着宏观或消息面新闻来问对当前持仓的影响

- 触发：新闻、公告、宏观事件、监管变化打断了原持仓判断。
- 核心问题：这条消息对当前仓位意味着什么；是继续拿、减仓、平仓还是重做计划。
- 流程要求：`OBS-POS` 必须能吸收外部事件，不能只看价格和订单状态。

### US-09 用户阶段性回来看当前持仓整体状态是否需要调整

- 触发：没有单一异常，但用户定期回来查看整体持仓健康度。
- 核心问题：当前总仓位、风险敞口、结构分布是否需要调整。
- 流程要求：`MANAGE` 不只服务单笔仓位，也要允许组合视角的周期性体检。

### US-14 持仓未到止损止盈位但用户因心理压力或落袋意愿提前询问离场

- 触发：原计划仍有效，但用户已经拿不住，或急于先把利润落袋。
- 核心问题：要不要提前离场、先减仓，或改出场节奏。
- 流程要求：`PLAN-POS` 不只响应客观价格，也要响应用户主观风险承受变化。

### US-15 多周期信号冲突下用户拿着仓位来问该怎么处理

- 触发：短周期和大周期方向冲突，用户已经持仓并承受回撤。
- 核心问题：继续扛、减仓、对冲，还是承认当前周期错配先处理。
- 流程要求：`OBS-POS` 需要支持多周期冲突的观察语境，`PLAN-POS` 要能表达短期与长期不一致时的管理意见。

### US-21 部分成交后用户来问剩余挂单怎么处理

- 触发：订单只成交一部分，仓位已经建立，但剩余挂单仍在。
- 核心问题：剩余挂单要不要保留、改价、撤掉，还是结合现有仓位重做计划。
- 流程要求：`partial fill` 之后会同时存在“仓位管理”和“剩余订单管理”；系统需要同时承接两者。

### US-22 已有仓位时用户询问还能不能再开另一笔

- 触发：已经持仓，但又出现新机会，或同一标的出现第二次入场时机。
- 核心问题：还能不能再开；是否会让整体风险敞口过高。
- 流程要求：支持多笔交易链并行，以及新机会与已有仓位共享风险预算。

### US-25 开仓后用户发现仓位大小或杠杆不舒服

- 触发：方向没变，但用户觉得仓位太大、太小、杠杆太高或太低。
- 核心问题：减一点、补一点、降杠杆，还是干脆平掉重来。
- 流程要求：把“方向调整”和“仓位规模调整”区分开；`MANAGE` 要明确覆盖 `ADD / REDUCE / LEV`。

### US-26 盈利仓用户犹豫继续拿还是轮动去别的机会

- 触发：已有盈利仓，但市场上出现了新的更强机会。
- 核心问题：继续持有、部分兑现，还是把资金轮动到新机会。
- 流程要求：允许 `MANAGE` 与新的 `OBSERVE / PLAN` 并行存在；后续要能解释“轮动”导致的离场。

### US-31 用户绕过系统自行下单后希望系统接管建议与后续追踪

- 触发：行情太快，用户先手动下单，再把仓位拿回来交给系统管理。
- 核心问题：这笔单现在该怎么处理；系统能不能补记录、接管建议、继续追踪。
- 流程要求：主线必须允许“外部已存在仓位”直接进入 `MANAGE`，而不是假设所有交易都从系统计划开始。

### US-33 仓位接近爆仓或保证金明显吃紧时用户来问怎么处理

- 触发：仓位接近爆仓、可用保证金不足、维持保证金压力升高。
- 核心问题：加保证金、减仓、降杠杆，还是直接平掉一部分或全部。
- 流程要求：`MANAGE` 需要覆盖生存优先级高于观点判断的场景；风险处理可以先于原策略计划。

### US-34 仓位已经浮盈后用户来问要不要把止损提到保本或开始移动保护

- 触发：仓位已经走出明显浮盈，但尚未到最终止盈。
- 核心问题：要不要把止损提到保本、锁一部分利润，或开始用移动止损保护收益。
- 流程要求：`PLAN-POS` 需要覆盖“从防亏转向护盈”的管理切换，而不只是静态止损止盈。

### US-35 用户觉得原方向逻辑失效并询问要不要直接反手

- 触发：用户认为原方向逻辑被打破，甚至出现了反方向机会。
- 核心问题：是先止损离场后重新观察，还是可以直接反手从多变空、从空变多。
- 流程要求：系统要把“结束旧仓位”和“开启新方向”拆成两个判断，避免把反手当成单一动作一笔带过。

### US-36 用户回来时发现仓位或订单状态已经变化并来问接下来怎么办

- 触发：用户一段时间没看，回来时发现止损、止盈、部分成交、撤单或强平已经发生。
- 核心问题：刚才发生了什么；现在还剩什么仓位和订单；接下来应该怎么处理。
- 流程要求：主线必须支持中断后恢复；系统需要能从状态变化倒推关键事件，并把用户重新接回当前节点。

### US-37 交易所执行异常后用户来问这单现在到底算没算下上

- 触发：下单失败、接口延迟、订单被拒、止损止盈没挂上、状态显示异常。
- 核心问题：这单到底有没有成功执行；当前真实暴露是什么；现在该先补单、补保护单还是撤退。
- 流程要求：`执行` 和 `等待` 不只处理正常订单生命周期，也要覆盖执行异常和状态不确定的恢复路径。

## 4. 投研与回测

### US-08 用户带着外部看到的交易策略来让系统评估或吸收启发

- 触发：用户带来社区、论坛、论文、KOL 或网络上看到的现成策略。
- 核心问题：这套策略有没有启发；值不值得吸收、改写和回测。
- 流程要求：优先投影到 `RESEARCH -> BACKTEST`，而不是直接进入主线执行。

### US-16 用户请求跨产品分析某种加密资产与其他市场的关系

- 触发：用户怀疑某种加密资产和其他市场或其他资产存在联动。
- 核心问题：这种联动是否真实；能否帮助理解当前资产，或启发新的策略。
- 流程要求：`RESEARCH` 需要接收跨产品联动假设；产出先是观察结论或候选假设，再决定是否进入回测。

### US-17 用户提出一个假想让系统帮忙回测验证有没有用

- 触发：用户自己想到一个粗糙规则或交易假设。
- 核心问题：这个想法历史上有没有用；值不值得继续发展成策略。
- 流程要求：`RESEARCH` 要能把原始假设整理到足够回测的粒度；`BACKTEST` 需要输出“有效 / 无效 / 需改写”的解释。

### US-18 用户拿视频里看到的交易方法来问是否有效和值得学习

- 触发：用户从视频里看到划线、价格行为、裸 K、多周期结构分析等方法。
- 核心问题：这种方法有没有学习价值；能不能被吸收进现有分析或策略体系。
- 流程要求：区分“值得学习的方法论”和“已经可回测的规则”；方法通常要先拆成可验证框架再进入回测。

### US-38 用户想知道当前市场阶段是否已经换了，原策略是否还适用

- 触发：用户感觉市场从趋势变震荡、从震荡变趋势，或从高波动切到低波动。
- 核心问题：当前市场阶段是不是变了；原来有效的策略现在是否还适用。
- 流程要求：`RESEARCH` 和 `REVIEW` 之间需要允许“策略适用环境”这一层判断，而不是只看策略本身好坏。

### US-39 用户希望系统从历史里找类似交易或类似行情来参考

- 触发：用户想知道当前场景以前是否出现过相似案例。
- 核心问题：过去有没有类似的仓位、走势、结构或事件；当时怎么处理，结果如何。
- 流程要求：系统需要支持把当前问题映射到历史案例，用于辅助 `OBSERVE`、`PLAN`、`REVIEW` 或 `BACKTEST`。

## 5. 解释、复盘与迭代

### US-11 用户阶段性复盘过去一段交易与策略迭代是否在变好

- 触发：用户定期回顾一段时间内的交易、事件、决策和策略调整。
- 核心问题：系统有没有越来越好；哪些量化指标在改善，哪些没有。
- 流程要求：`REVIEW` 不只服务单笔平仓，也要支持阶段性系统复盘，并能衔接 `BACKTEST -> ITERATE`。

### US-12 用户质疑系统挂单策略长期成交率过低

- 触发：一段时间内系统频繁给挂单计划，但成交率长期偏低。
- 核心问题：挂单策略是否过于保守；是不是因此错过了太多机会。
- 流程要求：把长期未成交结果沉淀到 `REVIEW`；`ITERATE` 需要允许围绕执行方式本身做调整。

### US-20 止损后用户怀疑被洗掉并询问要不要重新上车

- 触发：刚止损，行情很快又回到原方向。
- 核心问题：这是新的纪律性机会，还是情绪化重入。
- 流程要求：允许 `REVIEW` 与下一轮 `OBSERVE` 快速衔接，但必须把“基于新证据的重入”和“报复性追回”区分开。

### US-27 用户觉得系统观点前后变了想知道为什么

- 触发：同一标的在不同时点得到明显不同的判断或执行建议。
- 核心问题：为什么观点变了；变化来自结构、周期、环境还是执行条件。
- 流程要求：`OBSERVE` 和 `PLAN` 必须保留可追溯依据，能解释“这次相对上次到底变了什么”。

### US-40 用户觉得回测看起来不错但实盘阶段性变差，来问策略是不是失效了

- 触发：回测表现不错，但近期实盘结果、成交质量或执行体验明显变差。
- 核心问题：是策略失效、市场换阶段、执行偏差，还是样本太短导致的错觉。
- 流程要求：`REVIEW -> BACKTEST -> ITERATE` 需要支持“回测与实盘脱节”的诊断，而不只是机械重复回测。
