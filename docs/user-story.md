# 用户故事

## 1. 使用方式

- 只记录真实使用过程，用来校验流程；不定义 UI、字段、表结构或长期制度。
- 某个流程节点没有故事，或故事走不通、不可恢复，先补故事或改流程。
- 当前只覆盖高频、高风险场景，不追求一次写尽。

## 2. 开仓前

### US-01 明确标的问机会

- 触发：用户带着明确标的进入，或不指定标的、要求系统先扫描后收敛。
- 核心问题：现在哪些机会值得看；若只做一单，做哪一个；是否应正式 `不做 / 空仓`。
- 流程要求：开仓前起点既可来自全市场扫描，也可来自单标的观察；`不做 / 空仓` 必须是正式、可解释输出。

### US-02 系统不建议但用户仍想做

- 触发：系统给出完整计划，但明确提示盈亏比、胜率、位置或条件不理想。
- 核心问题：用户知情后，是否仍要继续执行。
- 流程要求：区分“风险提示”和“硬性阻断”；记录系统提醒与用户确认。

### US-03 认可方向但不同意执行方式

- 触发：用户认可方向，但不认可系统建议的执行方式。
- 核心问题：系统建议 `MKT`，用户想改 `PLACE`；或相反。
- 流程要求：记录“系统原建议”和“用户最终执行方式”的差异，复盘时拆开方向问题和执行问题。

### US-04 挂单等待中再次询问是否继续等

- 触发：订单未成交、快成交、行情波动放大，或用户主动怀疑原挂单位置。
- 核心问题：继续等、撤单、改价，还是重做计划。
- 流程要求：`WAIT-UNTIL-FILL` 是活跃判断阶段；必须能回到 `OBSERVE / PLAN`。

### US-13 拿到计划后要求解释依据

- 触发：用户拿到计划后，要求展开方向、位置和执行方式的依据，并先看最强反例、失效条件和明确 `不做` 条件。
- 核心问题：这些依据、反证和失效点是否足以支撑当前计划，而不是只挑支持论据。
- 流程要求：`OBSERVE / PLAN` 输出必须可解释、可追溯，并并列呈现支持证据、反方证据、失效条件与 `不做 / 延后` 分支。

### US-90 要求系统在多周期下标出结构区和执行区间

- 触发：用户要求系统把多周期结构、趋势线、结构区、执行区、斐波那契位和技术指标放进同一语境，而不只给方向结论。
- 核心问题：哪些层级和信号共振，哪些冲突；哪些只是参考，哪些足以转成入场、止损、止盈和挂单条件。
- 流程要求：`OBSERVE` 需要支持多周期结构、锚点和指标解释；`PLAN` 需要把这些信息映射回可执行区间，并区分参考项与执行项。

### US-19 错过入场点后问要不要追

- 触发：用户错过原本入场点，行情已经走了一段。
- 核心问题：现在还能不能追，还是原机会已结束。
- 流程要求：支持“原计划已错过”状态；通常回到新的 `OBSERVE / PLAN`。

### US-23 多个机会并存但只能做一个

- 触发：同时存在多个候选机会，或看似不同的机会本质上押同一类风险。
- 核心问题：优先做哪一个；哪些应保留代表性最好的一笔，哪些组合上不该一起做。
- 流程要求：`OBSERVE / PLAN` 需要支持候选机会排序、相关性和共同因子判断，而不只逐笔独立看。

### US-114 多个账户和交易所分散有仓位、挂单与保证金时，开新仓前先确认真实火力

- 触发：不同账户、交易所或载体里已分散有仓位、挂单和保证金，或资金还在划转、赎回、借入和释放途中。
- 核心问题：当前真实可用火力有多少；哪些暴露和占用已经存在；这笔新机会是现在能做，还是要等资金结算、撤旧单或降旧暴露后再做。
- 流程要求：开仓前需要汇总跨账户净暴露、挂单占用和可用保证金，并区分“理论能做”和“资金已结算可执行”；支持 `等到账再做 / 缩规模改做 / 取消计划`。

### US-24 结构未破但环境变了

- 触发：价格结构未破，但波动、成交量、情绪、宏观语境或临近事件已明显变化。
- 核心问题：原计划仍有效，还是应整体作废重做。
- 流程要求：回返 `OBSERVE / PLAN` 不能只靠价格触发，也要覆盖环境变化和事件前后语境切换。

### US-29 条件型计划反复回来确认是否满足

- 触发：系统之前给的是“等条件满足后再建仓”的计划，后来又出现反复确认、瞬时触碰、插针收回或不同场所触发不一致。
- 核心问题：条件现在算满足、歧义触发、仍需等待，还是已失效且不该追。
- 流程要求：区分“等待条件成熟”和“订单等待成交”；支持 `有效触发 / 歧义触发 / 继续等待 / 删除重做` 分支。

### US-30 提前要快速行情预案

- 触发：用户担心行情太快来不及交互，或将离线一段时间。
- 核心问题：快速拉升、急跌、假突破、关键时间窗内应如何预先处理。
- 流程要求：`PLAN` 需要支持条件分支和离线预案，而不只是单一结论。

### US-42 分批建仓或先试仓怎么布置

- 触发：用户认可方向，但不想一次性打满，打算先试仓、梯度挂单或确认后再加仓。
- 核心问题：首层仓位放哪里；后续加仓或挂单条件是什么；要不要一次挂出整套计划。
- 流程要求：`PLAN` 需要支持“初始仓位 + 后续条件仓位”的整体计划，并区分计划内分批建仓和亏损中摊平。

### US-43 方向没问题但流动性和滑点太差

- 触发：方向看起来成立，但标的流动性差、历史样本太短，或在现货、永续、不同交易所和计价方式之间难以选载体。
- 核心问题：这笔机会是否真的可执行；如果做，应用什么载体、仓位和节奏；如果不可比或成本过高，是否应降级为观察或直接放弃。
- 流程要求：`PLAN` 需要把“方向成立”和“是否值得这样做”拆开；支持 `缩仓 / 分批 / 换载体 / 只观察 / 不参与`。

### US-44 已有底仓，问要不要开对冲单

- 触发：用户本来就持有现货、主仓或中周期暴露，又想开对冲单或短线战术仓。
- 核心问题：这是新增方向性下注、临时对冲，还是在同一标的上叠加主仓和战术仓；它们该如何分别管理又合并看总暴露。
- 流程要求：区分既有暴露、对冲和战术仓；允许正式输出“只处理战术仓，不动主仓”或“主战两仓都要重写”。

### US-45 旧计划未失效时又出现新计划

- 触发：同一标的原来的条件单、提醒条件或挂单还没失效，市场结构却已经走出新机会，甚至方向相反。
- 核心问题：保留旧计划、撤掉旧计划改做新计划，还是允许两个计划并存。
- 流程要求：识别同一标的上的计划冲突，避免无意保留过期条件、形成意外对冲或重复暴露。

### US-86 机会成立，但今天自己的状态未必适合做

- 触发：市场上出现了合格 setup，但用户已连续亏损、明显疲劳、注意力被打断，或当天风险预算与可用保证金已接近上限。
- 核心问题：这笔机会是应该照做，还是虽然机会客观存在，但当前不该由自己来执行。
- 流程要求：`OBSERVE / PLAN` 需要把“机会是否成立”和“用户此刻是否适合参与”拆开；允许正式输出“机会存在，但当前应跳过或降级执行”。

## 3. 持仓中

### US-05 大回撤后恐慌问要不要扛单

- 触发：仓位出现较大回撤、短周期快速逆向，或大小周期信号冲突。
- 核心问题：当前该继续持有、减仓、调保护，还是承认持仓语境已经变了。
- 流程要求：`MANAGE` 需要覆盖恐慌下再判断，并把短期噪音、更大级别启动风险和周期错配分开处理。

### US-06 回撤放大时考虑反向短单或锁仓

- 触发：原仓位回撤放大，用户考虑反向短单、锁仓，或已有对冲腿失配、失效。
- 核心问题：现在该补回对冲、拆回单边，还是直接退出。
- 流程要求：`MANAGE` 需要识别总暴露与净暴露，并把 `继续维持对冲 / 拆回单边 / 直接退出` 作为正式分支。

### US-131 拆掉一条对冲腿看似减仓，实际可能把剩余仓位推向更危险状态

- 触发：用户持有互相对冲、共享保证金缓冲或共同表达同一风险的两腿仓位，其中一腿已经盈利、接近目标、要迁移场所，或看起来应该先关掉。
- 核心问题：现在先平这一腿是真在降风险，还是会让剩余暴露突然从“被对冲”变成裸露单边，并把保证金压力和事故概率一起抬高。
- 流程要求：`MANAGE / RECOVERY` 需要在拆对冲前先重算净暴露、保证金缓冲和最坏情况；支持 `先一起缩 / 先补保证金 / 保留对冲 / 整体退出`，而不是把“先平盈利腿”默认当成减风险。

### US-07 带消息面问对持仓影响

- 触发：消息面、外部观点或研究结论打断原持仓判断，用户反复追问开仓理由是否还在。
- 核心问题：哪些新信息只是噪音，哪些足以削弱、推翻或修正原 thesis。
- 流程要求：`OBS-POS / PLAN-POS` 需要持续复查开仓理由，并说明外部信息的采纳、剔除依据及其对管理动作的影响。

### US-09 定期回来看持仓整体是否要调整

- 触发：用户定期回来看整体持仓健康度，或要求系统按时间表自动拍快照并接续管理。
- 核心问题：当前总仓位、相关暴露、挂单和观察重点是否需要调整；用户回来后应从哪里接手。
- 流程要求：`MANAGE` 需要支持组合视角的周期性和自动化体检，把快照、变化摘要、相关性风险和待处理项串起来。

### US-21 部分成交后剩余挂单怎么处理

- 触发：订单只成交一部分，仓位已经建立，但剩余挂单仍在。
- 核心问题：剩余挂单要不要保留、改价、撤掉，还是结合现有仓位重做计划。
- 流程要求：`partial fill` 之后会同时存在“仓位管理”和“剩余订单管理”；系统需要同时承接两者。

### US-22 已有仓位还能不能再开一笔

- 触发：已有仓位时又出现第二次入场、新机会，或用户觉得仓位、杠杆、资金分配不舒服。
- 核心问题：还能不能再开、该加减多少、是否要把盈利仓轮动到更强机会。
- 流程要求：`MANAGE` 需要把方向判断、仓位规模、杠杆和资金轮动拆开，并与当前风险预算一起重算。

### US-31 自行下单后让系统接管

- 触发：用户先手动下单，或在交易所直接加减仓、改保护、补保证金、切账户模式后再让系统接管。
- 核心问题：当前真实仓位、保护和计划偏离是什么；还能继续沿用旧计划，还是必须先对齐后重做判断。
- 流程要求：接管前要先完成现实状态对齐，并把保护是否仍有效、是否会按当前语境如预期触发与成交作为独立状态；不能把“看起来已经设过保护”直接当成当前已受保护；再决定继续 `MANAGE`、补保护还是回到新的 `OBS-POS / PLAN-POS`。

### US-33 接近爆仓或保证金吃紧

- 触发：仓位接近爆仓、可用保证金不足、维持保证金压力升高，且补保证金、调仓资金或释放火力的动作还在路上。
- 核心问题：该先扛一会儿等资金、立刻减仓或对冲保命，还是把“正在补钱”视为尚未发生。
- 流程要求：`MANAGE / RECOVERY` 需要把“补救动作已发起”和“账户当前真实生存能力”拆开；在途资金不能直接当成保护已生效，并覆盖 `到账前先降风险 / 到账后重算计划 / 补救失败直接退出`。

### US-130 想减风险，但减仓或补保护动作先被已有占用挡住

- 触发：仓位承压时，用户想减仓、补保护、切换退出方式或迁移风险，但已有挂单、资金占用、规则约束或账户模式先把动作挡住、拒掉或只接受一部分。
- 核心问题：现在到底哪些旧单、占用或约束在阻止真正的降风险；在补救动作尚未被确认前，当前仓位是不是其实仍处在原风险，甚至更糟。
- 流程要求：`MANAGE / RECOVERY` 需要把“用户已经下达减风险意图”和“减风险动作已真正生效”拆开；先定位冲突占用、释放顺序和可行补救，再决定 `撤旧单后重提 / 直接改用别的退出方式 / 临时对冲 / 直接退出`。

### US-35 原方向失效，要不要直接反手

- 触发：用户认为原方向逻辑被打破，甚至出现了反方向机会。
- 核心问题：是先止损离场后重新观察，还是可以直接从多变空、从空变多。
- 流程要求：系统要把“结束旧仓位”和“开启新方向”拆成两个判断，避免把反手当成单一动作。

### US-36 回来时发现状态已变化

- 触发：用户离线后回来，发现状态已变化，或状态未变但忘了原计划；也可能系统记录与交易所、新鲜度或提醒状态对不上，且离线期间还可能错过了原定管理动作。
- 核心问题：刚才发生了什么；哪些状态现在可信；在没对齐前能不能继续给建议。
- 流程要求：恢复路径需要支持事件回放、状态对账、可信度判断和错过节点重建；在关键仓位、挂单、保护和管理动作未对齐前，不应直接推进后续管理。

### US-37 交易所执行异常，这单到底算没算

- 触发：下单失败、状态异常、止损失真、价格源异常，或交易所网页、App、API、规则本身出现故障、限仓、下架和迁移要求。
- 核心问题：当前真实暴露和最坏风险是什么；现在该补保护、迁移、临时对冲，还是直接退出。
- 流程要求：恢复路径需要覆盖执行异常、场所不可用和场所规则变化；先确认真实成交与触发链，再决定补救、迁移、拆回临时动作和主线回接。

### US-96 部分强平或自动减仓后，剩余仓位怎么接

- 触发：仓位经历部分强平、自动减仓、保险基金接管或交易所强制降杠杆，账户里还剩残余仓位、残余挂单或失配保护。
- 核心问题：现在到底还剩什么暴露；这笔交易还能不能按原 thesis 继续管理，还是必须先退出、重建保护，并把剩余部分当成新交易。
- 流程要求：恢复路径需要支持“非自愿减仓后的残余状态”；先完成仓位、保证金、保护单和计划身份重建，再决定继续 `MANAGE` 还是转入 `REVIEW`。

### US-46 已部分止盈后剩余仓位怎么拿

- 触发：用户未到最终目标就想先减仓或离场，或已部分止盈、提保本、移动保护、锁利后又想把卖掉的仓位加回来。
- 核心问题：剩余仓位继续按原计划拿，还是已经进入新的管理节奏；回补是计划内二次上车，还是情绪化追回。
- 流程要求：`MANAGE` 需要支持“已兑现部分收益后的剩余仓位”这一独立状态；一旦回补或重排保护，均价、风险、目标和持仓身份都要一起重算。

### US-51 亏损打到加仓区，这次补仓是计划内还是摊平

- 触发：仓位已经浮亏，价格来到原本设想的二次入场区，或用户临时想通过补仓拉低成本。
- 核心问题：现在加仓是合理延续原计划，还是只是在为错误找补；如果不加，是继续拿、减仓还是止损。
- 流程要求：`MANAGE` 需要区分“计划内分批建仓”和“情绪化摊平”；一旦加仓，原失效条件、总风险和退出节奏都要一起重算。

### US-52 长时间不推进，要不要时间止损

- 触发：仓位长时间不推进，或即将跨关键事件、低流动性时段。
- 核心问题：继续拿、缩仓、临时对冲，还是承认这笔交易虽然未触发硬止损但已不值得继续占用风险预算。
- 流程要求：`MANAGE` 需要把时间效率、事件风险和方向判断拆开；允许因时效或事件穿越成本而正式离场或临时降风险。

### US-70 方向没变，但资金费率或基差把持仓拖得越来越难受

- 触发：方向判断暂时没变，但资金费率、借币成本、基差、展期成本或其他持有成本持续侵蚀收益。
- 核心问题：继续承担持有成本等待方向兑现，还是换产品、降杠杆、做临时对冲，甚至承认这笔交易在当前工具上已不划算。
- 流程要求：`MANAGE` 需要把“方向是否还对”和“持有这笔暴露的经济性是否仍成立”拆开；允许因 carry 结构恶化而正式退出或迁移仓位。

### US-72 中断后同时有仓位、挂单、候选机会和研究线程，先恢复哪个

- 触发：恢复时同时存在活跃仓位、未完成挂单、候选机会和研究线程。
- 核心问题：现在第一优先级是什么；哪些必须先处理，哪些可以稍后再看。
- 流程要求：恢复需要按风险和时效排序，至少区分 `先处理活跃风险`、`再处理待执行订单 / 条件`、`最后回到观察与投研`。

### US-78 多次加减仓和对冲后，这笔仓位还是不是原来那笔

- 触发：同一标的经历多次加减仓、对冲、移保护，或用户想把短线拿成波段、把失纪律单改口成大周期持有。
- 核心问题：当前剩余暴露还是不是原来那笔；这是合法的周期切换，还是在为失误续命。
- 流程要求：`MANAGE / REVIEW` 需要支持“交易身份漂移”分支；一旦周期或身份变化，必须重建 thesis、失效条件、仓位规模和检查节奏。

### US-120 用户怀疑系统保存的数据已经混乱或出错，要求先做一轮自检和整理

- 触发：用户感觉系统里的仓位记录、订单状态、候选机会、研究结论或历史样本出现重复、缺失、冲突或明显过期残留。
- 核心问题：哪些数据只是还没同步，哪些已经真的错误、重复或失去引用对象；在这些问题没排清前，系统当前还能不能继续可靠地给建议。
- 流程要求：恢复路径需要支持“先做数据健康检查”这一正式分支，把 `可直接修正`、`需人工确认`、`应归档清理`、`暂时冻结不用` 拆开；在关键链路恢复可信前，不应继续推进主线。

## 4. 投研与回测

### US-39 从历史里找类似交易或行情

- 触发：用户想知道当前场景或某个特定历史时间窗口，以前是否出现过相似案例、事件或走势。
- 核心问题：过去有哪些类似样本；当时真正重要的事件和处理方式是什么；结果如何。
- 流程要求：支持把当前问题映射到历史案例，并支持按时间锚点回收事件，用于辅助 `回放 / OBSERVE / REVIEW`。

### US-56 从自己的历史交易里提炼候选策略

- 触发：用户已经积累了一批真实交易，也包含没成交、没做成、主动放弃的样本，怀疑其中有重复模式。
- 核心问题：哪些入场、出场、过滤或执行模式值得整理成候选策略；哪些只是事后看起来可惜。
- 流程要求：`RESEARCH` 既要能从已成交交易里抽出候选规则，也要能把未成交、未执行、主动放弃样本流入 `RESEARCH -> BACKTEST`，用于检验新的过滤条件或执行规则。

### US-57 策略有效后想知道适用边界

- 触发：某个策略、模式或假设看起来可用，但在不同市场阶段、日历语境、标的和周期里的表现不一致。
- 核心问题：它适用于哪些环境；哪些条件下一用就容易失效。
- 流程要求：`BACKTEST` 不能只给总收益和胜率，还要尽量产出适用边界、失效环境和不应滥用的前提，便于决定是否进入主线。

### US-58 回测很好但怀疑吃了未来信息或理想成交

- 触发：某个想法回测收益很好，但信号时点、确认方式、滑点、手续费、流动性或成交假设看起来过于理想。
- 核心问题：这套结果反映了真实可执行优势，还是吃了未来函数、数据穿越或不现实成交。
- 流程要求：`BACKTEST` 不能只给收益结论，还要识别测试前提是否破坏了真实可执行性，避免把纸面优势直接接回主线。

### US-59 两个只差一个条件的想法，哪个条件真有用

- 触发：用户已经形成两个很接近的规则版本，只差一个过滤条件、确认条件或执行约束。
- 核心问题：到底是哪一个条件带来了边际优势；去掉它后策略是否仍成立。
- 流程要求：`RESEARCH -> BACKTEST` 需要支持对相邻版本做变量隔离，而不只是给整套策略一个笼统结论。

### US-60 把投研或回测结论接回今天观察，但先不执行

- 触发：研究或回测结论初步成立，但用户想先确认今天是否处在适用语境，或先做影子跟踪、纸上执行、极小试错。
- 核心问题：这份结论现在应被视为可实盘策略、待观察候选，还是仍只适合继续 research。
- 流程要求：`RESEARCH / BACKTEST` 需要支持“转成观察清单或影子执行框架 -> 记录触发、漏做、滑点和语境偏差 -> 再决定是否进入 `OBSERVE / PLAN`”的过渡阶段。

### US-73 要求系统解读宏观材料或市场报告，并说明对交易标的的影响

- 触发：用户要求系统解读宏观材料、市场报告、跨资产关系或某标的相对同类资产的异常强弱，并追问它们对关注标的意味着什么。
- 核心问题：这些外部信息哪些真正重要；它们更像新语境、短时失真、风险提示还是背景噪音。
- 流程要求：`RESEARCH` 需要支持“消化外部材料 -> 提炼结论 -> 映射到具体标的”；输出既可服务 `OBSERVE / PLAN / MANAGE`，也可继续沉淀为候选研究结论。

### US-81 从极端历史日期开始做互动式回放，边走边模拟完整交易流程

- 触发：用户想检验自己的策略在极端行情、关键事件日或历史转折点里是否站得住，要求系统从某个历史日期开始回放并一步步走完整个交易过程。
- 核心问题：如果只能看到当时已经发生的信息，自己在那个时间点会怎么观察、怎么做计划、怎么管理仓位；随着时间推进，这套策略还能不能继续成立。
- 流程要求：`回放` 需要支持“选起点 -> 锁住未来信息 -> 生成当时语境下的 `OBSERVE / PLAN` -> 推进时间 -> 更新模拟仓位、订单和市场上下文 -> 继续 `MANAGE / REVIEW`”的循环。

### US-84 自动长期记录多市场快照，并在一段周期后做跨资产联动分析

- 触发：用户设定自动化，持续记录多个市场、资产、叙事线索和连续指标；积累一段时间后，再要求系统做周期分析。
- 核心问题：哪些资产和指标存在稳定或阶段性的相关性、领先滞后、共振、背离和拐点；哪些只是短期噪音。
- 流程要求：支持“持续采样 -> 按周期汇总 -> 输出跨资产和时间序列研究结论”；结果要能回接 `RESEARCH / OBSERVE / ITERATE`。

### US-88 已有方向性 thesis 时，要求系统主动找反例和失效证据

- 触发：用户已经偏多、偏空，或多份研究材料对同一标的给出冲突结论，担心系统只会顺着原立场补材料。
- 核心问题：当前 thesis 最强的反例是什么；哪些证据更领先、更可靠、与当前时间框架更相关；哪些信号必须再观察。
- 流程要求：`RESEARCH` 不能只罗列多空论据，还要支持主动寻找反证、证据分层、冲突归因和不确定性输出；必要时产出“继续观察哪些信号再决策”的回接条件。

### US-102 指定某套方法论，要求系统研究后沉淀成可版本化的策略

- 触发：用户带来外部策略、视频方法、粗糙假设、截图标注或直接点名某套方法论，要求系统先 research，再整理成可执行策略。
- 核心问题：其中哪些是叙事、术语和主观裁量，哪些能被拆成可观察条件、可执行规则和可回测变体；拆完后哪一版真的有用。
- 流程要求：`RESEARCH` 需要支持“方法消化 -> 拆成候选规则 -> 生成多个可回测版本 -> 回测 -> 迭代”的完整链路，并保留版本演化、适用语境、失效条件和淘汰原因。

### US-108 识别主力流动性猎杀痕迹，并把识别特征沉淀成可验证策略

- 触发：用户要求系统识别扫止损、扫流动性、假突破假跌破、快速回收等痕迹，或某类形态在历史上看起来总是很明显。
- 核心问题：这些痕迹到底是可前置识别的 edge，还是只适合事后贴标签；若继续研究，哪些条件必须先被定义到盘中可观察、可记录、可反驳的粒度。
- 流程要求：`RESEARCH` 需要支持“历史定位 -> 提炼特征 -> 形成识别规则或执行假设 -> 进入 `BACKTEST` -> 继续迭代筛掉伪特征”；若实时识别条件说不清，就应降级为观察提示。

## 5. 解释、复盘与迭代

### US-11 阶段性复盘：交易和策略是否在变好

- 触发：用户定期回顾一段时间内的交易、事件、决策和策略调整。
- 核心问题：系统有没有越来越好；当前结果是否过度依赖少数极端样本、违规盈利或同一类集中风险。
- 流程要求：`REVIEW / ITERATE` 需要把总收益、样本分布、过程合规性和组合集中度拆开，而不只看账户曲线。

### US-12 质疑挂单策略长期成交率过低

- 触发：一段时间内系统频繁给挂单或等待计划，但长期成交率偏低，且市场事后常走出行情。
- 核心问题：挂单策略是否过于保守；这些没做成的样本说明过滤太严、执行太慢，还是当时本就该放弃。
- 流程要求：`REVIEW` 需要覆盖未成交、未执行和正式 `不做 / 继续等` 的样本，并允许围绕执行方式和过滤条件做迭代。

### US-20 止损后怀疑被洗掉，要不要重新上车

- 触发：刚止损，行情很快又回到原方向。
- 核心问题：这是新的纪律性机会，还是情绪化重入。
- 流程要求：允许 `REVIEW` 与下一轮 `OBSERVE` 快速衔接，但必须把“基于新证据的重入”和“报复性追回”区分开。

### US-27 观点前后变了，为什么

- 触发：同一标的在不同时点得到明显不同的判断或执行建议。
- 核心问题：为什么观点变了；变化来自结构、周期、环境还是执行条件。
- 流程要求：`OBSERVE / PLAN` 必须保留可追溯依据，能解释“这次相对上次变了什么”。

### US-63 反复偏离计划执行，问题在策略还是执行

- 触发：一段时间里多次出现提前进、提前出、没按计划挂单、追单、移动止损、取消保护单或亏损中临时补仓等偏差。
- 核心问题：结果变差是策略本身有问题，还是执行偏差反复吞掉了原本可能存在的 edge。
- 流程要求：`REVIEW` 需要把“系统原计划”“用户实际执行”“最终结果”持续对照；若是重复性执行偏差，要能单独沉淀为执行层改进。

### US-64 连续亏损或明显失常后，是继续、缩手还是暂停

- 触发：用户连续止损、短期回撤扩大、执行动作变形，或暂停一段时间后准备恢复实盘。
- 核心问题：现在是继续、降风险继续、继续观察，还是先暂停；若要恢复，先看到哪些证据才算不是靠手痒或 FOMO 重启。
- 流程要求：`REVIEW / RECOVERY / ITERATE` 需要把“暂停决定”和“恢复门槛”连成一条路，并支持 `继续观察`、`影子恢复`、`小仓恢复`、`恢复常规` 的分级路径。

### US-65 中断后回来接上观察中的候选机会

- 触发：用户离开前还在观察多个标的或条件型计划，期间没有开仓；回来时市场继续运行，且离线窗口内可能出现过触发又失效。
- 核心问题：哪些候选机会仍有效，哪些已失效、错过或属于歧义触发；现在哪些计划还能留，哪些必须正式删除并重做。
- 流程要求：支持恢复“开仓前观察队列”，并重建离线窗口内的价格路径、提醒状态和计划条件，区分 `仍在等待`、`有效触发未执行`、`歧义触发`、`已失效需重做`。

### US-66 复盘单笔大赚或大亏，想知道是做对了还是运气

- 触发：某一笔交易结果明显超出平时，或刚经历一段连续盈利后开始想放大仓位、放宽过滤条件。
- 核心问题：这次结果来自正确过程、执行偏差、运气放大，还是风险失控；现在放大风险是否合理。
- 流程要求：`REVIEW / ITERATE` 需要把“结果好坏”和“过程质量”拆开；任何放大仓位或放宽条件的决定，都应绑定可解释证据，而不是跟着近期收益走。

### US-79 被强平、自动减仓或穿仓后，先还原事故链再决定怎么恢复

- 触发：仓位被强平、自动减仓、极端插针穿透保护，或因保证金与流动性问题在短时间内发生事故级亏损。
- 核心问题：究竟是哪一段决策、杠杆、保证金安排、保护缺口或执行异常让事故发生；现在还有没有继续交易的前提。
- 流程要求：`REVIEW` 不能只记录结果差；还要把计划、手动操作、交易所事件和错过的恢复窗口串成事故链，并给出明确恢复门槛。

### US-89 同一 thesis 连续重试几次后，要不要停止再打

- 触发：同一标的、同一结构或同一核心 thesis 在短时间内已经止损、被洗、错过后追回，或反复小亏小赚多次，用户仍想继续等下一次同类机会。
- 核心问题：这还是纪律性的重复试错，还是已滑向报复性重试、过度关注单一想法；现在该继续、降级、只观察，还是先把这条 thesis 暂停。
- 流程要求：`REVIEW / ITERATE` 需要把多次重试视为同一想法的连续样本，而不是几笔彼此无关的交易；输出要能区分“edge 仍在但样本未完”“想法暂时失效”“用户已被单一 thesis 绑住”。

### US-119 用户觉得策略太多了，要求系统做一轮整理、筛选和压缩

- 触发：系统里已经积累了不少候选策略、方法论变体、过滤版本和阶段性结论，用户明显感觉数量过多、边界模糊。
- 核心问题：哪些策略本质上只是同一 thesis 的不同说法、轻微变体或适用边界重叠版本；哪些仍值得保留，哪些应归并、降级、暂停或淘汰。
- 流程要求：`REVIEW / ITERATE` 需要支持面向策略池的存量治理，把 `保留`、`合并`、`降级观察`、`暂停`、`淘汰` 拆开，并说明依据来自样本质量、适用边界、相互重叠度、执行复杂度和真实使用频率。

### US-129 复盘时发现自己经常“方向看对但利润没拿到”，先定位到底卡在哪个环节

- 触发：一段时间里，用户反复在事后看对方向或结构，但真实结果表现为没成交、拿不住、过早止盈、追着回补，或回测看起来不错而实盘因成本、成交和管理链路变差。
- 核心问题：edge 真丢在观察、计划、下单、成交、持仓管理、恢复衔接还是情绪执行；最该修的是 thesis、执行链路、交易载体、持有经济性还是风险预算。
- 流程要求：`REVIEW / ITERATE` 需要把“看对了”和“赚到了”之间的链条正式拆开，至少区分 `没等到`、`不敢做`、`做了但仓位不对`、`拿不住`、`保护失真`、`恢复断档`、`成本吃掉 edge` 等节点。
