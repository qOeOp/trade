# 用户故事

## 1. 使用方式

- 只记录真实使用过程，用来校验流程；不定义 UI、字段、表结构或长期制度。
- 某个流程节点没有故事，或故事走不通、不可恢复，先补故事或改流程。
- 当前只覆盖高频、高风险场景，不追求一次写尽。
- 文中的 `观察 / 决策 / 执行 / 恢复 / 复盘` 主要指阶段；`WAIT-CONDITION`、`WAIT-UNTIL-FILL` 主要指当前 plan 或订单状态，不单独升格为主流程阶段，也不阻断其他 plan 继续流转。持仓后的加减仓、补保护、止盈止损等推进，仍属于持仓语境下的 `观察 / 决策 / 执行`。

### US-00 重新进入系统时先恢复并对账当前交易链

- 触发：用户重新进入系统，请求继续当前交易，或系统定时恢复当前语境。
- 用户入口用例：
  - 主动续接：“我回来了，继续刚才那笔。”
  - 先看当前状态：“你先帮我看下现在账户、挂单和持仓是什么状态。”
  - 定时恢复：“到点了，先把当前链路拉起来给我看。”
- 核心问题：当前真实状态是什么；哪些信息可信；当前链应从哪里继续。
- 当前语境：重新进入系统时，`RECOVERY` 不应再主要依赖用户回忆。只要交易所 API 可用，系统要先自动拉取订单、持仓、余额、活跃挂单和成交回报，再和本地上下文、当前计划、已有判断做一轮对账。用户补充主要用于解释 API 看不到的意图、场外动作、异常操作和接下来想继续还是放弃。
- 系统必须支持：
  - `RECOVERY` 默认先以交易所 API 为事实源，对齐 `持仓`、`挂单`、`成交`、`余额 / 可用保证金`
  - 对账时同时比较 `API 结果`、`本地持久化状态`、`当前交易计划` 和 `最近判断记录`，而不是只看某一层
  - 若 API 与本地记录不一致，系统要先给出当前可信状态和分歧点，再决定是否需要用户补充说明
  - 用户补充信息主要用于解释 `为什么出现这个状态`、`是否还有 API 看不到的动作`、`当前计划是否仍要继续`
  - `空仓` 不能直接等于“无事可做”；系统仍要判断当前链应回到 `OBSERVE / PLAN`、`WAIT-CONDITION`、`WAIT-UNTIL-FILL`，或在持仓语境下继续 `OBSERVE / PLAN / EXECUTE`，还是正式结束
- 不宜直接跳过 `RECOVERY` 进入后续阶段的情况：
  - 订单、持仓、成交和本地计划之间尚未完成 API 对账
  - API 回报与本地记录明显冲突，系统还没有说明当前以哪一层为准
  - 用户补充了 API 看不到的动作，但系统还没有把这些信息映射回当前链和当前计划
- 正式输出：
  - 已对齐的当前状态，并进入 `OBSERVE / PLAN`
  - 已对齐的当前状态，并进入 `WAIT-CONDITION`
  - 已对齐的当前状态，并进入 `WAIT-UNTIL-FILL`
  - 已对齐的当前状态，并回到持仓语境下的 `OBSERVE / PLAN / EXECUTE`
  - 标出待确认项，保留在 `RECOVERY`
- 回退 / 恢复路径：
  - 若 API 已足够对齐主要状态，结束 `RECOVERY` 进入对应阶段
  - 若仍存在关键状态缺口或事实冲突，留在 `RECOVERY` 继续补齐

## 2. 开仓前

### 固定要求

- `PLAN` 是开仓前的正式计划对象，不只是方向结论；至少要保留 `主要依据`、`反方证据`、`风险提示`、`失效条件`。
- 决策处理的是 `OBSERVE <-> PLAN` 的收敛过程；若用户对当前计划有调整、补充或反对，需要进入计划分歧处理；若在判断或分歧处理中发现上下文不足，可回到 `OBSERVE` 补信息后再回到 `PLAN`。
- `PLAN` 必须同时回答两层问题：`机会是否成立`，以及 `当前是否适合参与`；不能把“看对了”直接等同于“现在就做”。
- `PLAN` 默认支持 `简洁结果` 和 `展开结果` 两种视图；用户追问时，应能展开到 `入场`、`止损`、`止盈`、`分批`、`移动规则` 各自依据，而不是重新做一轮调查。
- `PLAN` 的参数层至少要能让用户直接看到 `量 / 价 / 止损止盈`；若是分阶段止盈或分阶段止损，每一段都要能展开到 `价格或条件`、`数量或比例`、`建议强度`。
- `建议强度` 要服务真实取舍，而不是装饰字段；用户需要能看懂哪一段是核心建议、哪一段只是弱建议或可选兑现位。
- 默认三段止盈不应被写成必须原样执行；若第一段强度偏低，系统要允许用户折叠成两段，并明确说明省略这一段后失去的主要作用。
- `PLAN` 默认支持条件分支、离线预案、梯度挂单和 `初始仓位 + 后续条件仓位`；这些都是计划形态，不单独升格为新的 story。
- `WAIT-CONDITION` 表示当前详细 plan 已经完成必要确认和计划分歧处理，但触发条件尚未成熟；它属于决策后的等待状态，不算进入执行。只有真正开始挂单或以市价等方式直接进入仓位，才算进入执行。
- `PLAN` 可以是一笔订单，也可以是一组组合；如果这次机会只有按整体理解才成立，系统不能默认把它拆成几笔互不相关的单。
- 开仓前正式输出统一落在这几类：`参与`、`调整后参与`、`继续等待`、`只观察 / 延后`、`不参与 / 放弃`、`回到 OBSERVE / PLAN 重做`、`进入 RECOVERY`。

### US-01 先看有什么机会，再决定做不做、先做哪一个

- 触发：用户带着明确标的进入，或不指定标的、要求先扫描再收敛；也可能先带着研究结论、宏观判断、关键事件解读或外部材料进入，要求先判断这些信息对当前市场判断到底有没有用，或本轮是否值得先做扫描再收敛。
- 用户入口用例：
  - 单标的观察：“看看 `btc` 现在还能不能做多。”
  - 全市场扫描：“现在市场里最值得看的机会是什么？”
  - 带研究结论进入：“我看到有人说 `sol` 这一轮会强于大盘，你帮我判断这个说法现在值不值得参考。”
  - 带宏观材料进入：“这份美联储纪要偏鹰，会不会改变我现在看 crypto 的方式？”
  - 带关键事件进入：“中东局势升级，油价暴涨，现在市场风险偏好要不要重估？”
- 当前语境：系统此时还不应默认“用户已经决定要做哪一笔”。用户给出的标的、事件、观点、材料或结论，首先只是线索，不等于已经可以直接下计划。系统要先补齐当前市场语境；如果用户带来的是事件、材料或研究结论，还要先做一轮求证，判断它是否值得进入当前语境，再进入计划形成。这里既要判断“这是不是机会”，也要判断“现在值不值得做”；如果同时有多个可做机会，还要一起比较，决定先做哪一个，哪些先不做。
- 核心问题：现在有哪些机会值得看；这些机会里哪个最值得先做；如果不做，是因为机会本身不成立，还是因为现在不适合参与。
- 系统必须支持：
  - 能接住不同入口，但默认都要先补齐基础判断材料，而不是只顺着用户一句话直接下结论
  - 若入口是全市场扫描，不应在 scan 内直接完成单标的细筛；应先粗筛产出候选列表，由主 agent 收敛 shortlist，再把少数标的送进与“用户直接给具体标的”相同的单标的判断路线
  - 如果用户带来的是事件、材料或研究结论，不能直接当真，要先求证，再判断它是否真的改变当前机会判断、主要风险和失效条件
  - 不能只堆信息；要把信息整理成后续能直接用于判断的材料：支持理由、反面理由、主要风险、失效条件
  - 要把判断拆成两层：`机会是否成立`，以及 `当前是否适合参与`
  - 即使机会成立，也要检查流动性、滑点、交易载体、执行成本、风险预算和用户状态这些现实约束
  - 如果同时出现多个可做机会，要能一起比较，并明确说明为什么先做这个、为什么不先做另一个
  - 当机会只有按整体理解才成立时，要能直接输出一整套组合计划，而不是默认拆成几笔互不相关的单
  - 进入计划时，至少要形成一个最小轮廓：做哪个、为什么做、主要风险是什么、什么情况下这轮判断作废
- 不宜过早收敛为单一具体交易计划的情况：
  - 用户当前主要在问影响、可信度或语境翻译，还没明确要交易建议
  - 用户输入里包含关键事件、材料或研究结论，但系统还没做完必要求证
  - 机会看起来成立，但流动性、滑点、执行成本、风险预算或用户状态已经明显不适合参与
- 正式输出：
  - 收敛出当前最值得先做的一笔
  - 保留多个候选机会，继续比较
  - 机会成立，但要缩仓、分批或换载体后再做
  - 先观察 / 延后 / 不参与
  - 把需要继续补充的问题交回研究
- 回退 / 恢复路径：
  - 若只是信息还不够，继续留在观察和判断
  - 若出现上下文断裂、已有计划与现实状态冲突，先进入 `RECOVERY`

### US-02 不完全接受建议，但还是想继续做

- 触发：系统已经形成一版计划或明确建议，但用户不完全接受，仍想继续推进；常见情况包括认可方向但不认可位置、认可判断逻辑但不认可仓位、认可机会但想改执行方式，或明知系统提示风险仍坚持继续。
- 用户入口用例：
  - 改执行方式：“方向我认同，但我不想等回踩了，我现在就先进。”
  - 改风险参数：“你这个止损太远了，我想把止损收紧一点，仓位照旧。”
  - 忽略风险提示：“盈亏比一般我知道，但我还是想做，你按我这个想法记下来。”
  - 只接受部分建议：“不要按你说的分批，我想一笔先打进去，后面再看。”
  - 被阻断后仍想继续：“你说这个位置不该做，但我还是想 follow 一点试试。”
- 当前语境：这时系统已经给出一版计划，用户不是在重新找机会，而是在回应这版计划。重点不再是“有没有机会”，而是“用户到底想改什么，改完之后当前真正有效的计划是什么”。这里必须进入正式的分歧处理，不能把用户一句口头偏离直接当成计划已经改完。
- 核心问题：用户到底不同意哪一层；这是在改方向、改风险、改入场方式，还是已经碰到不能继续的硬性阻断；用户继续推进后，当前真正生效的计划是什么。
- 系统必须支持：
  - 要把 `系统原建议`、`用户反馈`、`最终确认计划` 明确拆开，而不是只留下最后一句结论
  - 要区分用户是在改方向、改位置、改仓位和风险、改止损止盈，还是改执行方式
  - 要区分普通风险提示和硬性阻断；前者允许用户知情后继续，后者不能被默默降级
  - 如果用户改了计划但仍可继续，系统要生成一版新的当前计划，并说清楚哪些沿用原计划、哪些已经被改掉
  - 如果用户拒绝的是风险控制或保护建议，系统要同步重算最坏风险和剩余计划，而不是只改一个局部参数
  - 分歧处理后的结果必须可回看，后续复盘时能分清是系统判断问题、用户主动偏离，还是执行层问题
- 不宜直接跳过计划分歧处理进入后续执行的情况：
  - 用户只说“我不想这么做”或“我还是想上”，但还没说清到底改什么
  - 用户改了仓位、止损、止盈、分批或执行方式，但系统还没重算新的风险和计划结构
  - 系统判断已经触发硬性阻断，但用户仍要求继续
- 正式输出：
  - 接受系统原建议，沿原计划继续
  - 在保留原判断的前提下，按用户修改后的版本形成新计划并继续
  - 明确触发硬性阻断，当前不进入执行
  - 因状态不清、风险未重算或上下文不一致，转入 `RECOVERY`
- 回退 / 恢复路径：
  - 若只是修改当前计划，继续留在计划收敛
  - 若用户修改后的计划需要重新看依据，可回到观察和判断
  - 若分歧暴露出真实状态不清、风险失真或上下文断裂，先进入 `RECOVERY`

### US-04 挂单后还没成交，现在怎么办

- 触发：订单已经挂出，但是否真正成交、成交多少、还剩多少未决风险仍未最终确认；常见情况包括订单未成交、部分成交、快成交、撤单 / 改单在途、旧单与新单谁生效未定，或订单因为时效、场所约束、执行规则先失效。
- 用户入口用例：
  - 继续等确认：“单子我已经挂了，但一直没成交，这种情况要继续等吗？”
  - 撤单回报未同步：“我已经撤单了，但交易所那边还没确认，你先别把它当成已经撤干净。”
  - 改单结果待确认：“我刚把价格改了，你先帮我按最坏情况看下现在还挂着哪些单。”
  - 部分成交后重算：“已经成交了一点，剩下这部分还要不要继续挂？”
  - 执行方式失效：“原来的限价单没吃到，但我还是想做这笔，要不要换个方式进？”
- 当前语境：这时重点已经不是重新判断有没有机会，而是先确认“挂出去的订单现在到底处于什么状态”。只要交易所 API 可用，系统就应先以 API 返回的订单、成交和活跃挂单状态为事实源，再和当前计划、本地记录、用户补充说明做对齐，而不是只根据用户一句“我挂了 / 我撤了 / 我改了”就默认执行已经生效、取消或失败。只要订单回报、活跃挂单、已成交量和剩余风险没有对齐，这条链就仍处在正式未决状态。
- 核心问题：现在到底还有没有活跃订单；已经成交了多少；还剩多少风险；原来的机会还在不在；现在该继续等、撤单、改价、重挂，还是回到重做计划。
- 系统必须支持：
  - 要承接这种正式未决状态，而不是把“已经发出下单意图”直接当成“已经成交”或“已经失败”
  - 只要交易所 API 可用，就先看 API 返回的订单和成交状态，再用用户补充去解释 API 看不到的动作和意图
  - 要区分：提交结果未定、撤单在路上、改单在路上、部分成交、订单失效但原判断还没失效
  - 对每次状态变化都要重算：当前活跃订单、已成交数量、剩余风险、仍待执行部分
  - 当用户想继续保留或重挂时，要说明沿用的是原完整计划，还是只保留原方向、但执行细节已经改了
  - 如果订单层面已经失效，但交易想法仍成立，系统要支持回到新的计划，而不是把整条链直接判死
  - 对重复挂单、重复改单、旧单残留和部分成交后的剩余单，要能识别潜在的重复风险
- 不宜直接把当前状态当成已完成或已失败的情况：
  - 用户只说“我挂了”“我撤了”“我改了”，但系统还没有拿到足够可信的对账结果
  - 已成交数量、活跃挂单和剩余风险之间还对不上
  - 原订单已经不适合继续，但系统还没判断应该沿原计划重挂、改执行方式后重挂，还是回到重做计划
- 正式输出：
  - 继续等确认，保留当前未决状态
  - 先按最坏风险处理，再等待进一步确认
  - 撤回重复单或失效单，保留当前计划继续执行
  - 按已成交部分重算后重挂
  - 改执行方式后重挂
  - 正式删除当前执行尝试，并回到观察和判断
  - 因真实状态不清，转入 `RECOVERY`
- 回退 / 恢复路径：
  - 若只是订单继续等待成交，留在成交等待
  - 若原执行方式失效但原判断仍成立，回到计划收敛新执行版本
  - 若用户反馈与订单记录、成交记录或当前风险明显冲突，先进入 `RECOVERY`

### US-29 还没挂单，只是在等条件，现在还算数吗

- 触发：原计划要求等条件满足后再建仓，后来又出现反复确认、瞬时触碰、插针收回、不同场所触发不一致、错过原入场点、结构未破但环境或事件语境已变，或同一标的旧计划未失效时又冒出新计划。
- 用户入口用例：
  - 复查等待计划：“之前那笔我不是说等条件到吗，你现在看它算已经触发了吗？”
  - 处理歧义触发：“刚刚好像碰到了，但又马上收回，这种到底算不算触发？”
  - 错过后要不要追：“原来的点位已经过去了，现在还算原计划吗？”
  - 旧计划和新计划并存：“旧计划还没失效，但现在又冒出一个新想法，这两个怎么处理？”
  - 环境变化后重判：“结构还没坏，但事件变了，这个等待计划还有效吗？”
- 当前语境：这里处理的是“还没挂单，只是在等计划条件成熟”，不是“订单已经挂出后等成交”。重点不是订单有没有成交，而是原计划在现在这个时点是否还成立、是否已经触发、是否只是擦边触发，或者是否已经被新的市场语境和新计划替代。
- 核心问题：原计划现在算已经触发、还要继续等、已经错过、被新计划覆盖，还是应该正式作废重做。
- 系统必须支持：
  - 要区分“等待条件成熟”和“订单等待成交”，不能因为价格碰了一下就自动把计划推进到执行层
  - 要回看原计划的触发条件、失效条件和等待理由，判断当前是否仍在沿同一计划推进
  - 要能处理：已经触发、擦边触发、继续等待、已经错过不追、被新计划替代
  - 当环境、事件或研究结论变化时，要判断是“原计划还成立但条件未到”，还是已经需要进入新的观察和判断
  - 若同一标的同时存在旧计划和新计划，要说明它们是并存候选、互相替代，还是其中一个应正式删除
- 不宜直接沿旧计划继续推进的情况：
  - 触发条件只被部分满足或存在明显歧义，但系统还没说明为什么算触发或不算触发
  - 原入场窗口已经错过，但系统还没判断现在是追价、等下一次机会，还是正式删除
  - 事件、环境或新计划已经足以改变原判断，但系统还没说明旧计划是否被替代
- 正式输出：
  - 原计划已经触发，进入后续计划 / 执行
  - 原计划继续等待
  - 原计划只是擦边触发，回到观察和判断复查
  - 原机会已错过不追
  - 旧计划被新计划替代
  - 正式删除原计划，并回到观察和判断
- 回退 / 恢复路径：
  - 若只是等待条件尚未成熟，继续留在等待
  - 若对触发有效性或计划归属产生疑问，回到观察和判断
  - 若当前链上下文或市场状态不清，先进入 `RECOVERY`

## 3. 持仓中

### 固定要求

- 持仓中默认先对齐真实状态；只要交易所 API 可用，`持仓`、`挂单`、`成交`、`保护单`、`保证金 / 可用余额` 都先以 API 为事实源。
- 持仓中的正式计划不能只写“继续拿”或“准备出”；至少要明确 `剩余仓位`、`当前有效保护`、`主要风险`、`下一检查点`。
- 当用户已经明显是在追问当前判断，例如“这笔还拿吗”“现在还是看多吗”“我这仓位现在怎么办”，回复默认必须先给一个可快速扫读的 plan 摘要，不能把动作结论埋在大段分析里。
- 这个 plan 摘要至少要能直接扫到：`当前动作`、`方向状态`、`剩余仓位处理`、`保护 / 止损要求`、`下一检查点`。
- 持仓中的计划既可以是单边仓位，也可以是一整套对冲 / 组合；如果只有按整体理解才成立，不能默认拆成互不相关的几笔。
- `结束旧方向` 和 `开启新方向` 是两个判断，不能因为出现反方向机会，就把它们写成同一个动作。
- 任何恢复、补救、降风险、补保护动作，都要区分 `已经发起` 和 `已经真正生效`。
- 多次加减仓、回补、对冲后，系统必须判断当前动作是在继续管理原来的交易生命周期，还是已经变成独立新机会；如果已经不是原链，要明确结束旧链或拆出新链处理。
- 若新开的仓位只是原链里的保护、减灾或管理动作，仍留在原链；若它本质上是为旧链做对冲或风险转移而开的独立仓位，则应新开链，但保留两条链的关联。
- “继续盯着” 不是每次都重做一轮完整判断；只有关键位触发、结构变化、执行状态变化或真实风险变化时，才把当前链推回 `OBSERVE`、`PLAN`、`EXECUTE` 或 `RECOVERY`。

### US-05 现在这笔仓位该继续拿、减仓、对冲，还是直接退出

- 触发：仓位出现较大回撤、短周期快速逆向、大小周期信号冲突，或消息面、研究结论、关键事件、时间拖延、持有成本变化持续打断原判断；也可能原方向开始失效，同时冒出反方向机会。
- 用户入口用例：
  - 直接追问当前 plan：“我持仓的 `near` 怎么样，继续持有吗，现在市场还是看多的吗？”
  - 继续拿还是减：“这一段回撤有点大了，我现在是继续拿，还是先减一点？”
  - 保护利润：“这笔已经走出来了，现在是继续拿，还是该把保护收紧？”
  - 对冲补救：“这边仓位不舒服了，要不要临时加个对冲，还是直接认错走人？”
  - 方向反了：“原来这笔多单逻辑像是坏了，现在反手做空合适吗？”
  - 计划内推进：“原计划里的加减仓、止盈、移动保护已经到位了，现在还按原节奏走吗？”
- 当前语境：这里处理的是“仓位已经在手里，接下来怎么管理”。重点不只是价格涨跌，而是原判断是否还成立、原节奏是否还适用、当前风险和持有成本是否还能接受，以及这一步是在按原计划推进，还是已经变成新的判断。
- 核心问题：当前该继续持有、减仓、补保护、临时对冲，还是直接退出；如果反方向机会出现，是先结束旧方向再重新判断，还是仍在原管理路径里处理。
- 系统必须支持：
  - 把短期噪音、原判断变化、事件风险、时间效率和持有成本拆开判断，而不是全混成一句“先拿着”
  - 能区分这一步是在按原管理计划推进，还是已经正式改计划
  - 每推进一步都要重算：剩余仓位、当前有效保护、剩余风险、下一检查点
  - 当需要对冲或组合处理时，要按整体暴露判断，而不是只看单边那一腿
  - 当原方向失效、反方向机会出现时，要把“结束旧仓位”和“开启新方向”拆成两个判断
  - 当用户在追问“继续持有吗”这类当前判断时，要先输出可扫读的当前 plan，再展开解释为什么这样判断
- 不宜直接沿原计划继续拿的情况：
  - 原判断已经明显变化，但系统还没说明变的是哪一层
  - 原计划里的保护、减仓、止盈或加仓条件已经触发，但系统还没说明这一步算不算按原计划推进
  - 用户口头想“反手”，但系统还没先说明旧方向是否已经正式结束
- 正式输出：
  - 继续持有，按原计划推进
  - 减仓或补保护后继续持有
  - 临时对冲或改成组合暴露
  - 结束旧仓位，回到新的观察和判断
  - 直接退出当前仓位
- 输出口径要求：
  - 默认先给 `plan 摘要`，再给 `判断依据`
  - `plan 摘要` 至少包含：`当前动作`、`方向状态`、`仓位处理`、`保护要求`、`下一检查点`
  - 只有解释、没有明确动作和检查点的回复，不算合格输出
- 回退 / 恢复路径：
  - 若只是持仓节奏变化，继续留在持仓语境下的 `OBSERVE / PLAN`
  - 若真实仓位、保护或风险状态不清，先进入 `RECOVERY`

### US-06 当前这笔你继续盯着，有变化再提醒我

- 触发：用户已经有仓位，或当前链已经进入持仓语境下的观察阶段，希望系统持续盯住盘面、关键位、保护位和执行状态；也可能要求按固定频率复查，或只在出现明显变化时提醒。
- 用户入口用例：
  - 持仓后继续盯盘：“我 `ETH` 2149 进去了，你继续监控盘面。”
  - 到关键位再提醒：“这笔你先盯着，碰到我设的关键位再叫我。”
  - 固定频率复查：“你每 15 分钟帮我看一次，结构变了再提醒我。”
  - 只在有事时打断：“别每次都重做完整分析，有变化再说。”
  - 带着当前计划监控：“按我们刚才那版计划盯着，如果保护该收紧了就提醒我。”
- 当前语境：这里处理的是“当前链已经在推进，用户希望系统持续盯盘并在有意义的变化时把它重新推回判断或动作”。重点不是每一轮都重做完整计划，而是持续对照 `当前计划`、`关键价位`、`失效条件`、`当前保护` 和 `真实执行状态`，区分“暂时没事”和“已经该介入”。
- 核心问题：当前链现在只是继续观察，还是已经触发了需要回到判断、执行或恢复的变化；如果暂时没事，下一次最值得提醒的条件是什么。
- 系统必须支持：
  - 支持 `固定频率复查` 和 `事件触发提醒` 两种监控方式
  - 监控时优先对照 `当前计划`、`关键位`、`失效条件`、`保护状态`、`活跃挂单` 和 `真实仓位`，而不是每次从零开始重做一轮
  - 每次监控结论都要保留它所依据的 `策略版本`、`观察周期 / 触发周期`、`适用场景` 和 `结论有效窗口`，避免一句“当前不追”被误解成跨周期、跨场景的全局否定
  - 当监控问题是“现在还能不能追”时，输出仍要拆成两层：先给 `方向状态`，再给 `执行结论`；不能只剩 `追多 / 追空 / 不追`
  - 能区分：`无实质变化`、`结构转弱但未失效`、`关键位触发`、`保护该调整`、`执行状态变化`、`真实状态异常`
  - 若只是正常波动，应明确说明“当前还留在原阶段”，而不是动不动就改计划
  - 若已经触发关键变化，要明确把当前链推回 `OBSERVE`、`PLAN`、`EXECUTE` 或 `RECOVERY`，而不是只发一句模糊提醒
  - 输出里要保留 `方向状态`、`执行结论`、`触发原因` 和 `下一次最值得提醒的条件`
- 不宜过度打断用户的情况：
  - 只是正常波动，尚未碰到关键位、失效条件或执行状态变化
  - 固定频率复查时，当前阶段与结论并没有发生实质变化
  - 价格有瞬时触碰，但还不足以改变当前计划或持仓节奏
- 正式输出：
  - 方向已转强 / 转弱，但当前不追
  - 当前无实质变化，继续监控
  - 结构转弱但未失效，收紧关注或准备重判
  - 关键位已触发，回到 `PLAN` 或进入 `EXECUTE`
  - 保护位或仓位管理需要调整，回到持仓语境下的 `PLAN / EXECUTE`
  - 真实状态异常或执行状态不清，进入 `RECOVERY`
- 回退 / 恢复路径：
  - 若没有实质变化，继续留在当前阶段并保留监控
  - 若触发关键位、保护调整或执行变化，回到对应阶段继续推进
  - 若 API 事实与当前理解冲突，先进入 `RECOVERY`

### US-08 减仓 / 止损 / 止盈单已经挂出，但还没确认生效，现在怎么办

- 触发：持仓中的加仓、减仓、止损、止盈、调杠杆、补保护或其他管理动作已经发出，但订单未成交、撤单 / 改单在途、部分成交后旧保护和新保护谁生效未定；也可能仓位、剩余挂单、保护链、资金在途和场所规则彼此对不上。
- 用户入口用例：
  - 管理单未成：“减仓单已经挂了，但一直没成交，这时候先按什么状态看？”
  - 保护是否还在：“我之前设过止损，但现在仓位变了，这个保护还算数吗？”
  - 补救在路上：“钱在转、改单也在路上，这时候先按最坏情况算还是先等一下？”
  - 系统来接管：“我中间手动动过仓位和保护，你先帮我对一下现在到底是什么状态。”
  - 异常状态处理：“交易所刚才有异常，这笔现在到底还有多少真实风险？”
- 当前语境：这里处理的是“仓位已经在管，但执行结果还没真正对齐”。重点不是用户本来打算怎么补救，而是现在真实仓位是多少、哪些挂单和保护还有效、哪些动作只是已经发起、哪些已经真正生效。
- 核心问题：当前真实仓位、活跃挂单、真正生效的保护和最坏风险是什么；现在该继续等、撤单、重挂、补保护、先降风险，还是直接退出。
- 系统必须支持：
  - 先以 API 对齐现实仓位、成交、活跃挂单、保护链、保证金和可用余额，再结合用户补充解释
  - 区分“补救意图已发出”和“补救已经生效”，不能把在途资金、在途改单或口头说明直接当成风险已经处理完
  - 识别保护仍匹配、部分失配、孤儿单、重复单、残余暴露这些状态
  - 若状态明显冲突，要先按最坏风险处理，而不是乐观假设保护已经到位
  - 能支持 `继续等确认 / 先补保护 / 先撤孤儿单 / 撤旧单释放占用后重提 / 临时对冲 / 迁移场所 / 直接退出`
- 不宜直接继续管理的情况：
  - 当前真实仓位、剩余挂单和保护链还没对齐
  - 用户记忆中的操作和 API 返回状态明显冲突
  - 补救动作还在路上，但系统已经把它当成完成
- 正式输出：
  - 继续等待确认
  - 先按最坏风险处理
  - 先补保护或撤孤儿单
  - 按已成交部分重算后重挂
  - 临时对冲、迁移场所或直接退出
  - 因状态不清，转入 `RECOVERY`
- 回退 / 恢复路径：
  - 若只是管理动作继续等待结果，留在持仓中的 `WAIT-UNTIL-FILL`
  - 若真实状态还没对齐，先进入 `RECOVERY`

### US-10 不完全接受管理建议，但还是想继续拿

- 触发：系统建议减仓、退出、不要再加仓、提前保护利润、上移止损、取消后续加仓或暂停继续管理，但用户仍想继续持有、继续加仓、推迟止损、改变止盈方式，或只接受方向判断、不接受当前管理节奏。
- 用户入口用例：
  - 不想减仓：“你让我先减一点，但我还是想继续拿。”
  - 不想太早止盈：“这笔我还想再拿一段，不想这么早出。”
  - 推迟止损：“这个止损我想再放远一点，不然太容易被扫。”
  - 还想加仓：“你说别加了，但我还是想再补一层。”
  - 只认方向不认节奏：“方向我认，但你这个管理节奏我不想照做。”
- 当前语境：这里不是重新判断有没有机会，而是仓位已经在手里，系统已经给出管理建议，用户却想偏离这套建议继续走。重点是用户到底改了什么、承担了什么新增风险，以及当前真正有效的管理计划是什么。
- 核心问题：这次分歧是在改方向、改风险、改节奏，还是改执行方式；用户知情后到底在承担什么新增风险；这次偏离还是不是原管理计划的一部分。
- 系统必须支持：
  - 把 `系统原建议`、`用户反馈`、`最终确认动作` 明确拆开
  - 区分普通风险提示和硬性阻断；前者允许用户知情后继续，后者不能被默默降级
  - 若用户拒绝的是降风险或保护建议，要同步重算当前最坏风险、真实保护和剩余计划
  - 分歧处理后的结果必须可回看，复盘时能分清是系统判断问题、用户主动偏离，还是执行层问题
- 不宜直接照用户意思继续的情况：
  - 用户只表达情绪或倾向，但还没说清具体改什么
  - 用户改了仓位、保护、止盈、止损或继续加仓节奏，但系统还没重算新的风险
  - 系统判断已经触发硬性阻断，但用户仍要求继续
- 正式输出：
  - 接受系统原建议，沿原管理计划继续
  - 按用户修改后的版本形成新管理计划并继续
  - 明确触发硬性阻断，当前不继续推进
  - 因状态不清或风险未重算，转入 `RECOVERY`
- 回退 / 恢复路径：
  - 若只是修改当前持仓计划，继续留在持仓语境下的 `PLAN`
  - 若分歧暴露出真实状态不清或风险失真，先进入 `RECOVERY`

## 4. 投研与回测

### 固定要求

- 投研输出不只是“结论对不对”，还要明确 `主要依据`、`反方证据`、`适用范围`、`失效条件`。
- 回看历史时必须锁住未来信息，不能把后面才知道的内容提前带回当时语境。
- 研究结论可以直接服务当前观察、开仓前判断和持仓语境下的判断 / 执行，也可以继续沉淀成候选规则。
- 候选规则在进入主流程前，必须先经过更长历史或回放语境下的验证，不能只凭少量样本直接升格。

### US-39 看外部材料、找历史案例后，这些信息对现在到底有什么用

- 触发：用户要求系统解读宏观材料、市场报告、跨资产关系、某标的相对同类资产的异常强弱，或持续记录多市场线索后再做周期分析；也可能想知道当前场景、某个历史时间窗口，以前是否出现过相似案例、事件或走势，或要求从某个历史日期开始回放，一步步模拟当时的完整交易过程。
- 用户入口用例：
  - 解读外部材料：“这份报告、这条消息、这个宏观变化，对现在到底有什么影响？”
  - 找历史案例：“现在这个场景，以前有没有类似的时候？”
  - 只看当时能看到的信息：“如果回到当时，只看当时已发生的信息，这套判断还站得住吗？”
  - 找反例：“你别只找支持的证据，也帮我找最强反例。”
  - 回接主流程：“这些研究结论最后会怎么影响我现在这笔判断？”
- 当前语境：这里处理的是“把外部信息、长期线索和历史案例整理成对当前判断有用的研究结论”。重点不是信息越多越好，而是要区分哪些是真正重要的新语境、风险提示或反证，哪些只是背景噪音；如果走到历史回放，也要保证只能使用当时能看到的信息。
- 核心问题：这些外部材料、历史案例和长期线索里，哪些对现在真的有用；哪些只是噪音；哪些反证足以推翻原判断；如果回到当时重新看，这套判断还能不能站得住。
- 系统必须支持：
  - 消化外部材料、汇总长期线索、主动寻找反证，并把结论落回当前判断、候选比较、主要风险和失效条件
  - 把当前问题映射到历史案例和时间锚点，并把相关事件放回同一语境里
  - 回放时支持：选起点、锁住未来信息、生成当时语境、推进时间后继续观察 / 判断 / 执行 / 复盘
  - 输出既可以直接服务当前主流程，也可以继续沉淀为候选研究结论或候选规则
- 不宜直接把研究结论回接主流程的情况：
  - 还没区分真正的新信息和背景噪音
  - 只找到了支持证据，还没补足反方证据
  - 历史回看时混入了事后才知道的信息
- 正式输出：
  - 一版可直接回接主流程的研究结论
  - 一版基于历史案例和回放得到的可用判断
  - 一版明确反证和失效条件的研究结论
  - 暂时只保留为研究线索，先不回接主流程

### US-56 这些想法能不能变成可验证的规则，并通过历史检验

- 触发：用户已经积累了一批真实交易，也包含没成交、没执行、主动放弃的样本，怀疑其中有重复模式；或带来外部方法、粗糙假设、截图标注、过滤条件、执行约束与相邻版本，想放到更长历史里做系统检验。
- 用户入口用例：
  - 从样本里找规律：“这些交易里是不是有重复模式，能不能提炼成规则？”
  - 带外部方法来验证：“这个方法我想法还很粗，你帮我翻成能检验的规则试一下。”
  - 比较相邻版本：“这个条件到底有没有价值，还是只是看起来更漂亮？”
  - 检查可执行性：“回测看着不错，但真实滑点、流动性和成交假设能扛住吗？”
  - 决定去向：“这套东西最后该进主流程、先影子跟踪，还是继续留在 research？”
- 当前语境：这里处理的是“把想法从口头判断或样本感觉，翻成可以被检验的规则，再放到更长历史里验证”。重点不是做出一个好看的回测结果，而是拆清楚哪些条件真的带来边际价值、这套想法适用于什么环境、又会在什么前提下失效。
- 核心问题：哪些部分能被翻译成可量化、可重复检验的信号、过滤和执行规则；哪个条件真的带来价值；这套想法适用于哪些环境，又会在什么前提下失效。
- 系统必须支持：
  - 从历史样本和外部方法中抽出候选规则，并把想法拆成可量化信号、过滤条件、调仓规则、成本假设和评价指标
  - 对相邻版本做条件隔离，尽量说明到底是哪一条变化带来了差异
  - 检查时点、滑点、流动性和成交假设，避免回测结果看起来好、真实却不可执行
  - 输出适用边界、失效环境，以及它应进入主流程、转成观察清单或影子跟踪、继续留在研究，还是只适合回放验证
- 不宜直接把候选规则升格进主流程的情况：
  - 规则还停留在模糊描述，无法量化或无法重复检验
  - 回测结果过度依赖少量样本、宽松成交假设或明显不真实的成本条件
  - 还没说清这套规则适用于什么环境、在什么情况下会失效
- 正式输出：
  - 一版可验证的候选规则
  - 一版经过历史检验后的适用边界和失效环境
  - 进入主流程前继续观察或影子跟踪
  - 暂时留在研究，不进入主流程

## 5. 解释、复盘与迭代

### 固定要求

- 复盘不能只看结果好坏；至少要把 `系统原计划`、`用户实际执行`、`最终结果` 拆开看。
- 解释不能只给一句结论；必须能回到当时的证据、反证、事件、约束和用户确认。
- 暂停、恢复、放大或缩小风险，都要有明确门槛，不能只凭情绪切换。
- `阶段性闭合` 至少包括两类：交易已经结束；或详细 plan 已经完成必要确认与分歧处理，但最终没有进入执行。用户刚发起请求、系统尚未收敛的范 plan，不算阶段性闭合。
- 复盘产出的改法先作为候选，不直接升格为以后默认要用的策略。
- 系统需要继续对改法做验证；验证通过后，再建议用户是否更新策略。
- 主流程任一时点只读取当前明确生效的一版，不混用新旧版本。
- 若复盘暴露出止损后情绪化重试、连续亏损或事故级风险，系统必须额外给出 `继续观察 / 降风险继续 / 先暂停 / 分级恢复` 建议。
- 当用户拿自动化监控结果回来 challenge 时，系统必须能把 `策略不适配`、`时间窗口不匹配`、`识别漏判`、`监控结论过期未刷新` 和 `执行约束阻断` 拆开解释，而不是只重复一句“当时不追”。

### US-12 复盘这笔交易时，问题到底出在哪，为什么当时这样做、后来又为什么改

- 触发：一条交易链已经完全结束，或虽然没有彻底结束但已经阶段性闭合；用户可能想立刻复盘，也可能只是先留待后面统一回看。也可能用户直接追问：为什么当时给出那个建议、为什么后来改了计划、为什么提前或延后执行、为什么止损止盈或保护方式发生变化。还可能刚止损，用户想知道这次到底错在判断、执行、风控还是情绪，以及下次再遇到类似情况时应该重点看什么。
- 用户入口用例：
  - 现在就复盘：“这笔刚结束，你现在就帮我复一下。”
  - 先放池子里：“先别急着复盘，后面统一回看。”
  - 追问原因：“你当时为什么这么建议，后来又为什么改了？”
  - 追问动作变化：“到底是哪条证据、哪次事件，让你把计划改了？”
  - 回看链路：“你把这笔从最开始到后来变化的原因串起来讲清楚。”
  - 止损后复盘：“刚止损它又回去了，这次到底错在哪，下次再遇到类似情况该怎么看？”
- 当前语境：这里处理的是“交易已经结束或阶段性闭合，如何拉起来回看”。重点不只是要不要复盘，还包括复盘时能不能把当时看到什么、怎么判断、后来为什么变化、最终为什么变成这个结果讲清楚；并沉淀出问题定位、改法候选和下次再遇到类似情况时的观察重点。
- 核心问题：这条交易链现在要不要立刻复盘；如果复盘，能不能清楚说明当时为什么这样建议、后来又为什么改；下次遇到类似情况时应该重点观察什么。
- 系统必须支持：
  - 凡是已结束或阶段性闭合的交易链，都能进入待复盘池，不要求必须即时复盘
  - 复盘时能沿同一条交易链回看：当时看到什么、当时怎么判断、后来新增了什么、为什么采纳或放弃、因此改了什么动作
  - 解释不能只给结论，必须能把计划变化映射回具体证据、反证、事件、约束和用户确认
  - 至少能区分：原判断未变只是执行方式变了、原判断变了、风险约束变了、成本或场所约束变了、用户主动偏离系统建议
  - 若用户在止损后马上想再做一次，要能区分这是在回看问题、提炼观察重点，还是已经滑向情绪化追回
- 不宜直接给复盘结论的情况：
  - 还没把系统原计划、用户执行和最终结果拆开
  - 还说不清动作变化到底是由什么信息触发的
  - 这条链虽然结束了，但还没整理到可复盘状态
- 正式输出：
  - 立刻复盘
  - 先进入待复盘池，后续再拉起
  - 一版能解释“为什么当时这样做、后来为什么改”的复盘结果
  - 一版供下一轮开仓前继续观察的重点和改法候选

### US-57 自动化一直说不追，但后来行情启动了，到底是策略不适配还是我们没识别出来

- 触发：自动化监控、定时复查或事件触发提醒曾连续给出 `不追 / 继续等 / 无实质变化`，但事后行情很快启动，用户回来 challenge 系统：“当时为什么没识别到”“到底是策略不适配，还是识别逻辑有问题”；也可能用户进一步要求把这次 review 直接转成下一轮策略迭代输入。
- 用户入口用例：
  - 直接 challenge：“刚刚那段明明已经启动了，你为什么一直说不追？”
  - 查时间维度：“这波是 `5m` 启动，你是不是拿 `1h` 的规则在看？”
  - 查适用边界：“这个策略是不是根本不适合这种加速段，只适合回踩确认？”
  - 查识别问题：“如果策略本来适配，那到底漏看了哪条特征？”
  - 带着迭代意图进入：“这类情况以后要怎么改，先改监控还是先改策略定义？”
- 当前语境：这里处理的是“用户拿着一次监控失配或疑似漏判回来复盘，并希望沿着同一条链把原因查清，再决定要不要推进策略迭代”。重点不是事后诸葛地证明行情确实涨了，而是要还原当时监控所依据的策略版本、时间维度、场景假设、结论有效窗口和实际漏掉的异变，再判断问题出在 `策略边界` 还是 `识别实现`。
- 核心问题：当时那句 `不追 / 继续等` 在它自己的时间维度和适用场景里是否成立；如果不成立，是策略本身不适配、时间窗口错配、识别漏判、结论过期没刷新，还是执行约束被误说成了方向否定；这次 review 应该沉淀成什么候选改法。
- 系统必须支持：
  - 沿时间顺序回看当时监控依据的 `策略版本`、`观察周期 / 触发周期`、`适用场景`、`适用方向` 和 `结论有效窗口`
  - 能识别这类 prompt / 输出口径问题：方向已经识别到，但因为最终只允许报 `追多 / 追空 / 不追`，导致 `方向状态` 被埋掉
  - 把“当时说不追”拆成真正含义，例如 `方向不成立`、`方向成立但当前周期不该追`、`等更低周期确认`、`执行约束不满足`
  - 对照事后启动段，区分：`策略本身不覆盖这类场景`、`策略覆盖但时间窗口错配`、`策略和窗口都对但识别漏判`、`结论本来只在短窗口内有效但自动化没有及时重判`
  - 若问题暴露出策略适用边界不清，要能补出最小说明：适用什么周期、什么市场环境、做多/做空是否都适用、典型失效场景是什么
  - 若问题暴露出识别逻辑缺口，要能提出候选改法，例如补首发信号、补更低周期确认、补监控过期重判、拆成 sibling 版本，而不是直接把一次个案写成长期规则
- 不宜直接给“以后就这么改”的情况：
  - 还没把当时可见信息和事后结果拆开
  - 还没说清是 `策略边界问题` 还是 `识别实现问题`
  - 还没按具体时间维度重放这次监控结论和后续启动段
- 正式输出：
  - 一版说明当时 `不追 / 继续等` 在原时间维度内是否成立的 review 结果
  - 一版明确错因归属的结论，例如 `策略错配 / 时间窗口错配 / 识别漏判 / 结论过期未刷新 / 执行约束阻断`
  - 一版候选改法，进入后续验证或策略迭代
  - 暂时只保留为待验证假设，不直接改主流程默认策略

### US-11 回头看一段时间后，我们真正该改哪一部分

- 触发：用户定期回顾一段时间内的交易、事件、决策和策略调整，或在单笔大赚大亏、连续盈利后，开始怀疑结果是否过度依赖少数样本、运气、违规盈利或同一类集中风险；也可能反复感觉方向看对了，但真实结果表现为没成交、没按计划执行、拿不住、过早止盈、追着回补，或回测看起来不错而实盘被成本、成交和持仓推进链路吃掉。还可能已经连续止损、短期回撤扩大、执行动作变形，甚至刚经历强平、自动减仓、穿仓等事故级亏损，用户想知道接下来是继续、缩手还是暂停。
- 用户入口用例：
  - 总结一段时间：“最近这段时间我们到底该改哪里？”
  - 找真正的 edge：“该保留的到底是什么，哪些只是运气？”
  - 查问题在哪：“方向经常看对，但就是没赚到，问题到底丢在哪一环？”
  - 看策略冗余：“现在候选策略是不是太多、太重叠了？”
  - 看过程质量：“是判断本身有问题，还是执行链路吃掉了优势？”
  - 连续亏损后处理：“最近连着亏了几笔，我现在是继续、缩手，还是先停？”
  - 事故后恢复：“这次事故挺重的，之后怎么恢复才算有纪律？”
- 当前语境：这里处理的是“拉长到一段时间后，我们接下来到底该改什么”。重点不是单笔输赢，而是收益、样本分布、过程质量、集中风险、策略重叠度，以及‘看对了’和‘赚到了’之间到底断在哪一环；最后要落到明确改进方向，而不是停在一句‘最近不太顺’。这些改进方向会先作为候选，后续再进入验证与是否建议更新策略的内部流程。
- 核心问题：接下来真正该改的是判断、执行、持仓推进、恢复衔接、成本假设，还是策略本身；哪些东西应该保留，哪些应该合并、降级或停掉；在连续亏损或事故后，现在该继续观察、降风险运行，还是先暂停。
- 系统必须支持：
  - 把收益、样本分布、过程质量、组合集中度和策略池重叠度拆开看
  - 持续对照 `系统原计划 / 用户实际执行 / 最终结果`
  - 把“看对了”和“赚到了”之间的链条正式拆开，至少能区分：没等到、不敢做、做了但仓位不对、拿不住、保护失真、恢复断档、成本吃掉优势
  - 最后要落到明确改动建议，而不是停在抽象总结
  - 若已经连续亏损或发生事故，要把 `暂停决定 / 恢复门槛 / 分级恢复路径` 一起给出来
- 不宜直接给出改动方向的情况：
  - 结果过度依赖少量样本或运气
  - 只看收益，没有把过程质量和集中风险拆开
  - 还没说清主要问题到底发生在哪一环
- 正式输出：
  - 该保留什么
  - 该改哪一部分
  - 哪些候选策略或做法需要合并、降级或淘汰
  - 目前还不能形成可靠改动方向
  - 降风险继续观察
  - 先暂停，再进入分级恢复
  - 暂时不适合恢复常规交易
